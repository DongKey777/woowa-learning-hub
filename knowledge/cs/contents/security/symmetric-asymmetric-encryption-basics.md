## 핵심 개념

암호화는 데이터를 제3자가 읽을 수 없는 형태로 변환하는 것이다. 대칭키 암호화는 암호화와 복호화에 같은 키를 사용하고, 비대칭키 암호화는 암호화용 공개키(Public Key)와 복호화용 개인키(Private Key) 쌍을 사용한다. 입문자가 헷갈리는 이유는 HTTPS가 두 방식을 모두 쓰기 때문에, "HTTPS는 대칭이야 비대칭이야?"라는 질문에서 막히기 때문이다.

## 한눈에 보기

대칭키는 같은 키로 암호화·복호화하고, 비대칭키는 공개키로 암호화하고 개인키로만 복호화한다.

```
[대칭키 — AES]
암호화: 데이터 + 키A → 암호문
복호화: 암호문 + 키A → 데이터
└ 빠르지만 키를 양쪽이 안전하게 사전 공유해야 함
```

```
[비대칭키 — RSA]
공개키(Public Key): 누구에게나 공개 가능
개인키(Private Key): 본인만 보관
암호화: 데이터 + 공개키 → 암호문   (누구나 암호화 가능)
복호화: 암호문 + 개인키 → 데이터   (개인키 소유자만 복호화)
```

HTTPS(TLS)는 핸드셰이크에서 비대칭키로 대칭키를 교환한 뒤, 실제 통신은 빠른 대칭키로 처리한다.

## 상세 분해

- **대칭키의 장단점**: AES는 매우 빠르고 대용량 데이터 암호화에 적합하다. 단점은 같은 키를 양쪽이 사전에 안전하게 공유해야 한다는 점이다. 키가 중간에 탈취되면 모든 데이터가 노출된다.
- **비대칭키의 장단점**: RSA는 공개키를 자유롭게 배포할 수 있어서 키 배포 문제가 없다. 단점은 대칭키보다 수십~수백 배 느리기 때문에 대용량 데이터 암호화에는 비실용적이다.
- **디지털 서명에서의 역할 역전**: 서명할 때는 개인키로 서명하고 공개키로 검증한다. "이 개인키 소유자가 서명했다"는 것을 누구나 공개키로 확인할 수 있다.
- **HTTPS가 두 방식을 합치는 이유**: TLS 핸드셰이크에서 비대칭키로 대칭키를 안전하게 교환한 뒤, 실제 데이터 전송은 빠른 대칭키로 암호화한다. 속도와 키 배포 문제를 동시에 해결하는 방식이다.

## 흔한 오해와 함정

- **"비대칭키가 더 안전하니까 항상 써야 한다"** → 비대칭키는 느리기 때문에 대용량 데이터에는 실용적이지 않다. 실무에서는 두 방식을 결합해 사용한다.
- **"공개키는 공개해도 되니까 아무 데나 올려도 된다"** → 공개키 자체는 공개해도 되지만, 개인키와 쌍이 맞는 공개키임을 보증하려면 인증서(CA 서명)가 필요하다. 공개키만 올리면 중간자가 가짜 공개키를 교체할 수 있다.
- **"해시와 암호화는 같은 것이다"** → 해시는 단방향으로 원본을 복원할 수 없다. 암호화는 복호화가 가능한 양방향이다. 비밀번호는 암호화가 아니라 해시(단방향)로 저장해야 한다.

## 실무에서 쓰는 모습

- **HTTPS(TLS)**: 인증서는 비대칭키 기반, 실제 통신은 핸드셰이크에서 교환한 대칭키로 처리한다.
- **JWT 서명**: RS256은 RSA 개인키로 서명하고 공개키로 검증하는 비대칭 방식이다. HS256은 같은 시크릿 키를 서명과 검증에 모두 사용하는 대칭 방식이다.
- **API 시크릿 저장**: 대칭키나 시크릿 값은 환경 변수 또는 KMS에 저장하고, 코드에 직접 포함하지 않는다.

## 면접/시니어 질문 미리보기

- **"HTTPS는 대칭키를 쓰나요, 비대칭키를 쓰나요?"** → 키 교환(핸드셰이크)은 비대칭키, 실제 데이터 전송은 대칭키를 사용하며, 두 방식의 장점을 결합한 구조다.
- **"JWT에서 HS256과 RS256의 차이는 무엇인가요?"** → HS256은 대칭(동일 시크릿), RS256은 비대칭(개인키 서명/공개키 검증)이며, 서버가 여러 개인 분산 환경에서는 공개키를 배포할 수 있는 RS256이 적합하다.

## 한 줄 정리

대칭키는 빠르지만 키 공유 문제가 있고, 비대칭키는 키 배포 문제를 해결하지만 느리므로 실무에서는 두 방식을 결합해 쓴다.
