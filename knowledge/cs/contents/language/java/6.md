## ClassLoader

ClassLoader는 클래스를 메모리로 가져오는 객체다.  
교과서적으로는 "클래스를 적재한다"라고 설명하지만, 실무에서는 더 넓게 봐야 한다.

- 어떤 loader가 어떤 클래스를 볼 수 있는가
- class path / module path / context classloader가 어떻게 다르게 동작하는가
- parent delegation이 언제 깨지는가
- class unloading이 언제 일어나는가

대표 계층:

- Bootstrap
- Platform
- Application

그리고 보통 parent delegation model로 동작한다.

### 로더가 중요한 이유

class loading은 단순 I/O가 아니다.

- linkage
- verification
- initialization
- reflection visibility
- service loading
- plugin isolation

이 모두가 얽힌다.

### edge case

자세한 예외는 [ClassLoader Delegation Edge Cases](./classloader-delegation-edge-cases.md)에서 다룬다.  
특히 child-first loader, context classloader, webapp loader는 실무에서 자주 헷갈린다.

## Exception 설계

예외 설계는 "에러를 던지느냐 마느냐"가 아니라 **실패를 어디까지 의미 있게 표현하느냐**의 문제다.

### 핵심 질문

- 누구의 책임인지
- 다시 시도 가능한지
- 사용자에게 보여줄 메시지인지
- 시스템 장애로 번질지
- checked / unchecked 중 무엇이 더 적합한지

### 도메인 예외와 인프라 예외

- 도메인 규칙 위반: `IllegalArgumentException` 또는 domain-specific exception
- 외부 시스템 실패: DB, IO, network failure

둘을 섞으면 호출자가 복구 가능한 실패와 불가능한 실패를 구분하기 어렵다.

### 예외가 계약이 되는 순간

예외는 문서보다 강한 계약이 될 수 있다.  
특히 public API나 service boundary에서는 예외 타입이 곧 behavior contract다.

## equals와 hashCode

### `equals`

- 두 객체가 논리적으로 같은지 비교
- identity가 아니라 semantic equality를 정의한다

### `hashCode`

- 해시 기반 컬렉션에서 버킷을 찾기 위한 값
- equals와 계약이 맞아야 한다

### 중요한 규칙

- `equals`가 같으면 `hashCode`도 같아야 한다
- 그렇지 않으면 HashMap/HashSet이 예측 불가능해진다

### 실무에서 자주 놓치는 것

- mutable field를 `equals/hashCode`에 넣으면 객체를 컬렉션에 넣은 뒤 망가질 수 있다
- inheritance 계층에서 equality를 설계하지 않으면 대칭성/추이성이 깨질 수 있다
- record는 이 계약을 자동으로 잘 만들어 주지만, semantics를 자동으로 맞춰 주는 것은 아니다

### runtime perspective

hash 기반 컬렉션에서는 equality contract가 object identity보다 먼저 읽히는 경우가 많다.  
즉 "같아 보이는 객체"를 코드가 어떻게 정의하는지가 성능과 correctness를 같이 좌우한다.

## Comparable과 compareTo

`Comparable`은 객체의 natural ordering을 정의한다.

```java
public int compareTo(User other)
```

의미:

- 음수: 내가 더 작다
- 0: 같다
- 양수: 내가 더 크다

### equals와의 관계

`compareTo == 0`과 `equals == true`는 강제적으로 같아야 하는 건 아니지만, 가능한 한 일관적이어야 한다.  
정렬 컬렉션과 해시 컬렉션에 동시에 들어가는 타입에서는 특히 중요하다.

### 정렬 contract

정렬은 순서뿐 아니라 중복 제거, range query, tree 구조에서의 위치를 결정한다.  
즉 compareTo는 단순 편의가 아니라 자료구조 동작을 결정하는 핵심 계약이다.

## 실전 시나리오

### 시나리오 1: 같은 이름의 클래스가 두 버전으로 보인다

ClassLoader 경계 문제를 의심한다.  
webapp, plugin, reflection, module boundary를 함께 봐야 한다.

### 시나리오 2: 예외가 너무 많아 원인을 못 찾는다

도메인 예외와 인프라 예외를 분리하지 않았을 가능성이 크다.  
`try-with-resources`와 suppressed exception도 같이 봐야 한다.

### 시나리오 3: HashMap에서 키를 못 찾는다

equals/hashCode 계약이 깨졌거나 mutable state를 키로 쓴 경우다.

### 시나리오 4: TreeSet에 넣었더니 중복이 사라진다

compareTo가 0을 반환하는 기준이 equals와 다를 수 있다.  
의도한 동작인지 검토해야 한다.

## 코드로 보기

### 1. domain exception

```java
public class InsufficientBalanceException extends RuntimeException {
    public InsufficientBalanceException(String message) {
        super(message);
    }
}
```

### 2. 안전한 equals/hashCode

```java
import java.util.Objects;

public final class UserId {
    private final String value;

    public UserId(String value) {
        this.value = value;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof UserId other)) return false;
        return Objects.equals(value, other.value);
    }

    @Override
    public int hashCode() {
        return Objects.hash(value);
    }
}
```

### 3. compareTo

```java
public int compareTo(User other) {
    return this.name.compareTo(other.name);
}
```

## 트레이드오프

| 선택 | 장점 | 비용 |
|---|---|---|
| 명시적 domain exception | 실패 의미가 분명하다 | 예외 타입이 많아질 수 있다 |
| checked exception | 호출자가 처리 경로를 고민한다 | API가 장황해질 수 있다 |
| equals/hashCode 자동 생성 | 편하다 | mutable 필드/상속에서는 주의 필요 |
| compareTo 기반 정렬 | 정렬 자료구조와 잘 맞는다 | equals와 일관성을 설계해야 한다 |

핵심은 세 주제를 따로 외우는 게 아니라 "경계와 계약"이라는 공통 축으로 읽는 것이다.

## 꼬리질문

> Q: ClassLoader는 왜 필요한가요?
> 핵심: 클래스를 동적으로 로드하고 격리/재사용 경계를 만들기 위해서다.

> Q: domain exception과 infrastructure exception을 왜 나누나요?
> 핵심: 호출자가 복구 가능한 실패와 시스템 실패를 구분해야 하기 때문이다.

> Q: equals와 hashCode는 왜 같이 구현하나요?
> 핵심: 해시 기반 컬렉션에서 동일성 계약을 맞추기 위해서다.

> Q: compareTo와 equals는 꼭 같아야 하나요?
> 핵심: 강제는 아니지만, 컬렉션 동작 예측 가능성을 위해 일관성을 맞추는 것이 좋다.

## 한 줄 정리

ClassLoader, 예외 설계, equals/hashCode/compareTo는 각각 로딩, 실패, 객체 의미의 경계를 정의하고, 실무에서는 이 세 경계를 함께 설계해야 한다.
