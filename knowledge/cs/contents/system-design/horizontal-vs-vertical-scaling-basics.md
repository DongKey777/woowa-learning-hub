# 수평 확장과 수직 확장 기초 (Horizontal vs Vertical Scaling Basics)

> 한 줄 요약: 수직 확장은 서버 사양을 올리는 방식이고, 수평 확장은 서버 대수를 늘리는 방식이며, 대규모 시스템에서는 수평 확장이 한계 비용과 가용성 면에서 유리하다.

## 핵심 개념

서비스 트래픽이 늘어날 때 서버를 어떻게 키울 것인가에 대한 두 가지 방향이다.

- **수직 확장(Scale Up)**: 서버 한 대의 CPU, 메모리, 디스크를 더 좋게 바꾼다.
- **수평 확장(Scale Out)**: 같은 서버를 여러 대 추가한다.

입문자가 자주 헷갈리는 지점은 **왜 수직 확장만 하면 안 되는가**이다.

수직 확장에는 물리적 한계가 있고, 서버 한 대가 다운되면 전체 서비스가 멈춘다. 수평 확장은 서버를 더 추가하기만 하면 되고, 한 대가 죽어도 나머지가 계속 동작한다.

---

## 한눈에 보기

```text
수직 확장 (Scale Up)
  서버 A [2코어 4GB] -> 서버 A [16코어 64GB]
  (비용 급증, 한계 존재)

수평 확장 (Scale Out)
  서버 A          서버 A
                  서버 B  (추가)
  로드밸런서 필요  서버 C  (추가)
```

비교 요약:

| 기준 | 수직 확장 | 수평 확장 |
|---|---|---|
| 방법 | 서버 사양 업그레이드 | 서버 대수 추가 |
| 한계 | 물리적 최대치 존재 | 이론상 무제한 |
| 단일 장애점 | 있음 | 없음 (서버 분산) |
| 비용 | 사양 올릴수록 가파름 | 선형에 가까움 |
| 복잡도 | 낮음 | 높음 (상태 관리 필요) |

## 상세 분해

- **수직 확장의 실제 쓰임새**: 데이터베이스 서버에서 자주 쓰인다. DB는 수평 확장이 복잡하기 때문에 초기에는 사양을 올리는 방식으로 시작한다. 단, RDS 인스턴스 타입 변경은 재시작이 필요해 다운타임이 생긴다.
- **수평 확장의 전제 조건**: 서버가 상태를 로컬에 저장하지 않아야 한다(Stateless). 사용자 세션을 서버 메모리에 저장하면 로드 밸런서가 다른 서버로 라우팅했을 때 세션이 없어진다.
- **Auto Scaling**: 클라우드에서는 CPU 사용률이나 요청 수에 따라 서버 인스턴스를 자동으로 추가/제거하는 Auto Scaling을 쓴다. 비용 최적화와 탄력적 확장이 가능하다.
- **데이터베이스 수평 확장**: 애플리케이션 서버는 수평 확장이 쉽지만, DB는 샤딩이나 Read Replica 같은 추가 설계가 필요해 더 복잡하다.

---

## 흔한 오해와 함정

- **"수평 확장을 하면 비용이 항상 절감된다"**: 서버 대수가 늘어나면 관리 복잡도가 올라가고, 로드 밸런서, 분산 세션 저장소, 분산 트랜잭션 관리 비용이 추가된다.
- **"애플리케이션 서버를 늘리면 DB 병목이 없어진다"**: 애플리케이션 서버 수평 확장 시 DB 커넥션 수가 급증해 DB가 병목이 된다. 커넥션 풀과 DB 확장을 함께 고려해야 한다.
- **"Stateful 서비스는 수평 확장이 불가능하다"**: 불가능하진 않다. 상태를 외부 저장소(Redis, DB)로 분리하면 수평 확장 가능한 Stateless 서비스가 된다.

---

## 실무에서 쓰는 모습

가장 흔한 시나리오는 쇼핑몰 이벤트로 트래픽이 급증하는 경우다.

1. 평소에는 애플리케이션 서버 2대로 운영한다.
2. 이벤트 직전 Auto Scaling 최소값을 10대로 높인다.
3. 로드 밸런서가 10대에 균등 분배한다.
4. 이벤트 후 트래픽이 줄면 Auto Scaling이 서버를 줄인다.

이때 서버가 세션을 로컬에 저장하지 않도록 Spring Session + Redis로 세션을 외부화해야 수평 확장이 올바르게 동작한다.

## 면접/시니어 질문 미리보기

> Q: 수직 확장과 수평 확장의 차이를 설명해 주세요.
> 의도: 기본 확장 전략 구분 능력 확인
> 핵심: 수직은 서버 하나를 업그레이드, 수평은 서버를 추가한다. 수직은 간단하지만 물리 한계와 단일 장애점이 있고, 수평은 로드 밸런서와 Stateless 설계가 필요하다.

> Q: 수평 확장 시 서버가 Stateless여야 하는 이유는 무엇인가요?
> 의도: 상태 관리와 분산 환경의 관계 이해 확인
> 핵심: 로드 밸런서가 요청을 어느 서버로 보낼지 모르기 때문에, 상태를 서버 로컬에 두면 다른 서버에서 해당 상태를 찾을 수 없다. 상태는 Redis나 DB 같은 공유 저장소에 두어야 한다.

> Q: 애플리케이션 서버를 늘렸는데 DB가 느려진 이유는 무엇인가요?
> 의도: 계층별 병목 파악 능력 확인
> 핵심: 서버 수가 늘어나면 DB 커넥션 수도 비례해 증가한다. DB 커넥션 한계나 쿼리 부하가 새 병목이 된다. 커넥션 풀 설정과 DB 읽기 확장(Read Replica)을 함께 검토해야 한다.

---

## 한 줄 정리

수평 확장은 서버 대수를 늘려 단일 장애점을 없애고 이론상 무한 확장을 가능하게 하지만, Stateless 설계와 로드 밸런서가 전제 조건이다.
