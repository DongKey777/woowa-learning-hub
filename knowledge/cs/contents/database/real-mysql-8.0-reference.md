# Real MySQL 8.0 Reference

- 원본 PDF: [materials/real-mysql-8.0.pdf](./materials/real-mysql-8.0.pdf)

## 장별 구조

### 01장 소개 (pp. 1-5)

- MySQL의 위치와 강점, 생태계, 왜 MySQL을 선택하는지 설명한다.
- 이 장은 "왜 이 책이 SQL 문법보다 엔진과 운영에 집중하는가"를 이해하는 용도다.

### 02장 설치와 설정 (pp. 6-51)

- 버전/에디션 선택, 서버 시작과 종료, 업그레이드, 설정 파일 구조를 다룬다.
- `SET PERSIST`, `SET PERSIST_ONLY`, `my.cnf`, `mysqld-auto.cnf` 같이 운영에서 자주 부딪히는 설정 변경 방식이 핵심이다.
- 신규 서버 세팅, 업그레이드 체크리스트, 설정 변경 절차를 잡을 때 가장 먼저 참고할 장이다.

### 03장 사용자 및 권한 (pp. 52-75)

- 사용자 식별 방식, 계정 생성, 비밀번호 정책, 권한, 역할(Role)을 다룬다.
- 시스템 계정과 일반 계정 구분, 이중 비밀번호, 역할 기반 권한 부여처럼 운영 정책 설계에 직접 연결되는 내용이 많다.
- 애플리케이션 계정, 읽기 전용 계정, 운영자 계정을 분리할 때 유용하다.

### 04장 아키텍처 (pp. 76-153)

- MySQL 서버 전체 구조, 스레딩 모델, 메모리 구조, 쿼리 실행 구조를 설명한다.
- InnoDB 내부에서는 클러스터링 인덱스, MVCC, non-locking consistent read, 버퍼 풀, 언두/리두 로그, 체인지 버퍼, Double Write Buffer, 어댑티브 해시 인덱스를 다룬다.
- 로그 파일과 에러 로그, 제너럴 로그, 슬로우 로그 해석까지 포함된다.
- 성능 문제를 볼 때 가장 중요한 장이다. "왜 느린가"를 묻기 전에 이 장의 모델을 기준으로 원인을 나눠야 한다.

### 05장 트랜잭션과 잠금 (pp. 154-183)

- 트랜잭션 기본, 엔진 레벨 잠금, InnoDB 레코드 잠금, 격리 수준을 다룬다.
- 글로벌 락, 테이블 락, 네임드 락, 메타데이터 락, 인덱스 기반 잠금 범위, 팬텀 리드 해석이 핵심이다.
- 동시성 버그, 레이스 컨디션, 재고 차감, 중복 처리, 배치와 온라인 트래픽 충돌 문제를 볼 때 바로 연결된다.

### 06장 데이터 압축 (pp. 184-194)

- 페이지 압축과 테이블 압축의 차이, `KEY_BLOCK_SIZE`, 버퍼 풀 적재 방식, 압축 관련 설정을 설명한다.
- 저장 공간 절감만 보면 쉬워 보이지만 CPU 비용, I/O 패턴, 버퍼 풀 효율까지 같이 봐야 한다는 점이 핵심이다.
- 저장소 비용 절감이나 대용량 테이블 운영에서만 선택적으로 참고하면 된다.

### 07장 데이터 암호화 (pp. 195-212)

- 서버 데이터 암호화, 2단계 키 관리, `keyring_file` 플러그인, 테이블 암호화, 언두/리두 로그 암호화, 바이너리 로그 암호화를 다룬다.
- 성능과 운영 복잡도가 같이 늘어나므로, 보안 요구사항이 있을 때 어떤 범위까지 암호화할지 판단하는 기준점으로 좋다.
- 저장 데이터 보호, 백업/로그 노출 위험, 감사 대응이 필요한 환경에서 우선 참고한다.

### 08장 인덱스 (pp. 213-281)

- 디스크 읽기 방식, 랜덤 I/O와 순차 I/O, B-Tree 인덱스, R-Tree, 전문 검색 인덱스, 함수 기반 인덱스, 멀티 밸류 인덱스, 클러스터링 인덱스, 유니크 인덱스, 외래키를 다룬다.
- 이 책의 핵심 장 중 하나다. 인덱스의 구조와 읽기 패턴, 정렬/스캔 방향, 다중 칼럼 인덱스의 효율을 이해해야 쿼리 튜닝이 가능해진다.
- 성능 최적화의 1차 판단은 대부분 이 장에서 해결된다.

### 09장 옵티마이저와 힌트 (pp. 282-392)

- 쿼리 실행 절차, 풀 스캔, 풀 인덱스 스캔, 병렬 처리, `ORDER BY`, `GROUP BY`, `DISTINCT`, 내부 임시 테이블, 조인 최적화, 힌트를 다룬다.
- 힌트는 문제 해결용 도구이지 기본 해법이 아니라는 관점으로 읽는 것이 좋다. 먼저 옵티마이저가 왜 그런 선택을 했는지 이해해야 한다.
- 느린 조회, 정렬 비용, 임시 테이블 남발, 조인 순서 문제를 볼 때 가장 직접적으로 도움이 된다.

### 10장 실행 계획 (pp. 393-516)

- 통계 정보, 히스토그램, 코스트 모델, 실행 계획 출력 포맷, 실행 시간 확인, `EXPLAIN` 주요 칼럼 해석을 정리한다.
- `id`, `select_type`, `table`, `partitions`, `type`, `possible_keys`, `key`, `key_len`, `ref`, `rows`, `filtered`, `Extra`를 읽는 법이 핵심이다.
- 8장과 9장의 내용을 실제 쿼리 분석으로 연결하는 장이라서, 인덱스와 옵티마이저 학습 후 반복해서 보는 장이다.

## 실무 관점 핵심 정리

### 1. MySQL은 "SQL 문법"보다 "엔진 동작"을 이해해야 운영이 가능하다

- 같은 SQL이어도 스토리지 엔진, 인덱스 설계, 통계 정보 상태에 따라 성능과 잠금 범위가 크게 달라진다.
- 특히 InnoDB의 버퍼 풀, 리두/언두, MVCC, 클러스터링 인덱스를 이해해야 장애 대응과 튜닝이 가능하다.

### 2. 잠금 범위는 쿼리 문장보다 접근 경로에 더 크게 좌우된다

- 인덱스를 어떻게 타는지에 따라 레코드 잠금 범위와 대기 양상이 바뀐다.
- 따라서 잠금 이슈는 트랜잭션 코드만 보지 말고 인덱스와 실행 계획을 같이 봐야 한다.

### 3. 인덱스 튜닝은 랜덤 I/O를 줄이는 설계 문제다

- B-Tree 인덱스의 효율은 선택도, 정렬 방향, 다중 칼럼 순서, 커버링 여부에 영향을 받는다.
- 인덱스 추가는 정답이 아니라 쓰기 비용과 저장 비용을 동반하는 트레이드오프다.

### 4. 옵티마이저 결과는 통계 정보 품질에 의존한다

- 히스토그램, 인덱스 통계, 코스트 모델이 잘못되면 실행 계획도 잘못 잡힌다.
- 힌트를 넣기 전에 통계 정보와 데이터 분포를 먼저 의심하는 습관이 필요하다.

### 5. 운영 설정 변경은 재현 가능하게 남겨야 한다

- 임시 세션 변경과 영구 변경을 구분해야 한다.
- `SET PERSIST`와 설정 파일 기반 변경의 차이를 이해하면 장애 대응 후 설정 드리프트를 줄일 수 있다.

### 6. 압축과 암호화는 기능이 아니라 비용 구조 변경이다

- 저장 공간, 보안성은 좋아질 수 있지만 CPU 사용량, I/O 패턴, 운영 복잡도가 함께 바뀐다.
- 워크로드와 규제 요구를 근거로 선택해야 한다.
