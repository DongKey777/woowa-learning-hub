## 핵심 개념

**LRU(Least Recently Used)** 는 캐시가 꽉 찼을 때 **가장 오래 사용되지 않은 항목을 제거**하는 교체 정책이다.
"최근에 사용한 것은 곧 또 쓸 가능성이 높다"는 시간 지역성 원칙에 기반한다.

입문자가 헷갈리는 부분은 이것이다.

> "LRU는 오래된 것을 버린다는 게 삽입 순서를 기준으로 하나요?"

아니다. 삽입 시각이 아니라 **마지막으로 접근(읽기/쓰기)한 시각**을 기준으로 한다.
오래전에 넣었더라도 방금 읽었다면 제거 대상이 아니다.

## 한눈에 보기

```text
캐시 용량: 3

put(1) → [1]
put(2) → [1, 2]
put(3) → [1, 2, 3]  (가득 참)
get(1) → 1 조회, 1이 가장 최근 사용됨 → [2, 3, 1]
put(4) → 4 삽입, 가장 오래 사용 안 한 2 제거 → [3, 1, 4]
```

| 연산 | 동작 |
|---|---|
| `get(key)` | key 조회, 없으면 -1. 있으면 해당 항목을 "최근 사용"으로 갱신 |
| `put(key, value)` | 삽입/갱신. 용량 초과 시 가장 오래 미사용 항목 제거 |

## 상세 분해

LRU 캐시를 O(1)으로 구현하려면 두 자료구조가 필요하다.

- **해시맵 (HashMap)**: key → 노드 위치를 O(1)로 찾는다.
- **양방향 연결 리스트 (Doubly Linked List)**: 최근 사용 순서를 유지한다. 맨 앞이 가장 최근, 맨 뒤가 가장 오래된 항목이다.

조회(get) 흐름:

1. 해시맵으로 노드를 O(1)에 찾는다.
2. 해당 노드를 리스트 맨 앞으로 이동한다(가장 최근 사용).

삽입(put) 흐름:

1. 이미 있는 key면 값 갱신 후 맨 앞으로 이동.
2. 없는 key면 맨 앞에 새 노드 추가.
3. 용량을 초과하면 리스트 맨 뒤 노드(가장 오래된)를 제거하고 해시맵에서도 삭제.

## 흔한 오해와 함정

- **오해: LRU는 삽입 순서로 정렬된다.**
  삽입 순서가 아니라 접근 순서가 기준이다. get()을 호출해도 순서가 바뀐다.

- **오해: 단순 HashMap으로 LRU를 구현할 수 있다.**
  HashMap은 접근 순서를 추적하지 않는다. Java에서는 `LinkedHashMap`의 `accessOrder=true` 옵션을 쓰거나, 직접 양방향 연결 리스트를 구현해야 한다.

- **함정: 스레드 안전성.**
  일반 `LinkedHashMap` 기반 LRU는 멀티스레드 환경에서 안전하지 않다. 동시성이 필요하면 별도의 동기화를 추가해야 한다.

## 실무에서 쓰는 모습

**API 응답 캐시** — DB 조회 결과를 LRU 캐시에 담아두면, 같은 요청이 반복될 때 DB를 거치지 않고 캐시에서 바로 반환한다. 자주 조회되는 상품 정보, 사용자 프로필 등에서 활용된다.

**웹 브라우저 이미지 캐시** — 최근에 본 이미지를 메모리에 유지하고, 메모리가 부족하면 오래 보지 않은 이미지를 먼저 버린다.

## 면접/시니어 질문 미리보기

1. **LRU 캐시를 O(1)으로 구현하려면 어떤 자료구조가 필요한가요?**
   해시맵과 양방향 연결 리스트를 조합한다. 해시맵은 key → 노드 위치를 O(1)에 찾고, 연결 리스트는 접근 순서를 O(1)에 갱신한다.

2. **LRU 외에 어떤 캐시 교체 정책이 있나요?**
   LFU(Least Frequently Used, 가장 적게 접근한 항목 제거), FIFO(먼저 들어온 항목 먼저 제거), Random 등이 있다. LRU는 구현 난이도와 실제 효과의 균형이 좋아 가장 자주 쓰인다.

3. **캐시 히트율이 낮다면 어떻게 진단하나요?**
   캐시 용량이 너무 작거나, 접근 패턴이 시간 지역성이 낮거나(무작위 접근이 많거나), TTL이 너무 짧은 경우를 먼저 의심한다.

## 한 줄 정리

LRU 캐시는 "가장 오래 안 쓴 것을 먼저 버린다"는 정책을 해시맵 + 양방향 연결 리스트로 O(1)에 구현한 구조다.
