## 핵심 개념

재귀(Recursion)는 함수가 자기 자신을 호출해 문제를 해결하는 기법이다. 큰 문제를 같은 구조의 더 작은 문제로 쪼갤 수 있을 때 자연스럽게 쓸 수 있다.

재귀가 제대로 동작하려면 두 가지가 필요하다.
- **종료 조건(base case)**: 더 이상 쪼개지 않고 바로 답을 돌려주는 조건. 없으면 함수가 끝없이 자기 자신을 호출하다 스택 오버플로우가 난다.
- **재귀 호출(recursive case)**: 문제를 더 작게 만들어 자기 자신을 다시 부르는 부분. 매 호출마다 반드시 base case에 가까워져야 한다.

입문자가 가장 자주 저지르는 실수는 base case를 빠뜨리거나, 재귀 호출에서 문제가 작아지지 않아 무한 루프처럼 쌓이는 경우다.

## 한눈에 보기

팩토리얼을 예로 보자.

```
factorial(4)
  = 4 * factorial(3)
       = 3 * factorial(2)
            = 2 * factorial(1)
                 = 1  ← base case
```

| 항목 | 설명 |
|---|---|
| base case | n == 1 이면 1 반환 |
| recursive case | n * factorial(n-1) |
| 호출 깊이 | n에 비례 |
| 스택 공간 | O(n) |

## 상세 분해

- **base case를 먼저 쓴다**: 코드 상단에 `if n == 1: return 1` 처럼 가장 작은 경우를 먼저 처리한다. 그래야 무한 호출을 막는다.
- **재귀 호출마다 n을 줄인다**: `factorial(n-1)` 처럼 인자가 줄어야 언젠가 base case에 도달한다.
- **콜 스택**: 함수를 호출할 때마다 현재 상태(지역 변수, 리턴 주소)가 스택 프레임에 쌓인다. 깊이가 너무 크면 메모리가 부족해진다.
- **반환 경로**: base case에 닿으면 결과가 역순으로 올라오며 각 프레임에서 계산이 완료된다.

## 흔한 오해와 함정

- "재귀는 반복문보다 항상 느리다" → 틀렸다. 트리 순회처럼 자연스럽게 재귀로 표현되는 문제는 반복문으로 바꾸면 오히려 코드가 복잡해진다.
- "재귀면 무조건 DP다" → 재귀는 표현 방법이고, DP는 중복 계산을 저장하는 최적화다. 재귀만으로도 충분히 빠른 문제가 많다.
- 피보나치를 순수 재귀로 짜면 지수 시간이 걸린다. 중복 계산이 많은 재귀에는 메모이제이션이 필요하다.
- Java에서 기본 스택 깊이는 수천 프레임 수준이다. n이 10만이 넘는 재귀는 반복문이나 명시적 스택으로 바꿔야 한다.

## 실무에서 쓰는 모습

1. **트리 순회**: `inorder(node)` 처럼 왼쪽 자식 → 현재 노드 → 오른쪽 자식을 재귀로 표현하면 반복문보다 읽기 쉽다.
2. **파일 시스템 탐색**: 디렉터리 안에 디렉터리가 반복되는 구조는 재귀가 가장 자연스럽다. base case는 "파일이면 바로 처리".

## 면접/시니어 질문 미리보기

- "재귀와 반복문의 차이는 무엇인가요?" → 재귀는 콜 스택을 써서 자기 자신을 호출하고, 반복문은 조건이 바뀌면서 루프를 돈다. 재귀는 표현이 직관적이지만 스택 공간을 O(n) 쓴다.
- "스택 오버플로우가 왜 발생하나요?" → 재귀 호출마다 스택 프레임이 쌓이는데, base case 없이 무한 호출되거나 입력이 너무 크면 프레임이 가득 찬다.
- "꼬리 재귀(tail recursion)란 무엇인가요?" → 재귀 호출이 함수의 마지막 연산인 경우. 일부 언어·컴파일러는 이를 반복문으로 최적화해 스택 공간을 O(1)로 줄인다. Java는 기본 지원하지 않는다.

## 한 줄 정리

재귀는 "base case에서 값을 돌려주고, 나머지는 자기 자신에게 위임" 구조이며, 종료 조건과 축소 방향이 명확하면 트리·그래프·분할 정복 어디서든 쓸 수 있다.
