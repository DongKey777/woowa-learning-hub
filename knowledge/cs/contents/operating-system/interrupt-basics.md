## 먼저 잡는 멘탈 모델

인터럽트는 "CPU를 부르는 초인종"이라고 생각하면 이해가 빠르다.

- 폴링은 CPU가 계속 문을 열어 확인하는 방식이다.
- 인터럽트는 장치가 필요할 때만 CPU를 호출하는 방식이다.

즉, 인터럽트의 목적은 **CPU 시간을 낭비하지 않으면서 이벤트를 제때 처리하는 것**이다.

## 핵심 개념

CPU가 코드를 실행하는 도중 키보드 입력이 들어오거나, 네트워크 패킷이 도착하거나, 타이머가 울릴 수 있다. 이런 이벤트를 CPU가 놓치지 않고 즉시 처리하기 위한 메커니즘이 **인터럽트**다.

인터럽트가 발생하면 CPU는 현재 실행 중인 명령의 위치(프로그램 카운터)와 레지스터를 저장하고, 운영체제가 미리 등록해둔 **인터럽트 핸들러(interrupt handler)**로 점프한다. 핸들러 처리가 끝나면 저장했던 위치로 복귀한다.

## 한눈에 보기

인터럽트 처리 흐름을 단계별로 보면 다음과 같다.

```
CPU 실행 중
  │
  ├─ 타이머 인터럽트 신호 도착
  │
  ├─ 레지스터 + PC 저장 (커널 스택에)
  │
  ├─ 인터럽트 핸들러 실행 (커널 모드)
  │     예: 타이머 → 스케줄러 호출
  │
  └─ 저장했던 레지스터 + PC 복원 → 이전 작업 재개
```

핵심은 두 가지다. 첫째, CPU는 현재 실행 위치를 저장한 뒤 핸들러로 점프한다. 둘째, 핸들러가 끝나면 저장된 위치로 돌아와 중단 전 작업을 이어 간다. 이 흐름이 있기 때문에 프로그램은 인터럽트가 발생한 사실을 대부분 모른 채 실행을 이어 간다.

인터럽트 종류별 발생 주체:

| 종류 | 발생 주체 | 예시 |
|------|-----------|------|
| 하드웨어 인터럽트 | 외부 장치 | 키보드, 네트워크 카드, 타이머 |
| 소프트웨어 인터럽트(트랩) | 프로그램 | 시스템 콜 |
| 예외 | CPU 자체 | 0으로 나누기, 잘못된 메모리 접근 |

자주 헷갈리는 세 용어를 "발생 타이밍" 기준으로 다시 보면:

| 용어 | 발생 타이밍 | 대표 예시 | 초급자 체크 포인트 |
|------|-------------|-----------|---------------------|
| 인터럽트(Interrupt) | 현재 명령과 무관하게 비동기 발생 | NIC 패킷 도착, 타이머 틱 | "장치가 CPU를 호출" |
| 트랩(Trap) | 프로그램이 의도적으로 동기 발생 | 시스템 콜 진입 | "유저 코드가 커널 진입 요청" |
| 예외(Exception) | 현재 명령 실행 중 동기 발생 | divide-by-zero, page fault | "명령 실행 결과로 발생" |

자주 같이 비교하는 폴링과의 차이:

| 방식 | 누가 시작하나 | CPU 사용 패턴 | 적합한 상황 |
|------|----------------|---------------|-------------|
| 인터럽트 | 장치가 이벤트 발생 시 CPU 호출 | 필요할 때만 처리 | 일반적인 I/O 이벤트 처리 |
| 폴링 | CPU가 주기적으로 상태 확인 | 준비되지 않아도 반복 확인 | 매우 짧은 지연이 필요한 일부 특수 경로 |

초보자가 가장 자주 섞는 세 개념을 한 번에 놓고 보면:

| 비교 축 | 누가 시작하나 | 무엇이 바로 일어나나 | 항상 다른 스레드가 실행되나 |
|---------|---------------|----------------------|------------------------------|
| 인터럽트 | 장치나 타이머 같은 외부/커널 사건 | CPU가 현재 흐름을 잠깐 멈추고 핸들러로 들어간다 | 아니다 |
| 시스템 콜 | 현재 실행 중인 유저 코드 | 유저 모드에서 커널 모드로 요청을 보낸다 | 아니다 |
| 컨텍스트 스위치 | 스케줄러 판단 | CPU가 다른 태스크 상태를 복원한다 | 보통 그렇다 |

짧게 말하면, 인터럽트와 시스템 콜은 "커널로 들어가는 계기"에 가깝고, 컨텍스트 스위치는 "CPU 실행 대상을 바꾸는 결과"다. 그래서 인터럽트나 시스템 콜이 있어도 같은 태스크로 돌아오면 컨텍스트 스위치는 없을 수 있다.

## 상세 분해

- **하드웨어 인터럽트**: 외부 장치(디스크, 네트워크 카드, 키보드, 타이머)가 발생시킨다. 비동기적으로 언제든 발생할 수 있다.
- **소프트웨어 인터럽트(트랩, Trap)**: 프로그램이 의도적으로 발생시키는 인터럽트. 시스템 콜이 대표적이다. `read()`, `write()` 같은 시스템 콜을 호출하면 CPU는 트랩을 발생시켜 커널 모드로 전환한다.
- **예외(Exception)**: 0으로 나누기, 잘못된 메모리 접근(segfault) 처럼 CPU 실행 중 오류 상황에서 발생한다. 하드웨어 인터럽트와 달리 현재 명령 때문에 발생한다.
- **인터럽트 벡터 테이블**: 인터럽트 번호와 핸들러 주소를 매핑한 테이블. OS 부팅 시 초기화된다.

## 아주 작은 예시: 타이머 인터럽트와 스케줄링

하나의 코어에서 스레드 A가 오래 실행 중이라고 하자.

1. 주기 타이머가 인터럽트를 발생시킨다.
2. 커널이 현재 레지스터/PC를 저장하고 타이머 핸들러를 실행한다.
3. 스케줄러가 "A를 계속 돌릴지, 대기 중인 B로 바꿀지" 판단한다.
4. 선택된 스레드의 문맥으로 복귀한다.

이 흐름 때문에 "인터럽트 -> 스케줄러 판단 -> (필요 시) 문맥 전환"이 연결된다.

추가로, "키보드 입력 처리"를 같은 프레임으로 보면:

1. 키 입력 발생 -> 키보드 인터럽트.
2. 커널이 입력 버퍼를 갱신.
3. 애플리케이션이 `read()` 중이면 깨어나 데이터를 받음.

즉, 앱 입장에서는 단순 `read()` 호출처럼 보여도, 아래에서는 인터럽트가 이벤트 전달을 만든다.

## 흔한 오해와 함정

- "인터럽트가 발생하면 프로그램이 죽는다"는 오해다. 대부분의 인터럽트는 투명하게 처리되고 프로그램은 그 사실을 모른 채 이어서 실행된다. 단, 처리할 핸들러가 없거나 복구 불가능한 예외는 프로세스를 종료시킬 수 있다.
- "시스템 콜 = 함수 호출"이라는 단순화는 틀렸다. 시스템 콜은 트랩을 통해 커널 모드로 전환하는 과정이 포함되며, 일반 함수보다 훨씬 비용이 크다.
- "`인터럽트`, `시스템 콜`, `컨텍스트 스위치`가 전부 같은 커널 이벤트"라는 오해도 흔하다. 인터럽트는 외부 사건, 시스템 콜은 현재 코드의 요청, 컨텍스트 스위치는 스케줄러가 실행 대상을 바꾸는 일이라 질문 축이 다르다.
- "인터럽트는 커널만 사용한다"도 틀렸다. 신호(Signal) 형태로 유저 공간까지 전달된다. 예: `SIGINT`(Ctrl+C)는 키보드 인터럽트 → 커널 → 프로세스에 시그널 전달 경로로 온다.
- "인터럽트가 오면 항상 문맥 전환된다" — 항상은 아니다. 핸들러 후 같은 스레드로 복귀할 수도 있고, 스케줄러가 다른 스레드를 고르면 그때 문맥 전환이 일어난다.
- "예외는 전부 비정상" — page fault처럼 메모리 매핑을 완성하기 위한 정상 예외도 있다.

## 실무에서 쓰는 모습

백엔드 개발자가 가장 자주 만나는 인터럽트 관련 개념은 **타이머 인터럽트**다. 운영체제 스케줄러는 타이머 인터럽트를 기준으로 실행 중인 스레드를 바꾼다. 즉, CPU 스케줄링의 "틱(tick)"이 타이머 인터럽트다.

또한 NIC(네트워크 카드)가 패킷을 받으면 하드웨어 인터럽트가 발생해 커널 네트워크 스택이 깨어난다. I/O가 완료될 때 애플리케이션이 통지를 받는 구조의 기반이 여기에 있다.

## 면접/시니어 질문 미리보기

1. "시스템 콜과 인터럽트의 관계를 설명하세요."
   - 핵심 답: 시스템 콜은 소프트웨어 인터럽트(트랩)를 통해 구현된다. 호출 시 CPU가 유저 모드에서 커널 모드로 전환한다.
2. "인터럽트와 폴링(Polling)의 차이는 무엇인가요?"
   - 핵심 답: 폴링은 CPU가 주기적으로 장치 상태를 확인하는 방식이라 CPU를 계속 점유한다. 인터럽트는 장치가 준비됐을 때만 CPU에 알려 효율적이다.
3. "Page fault도 인터럽트의 일종인가요?"
   - 핵심 답: 네. Page fault는 CPU가 잘못된 또는 아직 매핑 안 된 주소에 접근할 때 발생하는 예외(동기적 소프트웨어 인터럽트)다.

## Self-check (자가 점검 5문항)

아래 질문은 시험이 아니라, 지금 이해한 축을 말로 꺼내 보고 다음 문서를 고르기 위한 점검이다. 먼저 짧게 답해 보고, 막히면 바로 아래 `힌트`만 확인해 보자.

1. 인터럽트와 폴링의 차이를 "누가 먼저 행동을 시작하나" 관점으로 설명할 수 있는가?
   힌트: 인터럽트는 장치가 CPU를 부르고, 폴링은 CPU가 주기적으로 장치를 확인하러 간다.
2. 하드웨어 인터럽트(비동기)와 예외/트랩(동기) 차이를 예시 하나씩 들어 구분할 수 있는가?
   힌트: 키보드 입력은 외부 이벤트라 비동기고, page fault나 divide-by-zero는 현재 명령 실행 중 생겨 동기적이다.
3. 인터럽트가 발생해도 항상 문맥 전환이 일어나는 것은 아니라는 점을 설명할 수 있는가?
   힌트: 커널이 잠깐 처리만 하고 같은 태스크로 바로 돌아오면 인터럽트는 있었어도 스케줄 교체는 없을 수 있다.
4. `read()` 호출, NIC 패킷 도착, 다른 스레드로 실행 전환을 각각 시스템 콜, 인터럽트, 컨텍스트 스위치 중 어디에 놓을지 구분할 수 있는가?
   힌트: `read()`는 요청 입구, 패킷 도착은 외부 사건, 다른 스레드 실행은 스케줄러가 CPU 대상을 바꾸는 결과다.
5. `read()` 같은 단순 API 호출 뒤에 인터럽트 기반 이벤트 전달 경로가 있다는 점을 말할 수 있는가?
   힌트: 앱은 `read()`만 보지만, 그 아래에서는 장치 완료 신호와 커널 깨움 경로가 이어져 있다.

## Self-check 후 어디로 갈까?
