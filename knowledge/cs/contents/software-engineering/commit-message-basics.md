## 핵심 개념

커밋 메시지는 코드 변경 이력의 제목이다. 잘 쓴 커밋 메시지는 `git log`만 봐도 프로젝트가 어떤 맥락에서 발전했는지 알 수 있다. 반대로 `fix`, `수정`, `작업` 같은 메시지만 있으면 나중에 버그 원인을 추적하거나 변경 이유를 파악할 때 코드를 전부 열어봐야 한다.

입문자가 헷갈리는 부분은 "어떤 type을 써야 하고, 제목을 얼마나 길게 써야 하냐"다.

## 한눈에 보기

| type | 언제 쓰는가 |
|---|---|
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변경 없는 코드 개선 |
| `test` | 테스트 추가/수정 |
| `chore` | 빌드 설정, 의존성 등 기타 |
| `docs` | 문서 변경 |
| `style` | 포맷팅, 세미콜론 등 비기능 변경 |

## 상세 분해

**기본 형식**

```
type(scope): subject

body (선택)

footer (선택)
```

- `type` — 변경 종류. 위 표 참고.
- `scope` — 변경된 모듈이나 컴포넌트. 예: `feat(order): 주문 취소 기능 추가`
- `subject` — 무엇을 했는지 50자 이하로. 명령형(동사 원형)으로 쓴다. 마침표 없음.
- `body` — 왜 변경했는지, 어떤 방식을 택했는지 (선택 사항).

**좋은 subject vs 나쁜 subject**

- 나쁨: `수정`, `작업중`, `버그 고침`, `fix`
- 좋음: `feat: 주문 금액 0원 이하 입력 시 예외 발생`, `fix(user): 이메일 중복 검사 누락 수정`

**body 활용**

큰 변경이라면 body에 배경을 쓴다. "기존 방식의 문제는 X였고, Y로 바꾼 이유는 Z다"처럼 의사결정 맥락을 남기면 미래의 팀원에게 큰 도움이 된다.

## 흔한 오해와 함정

- "커밋 메시지는 나만 보는 거다"라는 생각이 있다. 팀 저장소에서 `git blame`, `git log --oneline`은 모든 팀원이 사용한다. 메시지 품질은 팀 전체 생산성에 영향을 준다.
- "feat와 refactor를 구분하기 어렵다"는 경우가 많다. 기준은 **외부 동작이 바뀌었는가**다. 동작이 바뀌면 `feat` 또는 `fix`, 동작이 같고 내부만 바뀌면 `refactor`다.
- "하나의 커밋에 여러 type이 섞이는 경우"는 커밋을 더 작게 나누는 신호다. feat + fix가 한 커밋에 있다면 두 커밋으로 분리하는 편이 이력 추적에 유리하다.

## 실무에서 쓰는 모습

우아한형제들 미션에서 PR을 올릴 때 커밋 이력이 리뷰 대상이 되기도 한다. 리뷰어가 "커밋 단위가 너무 크거나 메시지가 모호해요"라는 피드백을 남기는 경우가 있다. 기능 하나, 테스트 하나를 한 커밋 단위로 잡고 `feat: ~기능 추가` → `test: ~기능 단위 테스트 추가` 패턴을 유지하면 이력이 깔끔해진다.

## 면접/시니어 질문 미리보기

- "커밋 메시지에서 가장 중요한 요소는 무엇인가요?" — 'why'다. 무엇을 바꿨는지는 diff를 보면 알 수 있지만, 왜 그렇게 결정했는지는 메시지와 body에서만 찾을 수 있다.
- "커밋을 얼마나 자주, 얼마나 작게 해야 하나요?" — 논리적 단위 하나(기능 하나, 버그 수정 하나)가 이상적인 크기다. 너무 잦으면 노이즈가 많아지고, 너무 크면 리뷰와 롤백이 어려워진다.
- "이미 push한 커밋 메시지를 고치는 방법이 있나요?" — `git rebase -i`로 과거 커밋을 수정할 수 있다. 단, 공유 브랜치에서는 팀원과 협의 없이 rewrite하면 이력 충돌이 생긴다.

## 한 줄 정리

커밋 메시지는 코드 변경의 '왜'를 남기는 최소 문서이며, type(scope): subject 구조로 팀이 이력을 빠르게 탐색할 수 있게 한다.
