## 핵심 개념

리팩토링(Refactoring)은 소프트웨어의 **외부 동작을 바꾸지 않으면서** 내부 구조를 개선하는 과정이다. Martin Fowler가 정의한 이 개념의 핵심은 "동작 보존"이다.

입문자가 자주 혼동하는 패턴:
- 기능을 추가하면서 코드도 정리했다 → 리팩토링이 아니라 기능 개발 + 정리의 혼합
- 버그를 고치면서 구조도 바꿨다 → 위험한 혼합, 어떤 변경이 버그를 고쳤는지 추적 불가

리팩토링은 **별도 커밋**으로 분리하는 것이 안전하다.

## 한눈에 보기

| 자주 쓰는 기법 | 목적 | 적용 신호 |
|---|---|---|
| 메서드 추출 (Extract Method) | 긴 메서드를 작은 단위로 분리 | 메서드가 20줄 이상, 주석으로 구분된 블록 |
| 변수명·메서드명 변경 | 의도를 코드에 드러내기 | `tmp`, `data` 같은 의미 없는 이름 |
| 매직 넘버 상수화 | 리터럴 숫자를 이름 있는 상수로 | `if (score > 60)` → `PASSING_SCORE` |
| 중복 코드 제거 | 같은 로직이 두 곳 이상에 있을 때 | 복사-붙여넣기 흔적 |
| 조건 단순화 | 복잡한 중첩 if를 명확하게 | 3단계 이상 중첩된 조건 |

## 상세 분해

**코드 냄새(Code Smell)**

리팩토링의 시작은 냄새를 발견하는 것이다. 대표적인 냄새들:
- **긴 메서드**: 한 번에 너무 많은 일을 하는 메서드
- **중복 코드**: 같거나 비슷한 코드가 여러 곳에 있음
- **의미 없는 이름**: `process()`, `data`, `tmp`처럼 의도를 드러내지 않는 이름
- **주석으로 설명이 필요한 코드**: 코드 자체가 의도를 전달하지 못한다는 신호

**테스트 없는 리팩토링의 위험**

리팩토링은 동작을 바꾸지 않아야 하는데, 테스트 없이 하면 "동작이 유지됐는지" 확인할 방법이 없다. 먼저 테스트를 작성하거나, 기존 테스트를 실행해 모두 통과하는 상태에서 리팩토링을 시작한다.

**보이스카우트 규칙**

"캠핑장은 왔을 때보다 깨끗하게 남겨라"에서 따온 원칙이다. 코드를 수정할 때 해당 파일 주변을 조금씩 개선하면, 코드베이스가 시간이 지날수록 건강해진다. 완벽한 리팩토링이 아니어도 된다.

## 흔한 오해와 함정

- "리팩토링은 기능 개발 다 끝나고 한다"는 생각이 있다. 리팩토링은 개발 과정 내내 작은 단위로 수행하는 것이 훨씬 안전하다. 쌓인 후에 한꺼번에 하면 위험이 크다.
- "깨끗한 코드를 위해 전체를 다시 짜자"는 충동도 흔하다. 동작하는 코드를 전면 재작성하면 숨어 있던 버그 수정 경험과 엣지 케이스 처리가 사라진다. 점진적 개선이 낫다.
- "리팩토링하면 성능이 저하된다"는 오해도 있다. 메서드 추출 수준의 리팩토링은 JIT 컴파일러가 최적화하므로 실제 성능 영향은 거의 없다.

## 실무에서 쓰는 모습

`OrderService.createOrder()` 메서드가 검증, 재고 확인, 결제, 저장을 모두 담당해 100줄이 됐다면, 각 블록을 `validateOrder()`, `checkStock()`, `processPayment()`, `saveOrder()`로 추출한다.

추출 전에 기존 테스트를 실행해 전부 통과함을 확인하고, 추출 후에도 동일하게 통과하는지 재확인한다. 이 두 단계가 "외부 동작 유지"를 보장한다.

## 면접/시니어 질문 미리보기

- "리팩토링과 기능 추가를 같은 커밋에 넣으면 안 되는 이유는?" — 동작 변경과 구조 변경이 섞이면 무엇이 버그를 일으켰는지 추적하기 어렵고, 롤백 범위도 커진다.
- "레거시 코드를 리팩토링할 때 가장 먼저 해야 할 것은?" — 테스트를 먼저 추가해 안전망을 만든다. 테스트 없이 리팩토링하면 동작 보존을 확인할 방법이 없다.
- "코드 냄새가 있어도 리팩토링하지 않아야 하는 경우는?" — 릴리스 직전이나 동작하는 코드가 수년 된 레거시라면 위험 대비 이득을 따져야 한다. 냄새가 있어도 당장 변경 필요성이 없다면 미루는 선택도 유효하다.

## 한 줄 정리

리팩토링은 테스트로 동작을 보호하면서 코드 냄새를 조금씩 제거하는 지속적 개선이지, 한 번에 다 고치는 재작성이 아니다.
