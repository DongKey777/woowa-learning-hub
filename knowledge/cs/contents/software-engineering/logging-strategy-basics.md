## 핵심 개념

로그(Log)는 애플리케이션이 실행되는 동안 발생한 사건을 기록한 텍스트다. 개발 중엔 `System.out.println`으로 확인하지만, 운영 환경에서는 레벨 관리, 성능, 검색 가능성을 위해 로깅 프레임워크를 써야 한다.

입문자가 가장 헷갈리는 부분은 "어디에 로그를 찍어야 하고, 어떤 레벨을 써야 하는가"다.

## 한눈에 보기

| 레벨 | 언제 쓰는가 | 운영 환경 |
|---|---|---|
| TRACE | 코드 흐름 추적, 매우 상세 | 끔 |
| DEBUG | 개발 중 상태 확인 | 보통 끔 |
| INFO | 정상 이벤트 기록 (요청, 주문 생성) | 켬 |
| WARN | 잠재 문제, 아직 오류는 아님 | 켬 |
| ERROR | 예외 발생, 처리 실패 | 켬 + 알림 |

## 상세 분해

**레벨 선택 기준**

INFO는 비즈니스 이벤트(주문 생성, 결제 완료)에 쓴다. DEBUG는 개발 시 값 확인에 쓰고 운영에서는 끈다. ERROR는 예외가 발생했을 때, 특히 복구 불가능한 상황에 쓴다. WARN은 재시도로 해결됐지만 반복되면 문제가 될 수 있는 상황에 쓴다.

**로그를 남겨야 할 위치**

- 시스템 경계(HTTP 요청/응답, 외부 API 호출)
- 비즈니스 핵심 흐름(주문 상태 변경, 결제 시작/완료)
- 예외 처리 지점

루프 내부나 매 DB 호출마다 INFO를 찍으면 로그 폭탄이 된다.

**구조화 로그**

로그를 검색 도구(ELK, CloudWatch)에서 분석할 때는 구조화 로그가 유리하다. `userId=1234 orderId=5678 action=payment_complete`처럼 키-값 형태로 담으면 필터링이 쉬워진다.

## 흔한 오해와 함정

- "로그를 많이 찍으면 디버깅이 쉬워진다"는 오해가 있다. 불필요한 로그가 많으면 중요한 로그를 찾기 어려워진다. 핵심 흐름에만 INFO를 두고 나머지는 DEBUG로 낮춘다.
- "예외를 catch해서 로그만 찍고 끝내면 된다"는 생각도 위험하다. 예외를 삼키면 호출자가 실패를 알 수 없다. catch 후 반드시 적절한 처리(재시도, 상위 전파, 에러 응답)를 해야 한다.
- "`System.out.println`도 로그 아닌가"라는 의문도 있다. `System.out`은 레벨 필터링, 비동기 출력, 포맷 제어가 안 된다. SLF4J + Logback 같은 프레임워크를 써야 운영 환경에서 제어가 가능하다.

## 실무에서 쓰는 모습

Spring Boot 프로젝트에서는 SLF4J + Logback이 기본이다. 컨트롤러에서 요청이 들어오면 INFO로 요청 정보를 남기고, 서비스에서 주요 상태 변경은 INFO, 외부 API 실패는 WARN, 복구 불가한 예외는 ERROR로 남긴다. `application.yml`에서 패키지별 레벨을 조정하면 개발 환경과 운영 환경을 다르게 구성할 수 있다.

## 면접/시니어 질문 미리보기

- "DEBUG 로그를 운영 환경에서 켜면 어떤 문제가 생기나요?" — 로그 출력량이 급증해 I/O 부하가 생기고 디스크 가득 참(disk full) 장애로 이어질 수 있다. 대용량 트래픽 시스템에서는 로그 레벨이 성능에 직접 영향을 준다.
- "로그에 민감 정보를 담으면 안 되는 이유는?" — 로그 파일은 다수의 담당자가 접근하고 외부 로그 수집 서비스로 전송된다. 비밀번호, 개인정보, 결제 카드 번호는 로그에서 마스킹하거나 제외해야 한다.
- "MDC(Mapped Diagnostic Context)가 무엇인가요?" — 요청 단위로 상관 ID(traceId)를 스레드 로컬에 저장해, 요청 하나의 로그를 추적할 수 있게 하는 기법이다. 분산 시스템에서 요청 흐름 추적에 필수다.

## 한 줄 정리

로그는 레벨로 중요도를 구분하고 시스템 경계와 비즈니스 이벤트에 집중해 찍어야 장애 시 빠르게 원인을 찾을 수 있다.
