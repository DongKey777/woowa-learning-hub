## 왜 배포 전략을 따로 생각해야 하는가

코드가 맞아도 배포가 잘못되면 서비스는 깨진다.

- 기능은 정상인데 설정 하나 때문에 장애가 날 수 있다
- 한 번에 전체 트래픽에 적용하면 실패 범위가 너무 크다
- 데이터베이스 변경은 애플리케이션 코드보다 되돌리기 어렵다

즉 배포 전략은 단순한 배포 순서가 아니라 **실패 확률과 실패 비용을 조절하는 설계**다.

---

## 배포와 릴리스는 다르다

두 용어를 분리해서 생각해야 한다.

- 배포(deploy): 새로운 코드를 운영 환경에 올리는 행위
- 릴리스(release): 사용자가 새 기능을 실제로 쓰게 되는 순간

Feature Flag를 쓰면 이 둘을 분리할 수 있다.

- 코드는 먼저 배포한다
- 기능 활성화는 나중에 한다
- 문제가 생기면 기능만 끄고 코드 배포는 유지할 수 있다

이 분리가 중요한 이유는 "배포는 기술 문제, 릴리스는 제품 문제"이기 때문이다.

---

## 대표 전략 비교

배포 전략은 보통 다음 축으로 비교한다.

- 얼마나 빨리 전체 트래픽에 노출되는가
- 실패 시 영향 범위를 얼마나 줄일 수 있는가
- 롤백이 얼마나 쉬운가
- 인프라 비용이 얼마나 드는가

대표 전략:

- Rolling
- Canary
- Blue-Green

각 전략은 정답이 아니라 **상황별 타협안**이다.

---

## Canary 배포

Canary는 전체 트래픽 중 일부만 새 버전으로 보내는 방식이다.

핵심 아이디어:

- 새 버전을 먼저 소수 사용자에게만 노출한다
- 지표가 정상인지 확인한다
- 문제가 없으면 트래픽 비율을 점진적으로 늘린다

장점:

- 장애를 작은 범위에 가둔다
- 실제 사용자 트래픽으로 검증할 수 있다
- 성능 저하나 비정상 응답을 조기에 발견할 수 있다

단점:

- 트래픽 분기와 관측 체계가 필요하다
- 지표 해석을 잘못하면 잘못된 결정을 내릴 수 있다
- 상태가 있는 세션, 캐시, 외부 연동이 있으면 검증이 복잡해진다

Canary에서 봐야 할 지표:

- 5xx 비율
- p95/p99 응답 시간
- 타임아웃 비율
- 핵심 비즈니스 전환율
- 신규 에러 로그와 알림

Canary는 "조금만 바꾸고 관찰한다"는 철학을 구현한 방식이다.

---

## Blue-Green 배포

Blue-Green은 동일한 환경을 두 개 둔 뒤, 트래픽 스위치를 한 번에 전환하는 방식이다.

- Blue: 현재 운영 중인 버전
- Green: 새 버전

장점:

- 전환이 빠르다
- 문제가 생기면 다시 원복하기 쉽다
- 전환 전까지 새 버전을 완전히 검증할 수 있다

단점:

- 같은 환경을 두 벌 유지하므로 비용이 든다
- 데이터베이스 스키마가 바뀌면 단순 전환으로 끝나지 않는다
- 배포는 쉬워도 데이터 호환성은 별도 문제다

Blue-Green이 잘 맞는 경우:

- 무상태(stateless) 서비스
- 트래픽 전환이 명확한 서비스
- 빠른 롤백이 중요한 서비스

즉 Blue-Green은 인프라 전환 문제를 잘 푸는 전략이지, 데이터 진화 문제까지 자동으로 해결하지는 않는다.

## Rolling 배포와 점진 확장

Rolling 배포는 일부 인스턴스를 새 버전으로 바꾸고, 상태를 보면서 나머지를 교체하는 방식이다.

장점:

- 전체 환경을 새로 만들지 않아도 된다
- 인프라 비용이 비교적 낮다
- Kubernetes 같은 오케스트레이션과 잘 맞는다

주의점:

- 이전 버전과 새 버전이 동시에 살아 있는 시간이 있다
- 호환성 없는 변경은 위험하다
- 배포 중간에 요청이 어느 버전으로 갈지 불안정할 수 있다

그래서 Rolling 배포에서는 다음이 중요하다.

- 전후 버전 호환성
- 세션/캐시 처리
- 무중단 스키마 변경

점진 확장은 기술적으로는 Rolling과 비슷해 보이지만, 운영 목표는 더 명확하다.

- 사용자 경험을 망치지 않는다
- 장애가 발생해도 일부에서 멈춘다
- 확장 속도보다 안정성을 우선한다

## 롤백 전략

롤백은 단순히 이전 버전으로 되돌리는 일이 아니다.

롤백에는 세 가지 층위가 있다.

1. 코드 롤백
2. 설정/플래그 롤백
3. 데이터 롤백 또는 보정

가장 쉬운 것은 코드 롤백이지만, 데이터가 이미 변경되었다면 충분하지 않을 수 있다.

### 코드 롤백

- 새 버전에서만 생긴 버그를 이전 버전으로 되돌린다
- 배포 아티팩트가 남아 있어야 한다
- 이전 버전이 현재 데이터와 호환되어야 한다

### 설정 롤백

- Feature Flag를 끄거나
- 라우팅 비율을 줄이거나
- 외부 연동을 비활성화한다

이 방식은 빠르고 위험이 낮다.

### 데이터 롤백

가장 어렵다.

- 이미 저장된 데이터는 코드만 되돌려서는 복구되지 않는다
- 스키마 삭제는 대개 되돌릴 수 없다
- 대량 재처리나 보정 작업이 필요할 수 있다

그래서 실무에서는 "되돌릴 수 있는 변경"보다 **앞으로만 가도 안전한 변경**을 선호한다.

예:

- 컬럼 추가 후 이전 코드와 새 코드를 동시에 지원
- 기존 필드는 바로 삭제하지 않기
- 읽기/쓰기 경로를 분리해 단계적으로 전환하기

즉 좋은 롤백은 "실패 후에 고치는 능력"보다 **실패해도 망가지지 않는 구조**를 만드는 데 가깝다.

## 배포를 안전하게 만드는 운영 장치

배포 전략은 모니터링이 없으면 작동하지 않는다.

필수 장치:

- 구조화된 로그
- 메트릭 대시보드
- 에러 추적
- 배포 알림
- 즉시 끌 수 있는 차단 스위치

관찰 포인트:

- 배포 직후 에러율 변화
- 특정 API만 느려지는지
- 외부 의존성 실패가 늘었는지
- 캐시 미스와 DB 부하가 변했는지

좋은 운영은 "문제가 생기면 알게 되는 것"이 아니라 **문제가 어느 단계에서 생겼는지 좁혀가는 것**이다.

---

## 전환 판단 기준

어떤 전략을 선택할지는 배포 대상의 특성에 따라 달라진다.

- 빠른 검증이 필요하면 Canary
- 빠른 원복이 중요하면 Blue-Green
- 인프라 비용을 아껴야 하면 Rolling
- 사용자 규모가 작고 위험이 낮으면 단순 배포도 가능

판단할 때는 다음 질문이 유용하다.

- 이 변경은 코드만 되돌리면 끝나는가?
- 이전 버전과 새 버전이 동시에 살아도 안전한가?
- 트래픽 일부에만 노출했을 때 의미 있는 신호를 볼 수 있는가?
- 장애가 나면 어떤 레버를 먼저 당길 것인가?

---

## 시니어 관점 질문

- 이 변경은 배포와 릴리스를 분리해야 하는가?
- 롤백이 코드로 끝나는가, 데이터까지 영향을 주는가?
- 새 버전과 구 버전이 동시에 동작해도 안전한가?
- 장애를 줄이는 데 가장 효과적인 제어 지점은 무엇인가?
- 지표가 좋아 보여도 실제 사용자 경험이 나빠질 수 있는가?
