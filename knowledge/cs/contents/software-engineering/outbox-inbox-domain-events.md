## 왜 중요한가

서비스가 하나의 데이터베이스 안에서만 끝나면 트랜잭션 하나로 설명이 된다. 하지만 실제 시스템은 보통 그렇지 않다.

- 주문이 생성되면 결제가 이어지고
- 결제가 완료되면 알림이 나가고
- 재고가 줄어든 뒤 다른 서비스가 반응해야 한다

이때 중요한 건 단순히 “이벤트를 보낸다”가 아니라, **이벤트가 유실되지 않고, 중복에도 안전하며, 도메인 의미를 유지하는가**다.

---

## Domain Event와 Integration Event

두 개는 비슷해 보이지만 목적이 다르다.

### Domain Event

도메인 내부에서 일어난 사실이다.

- `OrderPlaced`
- `PaymentApproved`
- `StockReserved`

핵심은 “무슨 일이 일어났는가”를 도메인 언어로 표현하는 것이다.

### Integration Event

다른 서비스나 외부 시스템과 주고받기 위한 이벤트다.

- 스키마 안정성
- 역호환성
- 외부 소비자 고려

가 중요하다.

실무에서는 도메인 이벤트와 외부 전송용 이벤트를 완전히 같은 것으로 다루지 않는 편이 안전하다. 도메인 모델을 그대로 외부에 노출하면 결합도가 빠르게 커진다.

---

## Outbox Pattern

Outbox Pattern은 **업무 데이터 저장과 이벤트 저장을 같은 트랜잭션으로 묶는 방식**이다.

문제 상황은 단순하다.

- DB 저장은 성공
- 메시지 브로커 발행은 실패

이 순서가 어긋나면 데이터는 바뀌었는데 이벤트가 사라지는 상태가 된다.

그래서 보통은 이렇게 한다.

1. 비즈니스 데이터 저장
2. 같은 트랜잭션에 outbox 테이블에 이벤트 기록
3. 별도 릴레이 프로세스가 outbox를 읽어 브로커에 발행
4. 발행 성공 시 outbox 상태 갱신

이 구조의 장점은 분명하다.

- DB와 이벤트 기록의 원자성을 확보하기 쉽다
- 메시지 유실 가능성을 낮춘다
- 재시도 전략을 두기 쉽다

하지만 대가도 있다.

- 릴레이가 추가된다
- 지연이 생긴다
- 정리 작업이 필요하다

즉 Outbox는 단순 패턴이 아니라 **최소한의 신뢰성을 얻기 위한 운영 설계**다.

---

## Inbox Pattern

Inbox Pattern은 **컨슈머가 받은 이벤트를 먼저 기록하고, 중복 처리 여부를 판단하는 방식**이다.

메시징 환경에서는 보통 이런 문제가 생긴다.

- 같은 메시지가 두 번 도착한다
- 처리 도중 실패해서 재전송된다
- 네트워크 문제로 ack가 꼬인다

그래서 컨슈머는 메시지를 처리하기 전에 inbox에 기록하거나, 처리 완료 키를 저장해서 **idempotency**를 확보한다.

전형적인 흐름은 다음과 같다.

1. 메시지 수신
2. 메시지 ID 확인
3. 이미 처리했으면 무시
4. 아니면 도메인 로직 수행
5. 처리 완료 기록

이 방식의 핵심은 “중복이 절대 발생하지 않는다”는 가정이 아니라, **중복이 와도 결과가 안전하다**는 가정이다.

## 이벤트 설계 기준

이벤트는 단순한 DTO가 아니다. 설계를 잘못하면 나중에 시스템 전체 결합도가 올라간다.

### 1. 이벤트는 사실이어야 한다

`OrderPlaced`는 사실이다.

반면 `SendEmailNow`는 사실이 아니라 명령에 가깝다.

이벤트 이름은 과거형이 더 자연스럽다.

### 2. payload는 작고 안정적이어야 한다

이벤트에는 꼭 필요한 정보만 넣는 편이 좋다.

- 식별자
- 상태 전이의 핵심 데이터
- version
- correlationId / causationId

도메인 객체 전체를 넣는 방식은 외부 결합을 키운다.

### 3. 소비자 기준으로 버전 관리해야 한다

이벤트는 발행자가 아니라 소비자가 깨지지 않도록 관리해야 한다.

- 필드는 추가 가능하게 설계
- 의미 변경은 새 이벤트로 분리
- 삭제는 매우 조심

### 4. 중복과 순서를 기본값으로 생각해야 한다

메시징 시스템은 기본적으로 순서와 정확히 한 번을 보장하지 않는 경우가 많다.

그래서 이벤트 핸들러는:

- 멱등해야 하고
- 순서가 뒤집혀도 안전하거나
- 최소한 순서 의존성을 명시해야 한다

## 일관성 경계와 아키텍처

이벤트 설계는 결국 일관성 경계를 어디에 두느냐의 문제다.

- 강한 일관성이 필요한 곳은 같은 트랜잭션으로 묶는다
- 느슨한 일관성으로 충분한 곳은 이벤트로 분리한다

Layered Architecture에서도 가능하지만, Application Service가 트랜잭션 경계를 잡고 Outbox를 쓰는 구조가 흔하다. Hexagonal Architecture에서는 포트 바깥의 메시지 브로커를 어댑터로 감싸서 도메인과 분리하기 쉽다.

중요한 건 유행하는 패턴 이름이 아니라:

- 어디까지를 동기적으로 보장할지
- 어디부터 비동기 허용할지
- 실패 시 누가 재시도 책임을 질지

를 분명히 정하는 것이다.
