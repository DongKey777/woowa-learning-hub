## 핵심 개념

TDD(Test-Driven Development, 테스트 주도 개발)는 구현 코드를 작성하기 **전에** 테스트를 먼저 작성하는 개발 방식이다. 단순히 테스트를 작성하는 것과 다른 점은, **테스트가 구현의 방향을 결정**한다는 것이다.

TDD의 핵심 리듬은 세 단계다.

- **Red** — 아직 구현이 없어서 실패하는 테스트를 작성한다
- **Green** — 테스트를 통과하는 가장 단순한 코드를 작성한다
- **Refactor** — 동작을 유지하면서 코드를 개선한다

입문자가 헷갈리는 부분은 "테스트를 먼저 작성하면 어떻게 컴파일이 되냐"는 것이다. 컴파일 오류도 Red 상태의 하나로 본다.

## 한눈에 보기

```
[Red]   테스트 작성 → 실패 확인
  ↓
[Green] 최소 코드 구현 → 통과 확인
  ↓
[Refactor] 중복 제거, 이름 개선 → 테스트 여전히 통과
  ↓
다음 테스트로 반복
```

| 단계 | 목적 | 주의 |
|---|---|---|
| Red | 기대 동작 명세화 | 두 개 이상 동시에 쓰지 않는다 |
| Green | 빠르게 통과 | 예쁜 코드보다 동작 먼저 |
| Refactor | 구조 개선 | 테스트를 먼저 돌려보고 고친다 |

## 상세 분해

**왜 테스트를 먼저 작성하는가**

구현 후 테스트를 작성하면 이미 구현에 맞춰 테스트를 쓰게 된다. 테스트 먼저 작성하면 사용자(호출자) 시점에서 인터페이스를 설계하게 되어 더 사용하기 편한 API가 나온다.

**Red 단계 요령**

테스트 하나에 검증 포인트를 하나만 담는다. `주문_금액이_0이면_예외를_던진다()`처럼 기대 동작을 이름으로 표현한다.

**Green 단계 요령**

가장 빨리 테스트를 통과하는 코드를 쓴다. 하드코딩해도 상관없다. 다음 테스트가 하드코딩을 깨뜨린다.

**Refactor 단계 요령**

테스트가 통과된 상태에서만 리팩토링한다. 중복 제거, 메서드 추출, 이름 개선 순으로 진행한다.

## 흔한 오해와 함정

- "TDD는 시간이 두 배 걸린다"는 오해가 있다. 초기에는 느리지만, 버그 디버깅 시간과 회귀 테스트 비용이 줄어 전체 개발 속도가 빨라지는 경우가 많다.
- "TDD를 하면 모든 버그가 사라진다"는 기대도 있다. TDD는 요구사항을 명세한 테스트를 통과하는 코드를 만드는 것이므로, 잘못된 명세는 잡지 못한다.
- "테스트가 통과하면 리팩토링은 나중에"라는 생각도 위험하다. Refactor 단계를 건너뛰면 기술 부채가 쌓인다. Red-Green까지만 반복하는 것은 TDD가 아니다.

## 실무에서 쓰는 모습

우아한형제들 미션에서 TDD를 연습할 때는 도메인 로직부터 시작한다. `OrderItem`의 금액 계산, 할인 적용 등 외부 의존이 없는 순수 도메인 로직을 TDD로 구현하면 사이클을 빠르게 돌릴 수 있다. DB나 HTTP가 연관된 부분은 통합 테스트로 별도 구성한다.

## 면접/시니어 질문 미리보기

- "TDD를 모든 코드에 적용해야 하나요?" — 도메인 핵심 로직에는 효과가 크지만, UI 렌더링이나 외부 API 연동처럼 명세가 자주 바뀌는 부분에는 테스트 유지 비용이 높아질 수 있다.
- "TDD와 BDD의 차이는 무엇인가요?" — TDD는 기술적 명세(단위 동작)에 초점을 두고, BDD는 비즈니스 시나리오(사용자 행동)를 중심으로 테스트를 작성한다. 대부분의 경우 TDD를 기반으로 BDD 스타일을 얹는다.
- "Green 단계에서 하드코딩이 정말 허용되나요?" — 허용된다. 다음 테스트 케이스(다른 입력값)가 하드코딩을 무효화하면서 자연스럽게 일반화가 진행된다. 이를 Triangulation 기법이라 한다.

## 한 줄 정리

TDD는 Red-Green-Refactor 사이클을 짧게 반복해 테스트가 보장하는 작은 증분으로 코드를 쌓아가는 개발 리듬이다.
