## 핵심 개념

처음 PR 리뷰를 받으면 문장이 추상적으로 들린다. 하지만 초심자에게 필요한 것은 이론 풀세트가 아니라 **문장을 `읽기 / 책임 / 테스트` 중 어디에 놓을지 먼저 고르는 것**이다.

짧게 외우면 이렇게 본다.

| 리뷰 문장 | 먼저 붙일 이름 | 바로 떠올릴 질문 |
|---|---|---|
| `추상화가 어색해요` | 표현과 경계 문제 | 이 이름이 역할을 설명하나, 아니면 구현 묶음을 감추기만 하나? |
| `책임이 섞였어요` | 레이어 문제 | 입력, 규칙, 저장 중 둘 이상이 한 메서드/클래스에 섞였나? |
| `테스트가 너무 무거워요` | 검증 범위 문제 | 지금 확인할 질문이 규칙 1개인가, 앱 전체 협력인가? |

- 멘탈 모델은 `리뷰 문장 -> 질문 1개 -> primer 1개 -> 첫 행동 1개`다.
- 한 번에 구조 전체를 고치지 말고, 이번 turn에 줄일 혼란 1개만 고른다.

## `추상화가 어색해요`를 들었을 때

보통 "역할보다 구현 디테일이 먼저 보인다"는 뜻이다. 초심자는 이를 `interface가 없어서 그런가?`로 오해하기 쉽다. 하지만 첫 질문은 더 작다.

| 먼저 확인할 것 | 예시 | 안전한 첫 행동 |
|---|---|---|
| 이름이 역할을 말하는가 | `OrderProcessor`가 사실 `saveAndPublishAndLog`를 다 한다 | 메서드/타입이 답하는 질문을 한 줄로 적는다 |
| 공통 규칙을 묶었는가, 잡동사니를 숨겼는가 | `DiscountPolicy`가 할인 규칙을 말하는지, 그냥 if 묶음인지 | 역할 이름과 실제 책임이 어긋나는 부분 1곳만 찾는다 |
| 추상화가 너무 이른가 | 구현이 하나뿐인데 인터페이스와 구현체가 이유 없이 갈라짐 | 불필요한 레이어 추가보다 이름과 책임부터 다듬는다 |

- safe move: `이 타입은 무엇을 약속하나?`를 한 문장으로 못 답하면 이름과 책임부터 다시 쓴다.
- follow-up: 역할과 구현의 차이가 더 헷갈리면 [객체지향 핵심 원리](../language/java/object-oriented-core-principles.md)의 추상화 부분으로 내려간다.

## `책임이 섞였어요`를 들었을 때

보통 "한 메서드나 클래스가 질문 여러 개에 답한다"는 뜻이다. 가장 흔한 장면은 Controller가 검증, 조회, 규칙, 저장, 응답 조립을 한 번에 다 하는 경우다.

| 섞인 장면 | 초심자 번역 | 안전한 첫 행동 |
|---|---|---|
| Controller가 재고 규칙까지 처리 | 입력과 업무 규칙이 섞임 | 입력 형식 검증 줄과 업무 규칙 줄을 다른 색으로 표시 |
| Service가 조회, 규칙, 응답 DTO 조립까지 다 처리 | 유스케이스 조립과 표현 책임이 섞임 | 이번 pass에서 옮길 책임 1개만 고른다 |
| Repository 이름인데 계산/정책 로직도 포함 | 저장 경계와 규칙이 섞임 | 저장 코드와 규칙 코드를 분리해서 읽는다 |

- safe move: 코드를 바로 옮기기 전에 `입력 / 규칙 / 저장` 세 라벨만 먼저 붙인다.
- stop rule: 라벨을 붙였는데도 한 줄이 어디 속하는지 모르겠다면 이동보다 [읽기 좋은 코드, 레이어 분리, 테스트 피드백 루프 입문](./readable-code-layering-test-feedback-loop-primer.md)으로 돌아간다.

## `테스트가 너무 무거워요`를 들었을 때

보통 "질문 1개를 확인하는데 환경을 너무 크게 띄우고 있다"는 뜻이다. 초심자는 이를 `테스트를 줄이라는 말인가?`로 오해할 수 있지만, 핵심은 **양이 아니라 범위**다.

| 지금 검증하려는 것 | 무거운 출발 | 더 안전한 첫 움직임 |
|---|---|---|
| 할인/재고/상태 전이 같은 규칙 1개 | `@SpringBootTest`로 주문 전체 플로우 실행 | 단위 테스트 1개부터 |
| 요청 JSON 바인딩, 검증, 응답 코드 | 서버 전체 실행 | `@WebMvcTest`부터 |
| 트랜잭션 경계, 여러 빈 협력 | 무조건 E2E부터 | 앱 통합 테스트 1개 또는 필요한 slice부터 |

- safe move: "이 테스트가 답해야 할 질문이 딱 1문장으로 말해지는가?"를 먼저 본다.
- follow-up: 어떤 테스트가 가장 싼지 고르기 어렵다면 [테스트 전략 기초](./test-strategy-basics.md)의 선택표로 내려간다.

## 흔한 오해와 함정

- `추상화가 어색해요`를 들으면 무조건 interface를 추가해야 한다고 생각하기 쉽다.
  - 더 안전한 해석은 "역할 이름과 실제 책임이 안 맞는다"부터 확인하는 것이다.
- `책임이 섞였어요`를 들으면 코드를 다른 계층으로 통째로 옮기기 쉽다.
  - 먼저 책임 라벨을 붙이지 않으면, 위치만 바뀌고 혼합은 그대로 남는다.
- `테스트가 너무 무거워요`를 들으면 테스트 수를 줄이거나 아예 안 쓰는 쪽으로 가기 쉽다.
  - 실제 뜻은 "지금 질문에 비해 검증 범위가 너무 넓다"에 가깝다.
- 세 문장을 한꺼번에 들었다고 한 turn에 다 해결하려 하면 구조 변경이 커진다.
  - beginner-safe 기준은 `책임 1개` 또는 `테스트 1개`만 줄이는 것이다.

## 실무에서 쓰는 모습

주문 생성 코드 리뷰에 아래 세 문장이 달렸다고 해 보자.

- `추상화가 어색해요`
- `Controller 책임이 너무 많아요`
- `테스트가 너무 무거워요`

이때 안전한 순서는 보통 아래처럼 작다.

| 순서 | 이번 turn에 하는 일 | 왜 이 순서가 안전한가 |
|---|---|---|
| 1 | `입력 / 규칙 / 저장` 표시부터 한다 | 책임 혼합을 말로만 보지 않고 코드 위치로 바꾼다 |
| 2 | 재고 부족 규칙 1개를 보호할 테스트를 고른다 | 리팩토링 전에 현재 동작 기준을 만든다 |
| 3 | `create()`에서 규칙 1개만 Service/Domain 쪽으로 옮긴다 | 구조 이동 범위를 한 pass로 제한한다 |
| 4 | 이름이 여전히 어색한 타입/메서드 1곳만 다듬는다 | 레이어 이동과 표현 수정을 한꺼번에 키우지 않는다 |

- 이 흐름은 "정답 구조 완성"이 아니라 "리뷰 문장을 안전한 첫 움직임으로 번역"하는 연습이다.
- 더 깊은 설계 논쟁은 다음 pass로 넘겨도 된다.

## 한 줄 정리

리뷰 문장이 추상적으로 느껴질수록 `의미 해석`보다 `primer 1개와 첫 행동 1개로 줄이는 라우팅`이 초심자에게 더 안전하다.
