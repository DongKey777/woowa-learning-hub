## 핵심 개념

코드 리뷰는 개발자가 자신의 변경(보통 Pull Request)을 팀원이 읽고 의견을 남기는 과정이다. 목적은 두 가지다.

1. **품질 보완** — 혼자서 보이지 않는 버그, 엣지 케이스, 설계 문제를 다른 시각으로 찾는다.
2. **지식 전파** — 리뷰를 통해 코드베이스 이해가 팀 전체로 퍼지고, 패턴과 관행이 공유된다.

입문자가 가장 헷갈리는 부분은 리뷰 댓글의 의도다. 리뷰어가 "이 부분은 어떻게 생각하세요?"라고 물으면 비판이 아니라 대화 시작이다.

## 한눈에 보기

| 리뷰 관점 | 확인하는 내용 | 예시 |
|---|---|---|
| 정확성 | 버그·엣지 케이스 | null 처리, 경계 조건 |
| 설계 | 책임 분리, 중복 | 서비스에 너무 많은 로직 |
| 가독성 | 변수명, 주석 | 의도 불명확한 이름 |
| 테스트 | 커버리지, 단언문 | 테스트 없는 비즈니스 로직 |

## 상세 분해

**PR 크기와 리뷰 품질**

PR이 크면 리뷰어가 맥락을 파악하기 어렵고 중요한 문제를 놓친다. 200줄 이하를 권장하는 팀이 많다. 기능 단위로 작게 나누면 리뷰 속도와 품질이 함께 올라간다.

**리뷰 댓글 유형**

댓글에는 강도가 있다. `nit:`(작은 취향 차이), 제안(Suggestion), 반드시 고쳐야 하는 지적(Blocker)을 구분해서 남기면 리뷰이가 우선순위를 알 수 있다. 강도 표시 없이 모든 댓글이 같은 무게처럼 보이면 불안감을 높인다.

**작성자 역할**

리뷰 요청 전에 자기 PR을 먼저 읽어보는 것이 중요하다. PR 설명에 변경 이유와 테스트 방법을 적으면 리뷰어가 맥락 파악에 드는 시간을 줄인다. 리뷰 댓글에는 방어적이지 않게, 질문으로 응답하면 토론이 생산적으로 이어진다.

**리뷰어 역할**

리뷰이의 코드가 아니라 코드 자체를 리뷰한다. "이 방식을 선택한 이유가 있나요?"처럼 의도를 묻는 질문이 "이렇게 하면 안 된다"는 단정보다 협업에 도움이 된다.

## 흔한 오해와 함정

- "리뷰는 무조건 빠를수록 좋다"는 오해가 있다. 빠르게 승인하는 LGTM(Looks Good To Me) 리뷰는 아무것도 보지 않는 것과 같다. 중요한 파일 몇 곳을 집중해서 보는 것이 낫다.
- "내 코드가 지적받으면 실력 문제다"라는 불안도 흔하다. 리뷰는 코드의 문제를 찾는 것이지 작성자를 평가하는 자리가 아니다.
- "모든 댓글에 동의해야 머지할 수 있다"는 생각도 있다. `nit` 수준의 의견은 선택 사항이다. 팀이 합의한 규칙에 따라 무엇이 머지 블로커인지 명확히 정해두면 좋다.

## 실무에서 쓰는 모습

우테코 미션 리뷰에서는 리뷰어가 "이 메서드의 책임이 두 개로 보이는데 의도적인 선택인가요?"처럼 질문한다. 이 댓글은 SRP 위반을 가리키는 것이지, 틀렸다고 단정하는 것이 아니다.

작성자는 설계 이유를 설명하거나, 납득되면 리팩토링해 다시 올린다. 이 왕복이 코드 리뷰의 핵심이다.

## 면접/시니어 질문 미리보기

- "좋은 코드 리뷰의 기준은?" — 작성자가 다음 PR에서 동일한 실수를 반복하지 않도록 학습이 일어났는지가 기준이다. 승인/거부 결정보다 이유를 남기는 것이 더 가치 있다.
- "리뷰 속도와 품질 사이의 균형은?" — 소규모 PR을 자주 올리면 속도와 품질을 동시에 높일 수 있다. 큰 PR은 리뷰 비용이 비선형으로 증가한다.
- "자동화와 코드 리뷰는 어떻게 나뉘나요?" — 포맷, 린트, 테스트 실패는 자동화에 맡기고, 설계 의도·맥락·트레이드오프 판단은 사람이 리뷰한다.

## 한 줄 정리

코드 리뷰는 버그를 잡는 도구이기도 하지만, 팀이 설계 기준을 맞추고 서로 배우는 가장 저렴한 협업 방식이다.
