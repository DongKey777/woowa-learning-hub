## 핵심 개념

처음에는 "작업 하나 했으니 커밋도 하나"라고 생각하기 쉽다. 하지만 `테스트 추가`, `구조 정리`, `동작 변경`은 **리뷰 질문이 서로 다르다.**

- 테스트 추가 커밋: "현재 동작을 무엇으로 고정했나?"
- 리팩토링 커밋: "동작은 그대로고 읽는 경로만 바뀌었나?"
- 동작 변경 커밋: "규칙이나 결과가 실제로 무엇이 달라졌나?"

이 셋을 한 커밋에 섞으면 리뷰어는 "테스트가 바뀌어서 통과한 건지, 구조가 바뀌어서 그런 건지, 규칙이 바뀐 건지"를 한 번에 추적해야 한다.

## 먼저 잡는 20초 멘탈 모델

커밋 분리는 일을 잘게 쪼개는 기술이 아니라, **질문을 한 번에 하나씩만 받게 만드는 기술**이다.

| 커밋 순서 | 커밋의 질문 | 기대하는 diff 모습 |
|---|---|---|
| `test` | 현재 동작을 무엇으로 잠갔나 | 테스트 파일 위주 변경 |
| `refactor` | 결과는 그대로인데 구조만 읽기 쉬워졌나 | 이름 변경, 메서드 추출, 중복 제거 |
| `feat` 또는 `fix` | 이제 어떤 규칙을 바꿨나 | 기대값 변경 + 제품 코드 변경 |

짧게 외우면 `잠그고 -> 정리하고 -> 바꾼다`다.

## 한눈에 보기

| 상황 | 한 커밋에 섞으면 생기는 일 | 분리하면 좋아지는 점 |
|---|---|---|
| 테스트 추가와 리팩토링을 같이 함 | 테스트가 안전망인지 결과 맞춤인지 헷갈린다 | 테스트가 먼저 기준선 역할을 한다 |
| 리팩토링과 동작 변경을 같이 함 | 구조 변경이 버그인지 의도인지 구분이 어렵다 | 리팩토링 커밋만 따로 되돌리기 쉽다 |
| 테스트, 구조, 기능을 전부 같이 넣음 | 리뷰 범위가 커지고 롤백 단위가 거칠어진다 | 커밋별로 리뷰와 revert가 가능하다 |

```text
1. test: 현재 동작을 설명하는 테스트 추가
2. refactor: 테스트는 그대로 두고 구조만 정리
3. feat/fix: 테스트 기대값과 제품 코드를 함께 바꿔 새 동작 반영
```

## 같은 예시로 보는 안전한 3커밋 흐름

주문 생성 코드에서 `quantity <= 0` 검증은 이미 있고, 메서드가 길어서 정리하고 싶다고 해 보자. 그리고 그 다음 요구사항으로 "재고가 0이면 다른 예외를 던진다"는 새 동작도 넣어야 한다.

| 커밋 | 들어갈 것 | 아직 넣지 않을 것 |
|---|---|---|
| `test(order): 주문 생성 현재 동작 테스트 추가` | 기존 수량 검증, 저장 성공 흐름을 고정하는 테스트 | 새 재고 정책 기대값 |
| `refactor(order): 주문 생성 단계 메서드로 분리` | `validate`, `save`, `buildResponse` 추출, 이름 개선 | 예외 규칙 변경 |
| `feat(order): 재고 부족 예외 규칙 추가` | 재고 0일 때 실패 테스트, 제품 코드 변경 | 순수 구조 정리 |

이 순서를 따르면 리뷰어는 첫 커밋에서 기준선을 보고, 둘째 커밋에서 "동작 유지"만 검사하고, 셋째 커밋에서 실제 요구사항 변경만 읽으면 된다.

## 상세 분해

**1. 테스트 추가 커밋**

먼저 현재 동작을 잠그는 최소 테스트를 넣는다. 이 커밋의 목적은 기능을 늘리는 것이 아니라, "지금 이 코드는 이렇게 동작한다"를 문서로 남기는 것이다.

**2. 리팩토링 커밋**

테스트가 초록인 상태에서 메서드 추출, 이름 개선, 중복 제거를 한다. 이 커밋에서는 테스트 기대값을 바꾸지 않는다. 기대값이 바뀌면 이미 구조 정리만의 커밋이 아니게 된다.

**3. 동작 변경 커밋**

새 요구사항이나 버그 수정은 마지막 커밋에서 한다. 여기서는 테스트 기대값이 바뀌는 것이 자연스럽다. 왜냐하면 이제 정말 바깥에서 보이는 결과가 달라지기 때문이다.

## 언제 한 커밋으로 묶어도 되나요?

초심자가 자주 묻는 질문은 "테스트와 구현은 원래 같이 가는 것 아닌가요?"다. 맞다. 다만 **무엇을 같이 묶는가**를 구분해야 한다.

| 묶어도 되는 것 | 이유 |
|---|---|
| `feat` 커밋 안의 새 테스트 + 새 동작 구현 | 둘 다 같은 새 요구사항을 설명한다 |
| `fix` 커밋 안의 회귀 테스트 + 버그 수정 | 실패 재현과 수정이 같은 문제를 다룬다 |

| 분리하는 편이 안전한 것 | 이유 |
|---|---|
| 현재 동작 고정 테스트 + 리팩토링 | 테스트가 기준선인지 결과 맞춤인지 분리해야 한다 |
| 리팩토링 + 새 기능 추가 | 구조 변경과 규칙 변경의 책임이 다르다 |

기준은 하나다. **리뷰 질문이 달라지면 커밋도 나눈다.**

## 흔한 오해와 함정

- "작은 커밋이 많으면 오히려 복잡하다"는 말이 있다. 의미 없는 잘게 쪼개기는 복잡하지만, 질문이 다른 변경을 나누는 것은 오히려 읽기 부담을 줄인다.
- "어차피 최종 코드는 같으니 한 커밋이면 된다"는 생각도 위험하다. 리뷰와 롤백은 최종 상태만 보지 않고, 거기까지 가는 경로를 본다.
- "리팩토링 커밋에서도 테스트를 조금 고쳐도 된다"고 느끼기 쉽다. 이름만 바꾸는 수준의 기계적 수정은 가능하지만, 기대값 변경이 들어가면 동작 변경으로 다시 분리하는 편이 안전하다.
- "테스트가 없으면 그냥 리팩토링부터"라는 습관도 위험하다. 이 경우는 커밋 분리 이전에 최소 테스트 확보가 먼저다.

## 실무에서 쓰는 모습

PR 리뷰에서 가장 자주 나오는 좋은 반응은 "커밋 따라 읽기 쉬워요"다. 리뷰어는 `test` 커밋에서 기준선을 확인하고, `refactor` 커밋은 빠르게 스캔하고, 마지막 `feat`나 `fix` 커밋에 시간을 쓴다.

롤백도 비슷하다. 새 동작 커밋만 되돌릴지, 구조 정리 커밋까지만 되돌릴지 선택지가 생긴다. 반대로 세 종류가 한 커밋에 섞이면 되돌릴 때 테스트 안전망까지 함께 잃을 수 있다.

## 면접/시니어 질문 미리보기

- "왜 리팩토링과 기능 변경을 분리해야 하나요?" — 리뷰 질문과 rollback 단위가 달라지기 때문이다. 동작 보존 검증과 규칙 변경 검증을 한 덩어리로 만들면 추적 비용이 커진다.
- "테스트를 먼저 따로 커밋하지 못하는 경우도 있나요?" — 있다. 하지만 최소한 `현재 동작 고정`과 `새 요구사항 반영`은 머릿속에서라도 분리해야 하며, 가능하면 커밋에도 드러내는 편이 안전하다.
- "커밋 메시지는 꼭 `test -> refactor -> feat` 순서여야 하나요?" — 순서 자체보다 의미 분리가 중요하다. 다만 beginner 기준으로는 이 순서가 가장 읽기 쉽고 실수도 적다.

## 한 줄 정리

`테스트 추가 -> 리팩토링 -> 동작 변경`으로 커밋을 나누면 리뷰어는 질문을 하나씩만 읽을 수 있고, 작성자는 더 안전하게 되돌릴 수 있다.
