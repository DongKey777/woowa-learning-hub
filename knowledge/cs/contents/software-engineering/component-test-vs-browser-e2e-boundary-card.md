# 컴포넌트 테스트 vs 브라우저 E2E 경계 카드

## 핵심 개념

이 문서의 질문은 "`이 UI 동작을 컴포넌트 테스트에서 끝내도 되나, 진짜 브라우저 E2E까지 올려야 하나?`"다. 초심자는 화면 이야기가 나오면 전부 Playwright나 Cypress로 가야 한다고 느끼기 쉽지만, 실제로는 **질문이 어디서 끝나는가**를 먼저 보면 된다.

짧게 외우면 이렇다. `한 화면/한 컴포넌트 안에서 상태 변화와 렌더링만 보면 된다 -> 컴포넌트 테스트`, `라우팅, 로그인 리다이렉트, 쿠키/세션, 여러 페이지 연결이 핵심이다 -> 브라우저 E2E`다.

## 한눈에 보기

| 지금 확인하려는 질문 | 먼저 둘 곳 | 이유 |
|---|---|---|
| 버튼을 누르면 loading spinner가 뜨고 버튼이 비활성화되나 | 컴포넌트 테스트 | 한 컴포넌트 안의 상태 변화다 |
| 잘못된 입력이면 에러 문구가 바로 보이나 | 컴포넌트 테스트 | 렌더링과 입력 검증 피드백이 핵심이다 |
| 로그인 후 대시보드로 실제 이동하고 세션이 유지되나 | 브라우저 E2E | 라우팅과 브라우저 상태가 핵심이다 |
| 결제창, 외부 인증, 파일 업로드가 실제 브라우저에서 끝까지 이어지나 | 브라우저 E2E | 팝업, 리다이렉트, 브라우저 API가 핵심이다 |

기억할 기준은 `로컬 UI 규칙이면 component`, `브라우저 경계 약속이면 e2e`다.

## 컴포넌트 테스트에 남겨도 되는 질문

아래처럼 **한 화면 안에서 답이 끝나는 질문**은 보통 컴포넌트 테스트가 더 싸고 읽기 쉽다.

| 질문 | 왜 component test가 맞나 |
|---|---|
| 폼 입력이 비었을 때 submit 버튼이 비활성화되나 | DOM 상태와 props/state 계산이 핵심이다 |
| API 실패 시 error banner가 보이고 재시도 버튼이 나타나나 | 네트워크 결과에 대한 화면 반응이 핵심이다 |
| 장바구니 수량을 바꾸면 합계 금액이 바로 갱신되나 | 계산 결과와 렌더링 연결을 빠르게 확인할 수 있다 |
| 저장 중 spinner가 뜨고 중복 클릭이 막히나 | 타이밍과 상태 전이가 한 컴포넌트 안에서 끝난다 |

이때는 실제 서버를 모두 붙일 필요가 없다. MSW 같은 가짜 네트워크나 간단한 stub 응답으로도 "`이 화면이 어떤 상태를 보여 줘야 하나`"를 충분히 검증할 수 있다.

즉 컴포넌트 테스트는 "`API가 진짜인가`"보다 "`사용자가 이 화면에서 어떤 피드백을 보나`"를 싸게 고정하는 자리다.

## 브라우저 E2E로 올려야 하는 신호

아래처럼 **브라우저 자체가 답의 일부**가 되는 순간에는 컴포넌트 테스트만으로는 부족할 수 있다.

| 신호 | 왜 browser E2E가 필요한가 |
|---|---|
| 로그인 후 리다이렉트와 보호 페이지 진입이 핵심이다 | 쿠키, 세션, 라우터 연결이 실제로 이어져야 한다 |
| 결제창, OAuth, 외부 인증 페이지를 거친다 | 여러 도메인과 팝업, redirect chain이 개입한다 |
| 파일 업로드/다운로드, drag-and-drop 같은 브라우저 API가 핵심이다 | test double만으로는 실제 사용자 흐름과 차이가 나기 쉽다 |
| 여러 페이지를 오가며 장바구니, 주문 완료, 새로고침 복구를 본다 | 한 컴포넌트 바깥의 사용자 여정 전체가 질문이다 |

한 줄 기준은 `브라우저가 없으면 질문 자체가 성립하지 않는가`다. Yes면 E2E 후보가 된다.

## 주문 화면 예시로 바로 나누기

주문 화면을 예로 들면 아래처럼 자르면 초심자도 덜 흔들린다.

| 장면 | 먼저 둘 테스트 | 이유 |
|---|---|---|
| 수량이 0이면 `주문하기` 버튼이 비활성화된다 | 컴포넌트 테스트 | 화면 내부 규칙이다 |
| 주문 요청 중 spinner가 뜨고 버튼 연타가 막힌다 | 컴포넌트 테스트 | 비동기 상태 전이를 한 화면에서 본다 |
| 로그인하지 않은 사용자가 주문 버튼을 누르면 로그인 페이지로 이동한다 | 브라우저 E2E | 라우팅과 인증 흐름이 핵심이다 |
| 로그인 후 장바구니에서 주문 완료 페이지까지 실제로 이어진다 | 브라우저 E2E | 멀티페이지 사용자 여정이다 |

그래서 "`버튼 연타 방지`"라는 말만 보고 무조건 E2E로 가기보다, **그 질문이 한 화면에서 닫히는지**를 먼저 확인해야 한다. 한 화면에서 닫히면 컴포넌트 테스트가 더 빠르고 실패 원인도 더 좁다.

## 흔한 오해와 함정

- "`API를 호출하니까 무조건 E2E 아닌가요?`"는 흔한 오해다. API 호출 결과에 대한 화면 반응만 보면 컴포넌트 테스트로도 충분한 경우가 많다.
- "`컴포넌트 테스트는 너무 가짜라 믿기 어렵다`"도 과한 생각이다. 질문이 로컬 UI 상태라면 오히려 더 직접적이다.
- "`브라우저 E2E 하나면 다 커버되겠지`"도 위험하다. 너무 긴 E2E는 실패 원인이 흐려지고 flaky cost가 커진다.
- "`라우터를 쓰면 전부 E2E다`"도 아니다. 메모리 라우터 수준에서 닫히는 화면 분기라면 component/integration 경계에서 충분할 수 있다.

## 실무에서 쓰는 모습

안전한 기본 포트폴리오는 보통 아래처럼 시작한다.

1. 컴포넌트 테스트로 버튼 상태, 에러 문구, loading UI, 낙관적 업데이트 같은 로컬 상호작용을 많이 잡는다.
2. 브라우저 E2E는 로그인, 결제, 외부 인증, 멀티페이지 완료 흐름 몇 개만 남긴다.
3. 같은 질문을 이미 컴포넌트 테스트가 잡고 있다면 E2E에는 다시 싣지 않는다.

초심자가 바로 적용할 수 있는 한 줄 선택법은 `한 화면 안에서 닫히면 component`, `브라우저를 실제로 지나가야만 답이 나오면 e2e`다.
