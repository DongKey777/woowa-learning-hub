## 핵심 개념

이 문서의 질문은 "`컴포넌트 테스트인데 API를 진짜로 붙여야 하나요, MSW로 막아도 되나요?`"다. 초심자는 `가짜면 못 믿는 것 아닌가`와 `진짜면 항상 더 좋은 것 아닌가` 사이에서 흔들리기 쉽다.

먼저 이렇게 자르면 된다. **화면 반응을 검증하는 질문이면 MSW stub**, **서버 계약이 실제로 맞물리는지까지 검증하는 질문이면 실제 백엔드/테스트 서버 후보**다. 즉 기준은 "진짜냐 가짜냐"보다 **어디서 답이 끝나는가**다.

## 한눈에 보기

| 지금 확인하려는 질문 | 먼저 둘 곳 | 이유 |
|---|---|---|
| loading spinner, error banner, 빈 상태 문구가 맞게 보이나 | MSW stub 컴포넌트 테스트 | 화면 상태 전이가 핵심이다 |
| 500, 404, timeout일 때 재시도 버튼이 보이나 | MSW stub 컴포넌트 테스트 | 실패 응답을 싸고 안정적으로 재현하기 쉽다 |
| 서버가 보내는 날짜/enum/nullable 필드 형태가 프론트 파서와 정말 맞나 | 실제 백엔드 또는 테스트 서버 | 실제 응답 shape와 직렬화 계약이 핵심이다 |
| 쿠키, 인증 헤더, 프록시, CORS 설정까지 포함해 붙는지 보나 | 실제 백엔드 또는 더 넓은 통합/E2E | 브라우저-서버 wiring이 질문의 일부다 |

- 초심자 기본값은 `component test = MSW stub 우선`이 안전하다.
- 다만 이 기본값은 "프레임워크 안에서 네트워크를 잘 가로챌 수 있고, 화면 상태 검증이 목적일 때" 특히 잘 맞는다.
- 반대로 실제 서버를 붙였더라도 질문이 UI 상태뿐이면 테스트가 불필요하게 느려지고 flaky해질 수 있다.

## MSW stub이 먼저 맞는 경우

MSW는 테스트에서 `fetch`/`XHR` 같은 네트워크 호출을 가로채서, 화면이 기대한 응답을 받은 것처럼 재현하게 도와주는 도구다. 입문자에게 중요한 포인트는 구현 디테일보다 **"응답 장면을 싸게 만들어 준다"**는 점이다.

아래 질문은 보통 MSW stub으로 먼저 잠그는 편이 좋다.

| 장면 | 왜 MSW stub이 잘 맞나 |
|---|---|
| 저장 버튼 클릭 후 spinner가 뜨고 중복 클릭이 막히나 | 성공/지연 응답을 쉽게 재현할 수 있다 |
| 서버 에러가 오면 에러 배너와 재시도 버튼이 보이나 | 500, 404 같은 상태를 안정적으로 만들 수 있다 |
| 빈 목록 응답이면 empty state가 보이나 | 데이터 shape를 작게 통제하며 UI 분기를 읽기 쉽다 |
| 검색 결과가 늦게 오면 skeleton이 유지되나 | 타이밍 장면을 실제 서버보다 예측 가능하게 만든다 |

여기서 핵심은 "`서버가 진짜로 돌아갔나`"가 아니라 "`이 응답을 받았을 때 화면이 어떻게 바뀌나`"다. 이 질문은 실제 서버 없이도 충분히 강하게 검증할 수 있다.

## 실제 백엔드나 테스트 서버를 붙일 신호

실제 서버를 붙여야 할 때도 있다. 다만 이때는 "진짜라서 더 멋져 보여서"가 아니라, **stub이 놓칠 수 있는 계약 위험이 실제로 보일 때**여야 한다.

| 신호 | 왜 실제 서버 후보가 되나 |
|---|---|
| 프론트가 기대하는 필드 이름/nullable/date format이 자주 어긋난다 | 응답 직렬화 계약을 진짜 payload로 확인해야 한다 |
| 인증 쿠키, CSRF, 프록시, base URL 조합에서 자주 깨진다 | 네트워크 wiring 자체가 질문이다 |
| BFF나 테스트 서버가 프론트와 함께 관리되어 API drift를 자주 잡아야 한다 | 컴포넌트 바깥 계약 검증 가치가 크다 |
| 팀 규칙상 컴포넌트 테스트보다 넓은 app integration 계층에서 프론트-백엔드 연결을 확인한다 | 계층 책임을 팀 컨벤션에 맞춰야 한다 |

주의할 점도 있다. 많은 팀에서는 이런 질문을 **컴포넌트 테스트 자체**보다 `app integration`, `contract`, `browser E2E`에 두기도 한다. 즉 "실제 백엔드를 붙인다"와 "반드시 컴포넌트 테스트에서 한다"는 같은 말이 아니다.

## 주문 목록 화면 예시로 바로 나누기

주문 목록 컴포넌트를 테스트한다고 해 보자.

| 장면 | 추천 시작점 | 이유 |
|---|---|---|
| 로딩 중 skeleton이 보인다 | MSW stub 컴포넌트 테스트 | 지연 응답만 만들면 된다 |
| 빈 배열이면 `주문이 없습니다` 문구가 보인다 | MSW stub 컴포넌트 테스트 | UI 분기 검증이다 |
| `createdAt` 날짜 포맷이 서버 실응답과 자주 어긋난다 | 실제 백엔드/테스트 서버 확인 추가 | 직렬화 계약이 의심된다 |
| 운영에서는 쿠키 인증 때문에 목록 호출이 자주 401 난다 | 실제 백엔드 또는 더 넓은 통합/E2E 확인 | 인증 wiring이 핵심이다 |

그래서 시작 순서는 보통 이렇다. `UI 상태 질문 -> MSW stub 컴포넌트 테스트`, `계약/wiring 불안 -> 실제 서버 검증을 별도 계층에 추가`. 둘 중 하나만 절대 정답으로 고정하는 게 아니라, **질문에 따라 층을 나눠 갖는 것**이 더 안전하다.

## 흔한 오해와 함정

- "`컴포넌트 테스트인데 네트워크를 가짜로 만들면 의미 없지 않나요?`"
  아니다. 질문이 화면 반응이라면 오히려 더 직접적이다. 진짜 서버는 그 질문에 비해 비용이 클 수 있다.
- "`실제 백엔드를 붙이면 무조건 더 신뢰도가 높죠?`"
  보통 더 넓은 위험을 잡을 수는 있지만, 느리고 flaky해질 수 있다. 넓은 신뢰와 빠른 피드백은 같은 것이 아니다.
- "`MSW가 있으면 API 계약 문제도 다 잡히나요?`"
  아니다. MSW handler를 실제 응답과 다르게 써 버리면 drift를 놓칠 수 있다. 계약 불안이 크면 contract test나 실제 서버 검증을 별도로 둬야 한다.
- "`실제 서버 검증은 컴포넌트 테스트에서만 해야 하나요?`"
  아니다. 팀에 따라 더 넓은 integration/E2E 계층이 더 적절할 수 있다.

## 실무에서 쓰는 모습

많은 프론트엔드 팀의 안전한 기본 포트폴리오는 아래에 가깝다.

1. 대부분의 화면 상태 테스트는 MSW stub 컴포넌트 테스트로 빠르게 잡는다.
2. API 계약이 자주 깨지는 경계는 별도의 contract/app integration 테스트나 테스트 서버 검증으로 보강한다.
3. 로그인, 리다이렉트, 쿠키, 외부 인증처럼 브라우저와 서버 wiring이 핵심인 흐름만 소수의 E2E에 남긴다.

MSW를 쓰더라도 handler 데이터는 실제 API 예시에 가깝게 유지하는 편이 좋다. 그래야 "화면 상태 검증"과 "응답 shape 감각"이 너무 멀어지지 않는다.

## 한 줄 정리

프론트엔드 컴포넌트 테스트에서는 **화면 반응을 묻는 질문은 MSW stub으로 먼저 잡고**, **실제 응답 계약과 인증 wiring이 질문의 핵심일 때만 실제 백엔드나 테스트 서버 검증을 추가**하면 된다.
