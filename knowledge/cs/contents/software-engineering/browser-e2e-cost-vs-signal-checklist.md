## 핵심 개념

브라우저 E2E는 브라우저 클릭, 화면 전환, 쿠키/스토리지, 네트워크, 백엔드까지 이어지는 **실제 사용자 흐름**을 검증한다. 그래서 신호는 크지만, 실행 시간과 유지 비용도 가장 크다.

초심자가 자주 하는 실수는 "중요하니까 많이 두자"로 가는 것이다. 하지만 브라우저 E2E는 개수가 늘수록 아래 비용도 같이 커진다.

- 느려진 CI 때문에 피드백이 늦어진다
- DOM/CSS/타이밍 변화로 flaky test가 늘어난다
- 실패 원인이 UI인지 API인지 데이터 준비인지 바로 안 보인다

짧게 외우면 `브라우저 E2E = 마지막 확인`, `대부분의 규칙 검증 = unit/slice/app integration`이다.

## 한눈에 보기

| 질문 | Yes면 브라우저 E2E 후보 | No면 먼저 둘 곳 |
|---|---|---|
| 사용자가 브라우저에서 실제로 겪는 전환/리다이렉트가 핵심인가 | 남길 가치가 크다 | `MockMvc`, `TestRestTemplate` 같은 슬라이스/앱 통합 테스트 |
| JS, 버튼 비활성화, 중복 클릭 방지, 파일 업로드가 핵심인가 | 남길 가치가 크다 | 서버 테스트로 먼저 내린다 |
| 실패가 나면 매출/주문/로그인처럼 핵심 흐름이 직접 끊기나 | 남길 가치가 크다 | slice나 contract로 충분한지 다시 본다 |
| 같은 위험을 더 싸고 안정적인 테스트가 이미 잡고 있나 | 브라우저 E2E를 줄인다 | 더 싼 테스트를 유지한다 |

`signal`은 "이 테스트가 아니면 놓치기 쉬운 사용자 위험"이고, `cost`는 "느림 + flakiness + 디버깅 시간"이다. 브라우저 E2E는 `signal > cost`인 흐름만 남긴다.

## 몇 개만 남길지 고르는 체크리스트

아래 5문항 중 `Yes`가 4~5개면 브라우저 E2E로 남길 후보, `2~3개`면 app integration이나 contract로 내릴지 재검토, `0~1개`면 보통 브라우저 E2E까지는 과하다.

| 체크 질문 | 왜 중요한가 |
|---|---|
| 1. 이 흐름은 로그인, 결제, 주문 완료처럼 핵심 사용자 여정인가 | 실패 비용이 큰 흐름만 마지막 안전망 가치가 크다 |
| 2. 브라우저 상태 변화가 핵심인가 | 쿠키, 리다이렉트, 화면 전환, 버튼 상태는 서버 테스트만으로 다 못 잡는다 |
| 3. 최근 버그가 실제 화면 연결에서 났는가 | 과거 장애가 같은 경계에 몰렸다면 신호가 커진다 |
| 4. 이 흐름을 팀이 데모나 배포 체크에서 자주 확인하는가 | 사람이 반복 수동 확인하는 흐름은 자동 E2E 후보가 된다 |
| 5. 실패했을 때 "어디가 깨졌는지"를 여전히 좁힐 수 있게 시나리오를 짧게 유지했는가 | 너무 긴 E2E는 신호보다 디버깅 비용이 커진다 |

초심자용 안전한 기본값은 이렇다.

| 제품 성격 | 처음 남길 브라우저 E2E 수 |
|---|---|
| 단순 CRUD 백엔드 + 얇은 화면 | `0~2`개 |
| 로그인, 장바구니, 주문 같은 핵심 흐름이 있는 서비스 | `2~5`개 |
| 결제, 외부 인증, 업로드처럼 브라우저 상호작용이 매우 중요한 서비스 | `3~7`개부터 시작하고 계속 줄일지 본다 |

핵심은 "정답 개수"가 아니라 `배포 때 꼭 확인할 흐름 몇 개만 자동화했는가`다.

## flaky cost가 커지는 신호

아래 항목이 많아질수록 브라우저 E2E는 유지보다 손해가 커질 수 있다.

| 보이는 증상 | 보통 더 나은 첫 대응 |
|---|---|
| 같은 테스트가 가끔만 실패한다 | sleep을 늘리기보다 wait 조건과 데이터 격리를 먼저 본다 |
| 하나의 시나리오가 로그인부터 관리자 승인까지 너무 길다 | 핵심 경계별로 잘라 app integration 또는 contract로 내린다 |
| CSS 선택자나 DOM 구조 변경 때마다 많이 깨진다 | 테스트용 안정 selector를 두고, 화면 세부 검증은 줄인다 |
| 같은 실패를 unit/slice/app integration이 이미 잡고 있다 | 브라우저 E2E를 삭제하거나 smoke만 남긴다 |
| CI가 느려서 PR 피드백이 밀린다 | 핵심 1~2개 smoke만 PR gate에 두고 나머지는 nightly로 내린다 |

`flaky라서 E2E가 나쁘다`가 아니라, `flaky cost를 감수할 만큼 사용자 신호가 큰가`를 계속 다시 묻는 것이 중요하다.

## 핵심 사용자 흐름을 고르는 기준

브라우저 E2E로 남길 흐름은 "기능 목록"이 아니라 **사용자 약속**으로 고르면 흔들림이 적다.

| 나쁜 고르기 | 더 나은 고르기 |
|---|---|
| 회원 API가 12개니까 화면 테스트도 12개 | "신규 사용자가 로그인해서 첫 주문 완료까지" 1개 |
| 버튼마다 E2E 1개씩 | "결제 실패 후 다시 시도해도 중복 결제되지 않는다" 1개 |
| 관리자 화면의 모든 필터 조합 | "관리자가 주문 상태를 바꾸면 고객 화면에 반영된다" 1개 |

beginner 기준으로는 아래 세 묶음부터 보면 충분하다.

- `로그인/세션 복구`: 로그인 후 진입, 로그아웃, 재접속 같은 흐름
- `돈/주문/신청 완료`: 실패 비용이 큰 완료 흐름
- `외부 연동 화면`: 결제창, 파일 업로드, 외부 인증처럼 브라우저 상호작용이 핵심인 흐름

세 묶음 밖의 시나리오는 먼저 `정말 브라우저에서만 잡히는가`를 의심해 보는 편이 안전하다.

## 흔한 오해와 함정

- "`@SpringBootTest`가 넓으니 그냥 E2E와 비슷하다"는 오해가 있다. 앱 통합 테스트와 브라우저 E2E는 비용 구조와 잡는 위험이 다르다.
- "브라우저 E2E가 많을수록 품질이 높다"는 생각도 위험하다. 너무 많으면 실패를 믿지 못하게 되어 오히려 신호가 약해진다.
- "flaky하면 재시도 몇 번 넣으면 된다"는 접근도 흔하다. 재시도는 증상을 가릴 수 있고, selector/동기화/데이터 격리 문제를 먼저 봐야 한다.
- "한 시나리오에 여러 화면을 다 태우면 든든하다"는 착각도 있다. 길수록 실패 원인 분리가 어려워진다.
- "프런트 버그를 잡으려면 전부 브라우저로 봐야 한다"도 과하다. 컴포넌트 테스트, app integration, contract test가 더 싼 경우가 많다.
- "`버튼 비활성화`나 `에러 문구`도 다 E2E로 가야 하나요?"가 먼저 막히면 [컴포넌트 테스트 vs 브라우저 E2E 경계 카드](./component-test-vs-browser-e2e-boundary-card.md)에서 한 화면 안에서 닫히는 질문을 먼저 내려 보면 된다.

## 실무에서 쓰는 모습

예를 들어 주문 서비스가 있다고 하자. 초심자에게 안전한 분리는 보통 아래와 같다.

| 검증 질문 | 더 싼 테스트 | 브라우저 E2E로 남길지 |
|---|---|---|
| 재고가 없으면 주문이 실패하는가 | unit/app integration | 보통 내린다 |
| `POST /orders`가 `400`/`201`으로 끝나는가 | `@WebMvcTest` | 보통 내린다 |
| 로그인 후 장바구니에서 결제 완료 화면까지 실제로 이어지는가 | 브라우저 E2E | 남긴다 |
| 결제 실패 후 다시 시도해도 버튼 중복 클릭이 막히는가 | 브라우저 E2E 또는 프런트 컴포넌트 테스트 | 화면 상호작용이 핵심이면 남긴다 |

즉 브라우저 E2E는 "서버가 맞는가"보다 "`사용자가 끝까지 성공했는가`"를 최종 확인하는 자리로 남겨 두면 된다.

## 면접/시니어 질문 미리보기

- "왜 브라우저 E2E를 줄이려 하나요?" — 느림과 flakiness 비용이 커서, 더 싼 테스트가 잡을 수 있는 위험은 아래 계층으로 내리는 편이 전체 피드백 품질이 좋아진다.
- "그럼 브라우저 E2E는 무엇만 남기나요?" — 로그인, 결제, 주문 완료, 외부 인증처럼 사용자 여정과 브라우저 상호작용이 동시에 핵심인 흐름을 남긴다.
- "flaky test를 줄이는 첫 기준은?" — retry를 먼저 넣기보다 selector 안정성, wait 조건, 데이터 격리, 시나리오 길이를 먼저 줄인다.

## 한 줄 정리

브라우저 E2E는 핵심 사용자 여정 몇 개만 남기고, 나머지 위험은 unit/slice/app integration으로 내려서 `신호는 크게, flaky cost는 작게` 유지하는 것이 기본 전략이다.
