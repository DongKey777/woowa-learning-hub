# CDN Error HTML vs App Error JSON Decision Card

> 한 줄 요약: DevTools에서 에러 body가 보이면 `status`, `content-type`, body 첫 줄만 먼저 묶어 branded CDN 에러 HTML과 app이 직접 만든 JSON 에러를 초급자 기준으로 빠르게 분리할 수 있고, CloudFront/Cloudflare의 자주 보이는 첫 문장도 같이 외울 수 있다.

## 핵심 개념

처음에는 body 전체를 읽지 말고 "이 에러를 누가 말하고 있나"만 가르면 된다.

- branded CDN 에러 HTML: edge/CDN이 자기 브랜드 문구나 기본 차단 페이지를 보여 주는 장면 후보
- app-owned JSON error: 애플리케이션이 자기 API 계약 형식으로 에러를 내려준 장면 후보

초급자가 자주 헷갈리는 이유는 둘 다 DevTools `Response` 탭에서는 그냥 "에러 본문"처럼 보이기 때문이다. 그래서 첫 pass에서는 아래 3칸만 묶는다.

1. `Status`
2. `Content-Type`
3. body 첫 줄이나 첫 문장

## 한눈에 보기

| 보인 조합 | 초급자 첫 해석 | owner 후보 | 안전한 다음 질문 |
|---|---|---|---|
| `403`/`429`/`503` + `text/html` + branded 문구 | app JSON보다 CDN/edge 에러 페이지 후보다 | CDN/edge HTML | `Server`, `CF-Cache-Status`, `X-Cache`가 보이는가 |
| `400`/`401`/`404`/`500` + `application/json` + 서비스 필드 | app이 자기 에러 계약으로 응답했을 가능성이 크다 | app-owned JSON | `errorCode`, `message`, `traceId`가 서비스 규칙과 맞는가 |
| `502`/`504` + `application/json` + generic timeout/title | app JSON보다 gateway/proxy JSON일 수 있다 | gateway/proxy JSON | `Server`/`Via`와 generic `title`/`detail`이 보이는가 |
| `200` + `text/html` + `login`, `<form` | CDN 에러보다 login/page fallback 후보가 더 크다 | login HTML | final URL이 `/login`인가 |

짧게 외우면 이렇다.

```text
403/429/503 + branded html -> CDN/edge 후보
400/401/404/500 + service json -> app JSON 후보
502/504 + generic json -> app 확정 보류
```

## status + content-type + 첫 줄을 이렇게 읽는다

### 1. status

`403`, `429`, `503`은 edge/CDN 차단이나 보호 페이지와 자주 붙는다.
`400`, `401`, `404`, `500`은 app이 직접 JSON 에러를 내려줄 때 더 자주 보인다.

물론 예외는 있지만, beginner 첫 분기에서는 이 정도만 기억해도 충분하다.

### 2. `Content-Type`

- `text/html`: page 계열 후보를 연다. 이때 branded CDN 에러 페이지인지, login HTML인지 더 본다.
- `application/json`: API payload 후보를 연다. 이때 app-owned JSON인지 gateway generic JSON인지 더 본다.

핵심은 "`text/html`이면 CDN", "`application/json`이면 app"처럼 한 칸으로 끝내지 않는 것이다.

### 3. body 첫 줄

첫 줄이나 첫 문장에서는 "말투"만 본다.

| body 첫 인상 | first pass 해석 |
|---|---|
| `Request blocked`, `Access denied`, vendor 브랜드, support/reference 문구 | branded CDN/edge 에러 페이지 후보 |
| `{ "errorCode": ... }`, `{ "message": ... }`, `{ "traceId": ... }` | app-owned JSON 후보 |
| `{ "status": 504, "title": "Gateway Timeout" }` | app JSON보다 gateway/proxy generic JSON 후보 |
| `<html><title>Sign in` 또는 `<form` | CDN 에러보다 login/page 후보 |

## CloudFront / Cloudflare branded first-line 미니 카드

beginner가 가장 빨리 쓰는 힌트는 body의 첫 줄에 vendor 이름이 박혀 있는지다.
아래 문구는 "자주 보이는 branded clue" 정도로만 외우면 충분하다. 정확한 원인 분류표로 외울 필요는 없다.

| body 첫 줄이나 눈에 띄는 문구 | 초급자 첫 번역 | 너무 빨리 단정하면 안 되는 것 | 안전한 다음 확인 |
|---|---|---|---|
| `Generated by cloudfront (CloudFront)` | CloudFront branded error/footer가 붙은 HTML 후보 | CloudFront가 왜 막았는지 원인 확정 | `X-Cache`, `Server`, status를 같이 본다 |
| `The request could not be satisfied.` | CloudFront 기본 에러 문구 후보 | app business error다 | body가 HTML인지, `403/503`인지 본다 |
| `Request blocked.` | edge/CDN/WAF 계열 차단 페이지 후보 | app authorization 실패다 | vendor 헤더와 final URL을 같이 본다 |
| `Attention Required! | Cloudflare` | Cloudflare branded challenge/block page 후보 | origin app이 직접 낸 HTML이다 | `CF-Ray`, status, `text/html`을 같이 본다 |
| `Sorry, you have been blocked` | Cloudflare block page 후보 | app의 일반 403 화면이다 | `Server: cloudflare`, `CF-Ray`가 붙는지 본다 |
| `Error 502`/`Error 504`와 Cloudflare 표기 | Cloudflare branded error page 후보 | app가 JSON 대신 HTML로 바꿨다 | body 아래 vendor 표기와 헤더를 같이 본다 |

짧게 줄이면 이렇다.

```text
Generated by cloudfront / The request could not be satisfied. -> CloudFront branded HTML 후보
Attention Required! | Cloudflare / Sorry, you have been blocked -> Cloudflare branded HTML 후보
```

이 섹션의 목적은 vendor 운영 세부를 외우는 게 아니라 "`이건 app JSON 말투가 아니다`"를 5초 안에 알아차리는 것이다.

## 30초 결정표

| 내가 지금 본 것 | 바로 적을 메모 |
|---|---|
| `403` + `text/html` + branded first line | "CDN/edge가 직접 보여 준 에러 HTML 후보" |
| `429` + `text/html` + rate limit/blocked 문구 | "CDN/WAF rate limit 페이지 후보" |
| `500` + `application/json` + 서비스 `errorCode` | "app이 자기 JSON 에러 계약으로 응답한 후보" |
| `401` + `application/json` + 서비스 message | "app auth/API JSON 에러 후보" |
| `504` + `application/json` + generic title/detail | "app JSON보다 gateway timeout JSON 후보" |

초급자는 원인 확정보다 이런 메모 한 줄이면 충분하다.
핵심은 "branded HTML vs service JSON"을 먼저 가르는 것이다.

## 흔한 오해와 함정

- HTML이면 무조건 CDN이라고 단정한다. login HTML이나 gateway 기본 페이지도 있다.
- JSON이면 무조건 app이라고 단정한다. `502`/`504` generic JSON은 gateway가 만들 수도 있다.
- `403`이면 전부 app authorization 실패라고 읽는다. edge/WAF/CDN 거절도 흔하다.
- body를 길게 읽다가 첫 분기를 놓친다. 초급자 첫 pass에서는 첫 줄과 핵심 단어만 보면 된다.
- status를 무시하고 브랜드 문구만 본다. branded page는 보통 `403`/`429`/`503`과 같이 읽을 때 더 안전하다.
- `Generated by cloudfront`, `Attention Required! | Cloudflare`를 보면 바로 vendor 기능 이름부터 찾으러 간다. beginner 첫 목적은 운영 원인 추적이 아니라 `app JSON이 아님`을 분리하는 것이다.

## 실무에서 쓰는 모습

예를 들어 `/api/orders`를 눌렀는데 아래처럼 보였다고 하자.

| 장면 | DevTools에서 보이는 것 | 초급자 첫 판독 |
|---|---|---|
| A | `403`, `text/html`, `Request blocked`와 vendor 문구 | CDN/edge 에러 HTML 후보 |
| A-1 | `403`, `text/html`, `Generated by cloudfront (CloudFront)` footer | CloudFront branded error HTML 후보 |
| A-2 | `403`, `text/html`, `Attention Required! | Cloudflare` 제목 | Cloudflare branded error HTML 후보 |
| B | `500`, `application/json`, `{ "errorCode": "ORDER_..." }` | app-owned JSON error 후보 |
| C | `504`, `application/json`, `{ "title": "Gateway Timeout" }` | app JSON보다 gateway/proxy JSON 후보 |

이때 다음 한 걸음은 한 가지면 된다.

- A면 CDN/vendor header를 본다
- B면 app 공통 에러 포맷과 로그를 본다
- C면 app 코드보다 앞단 gateway/proxy timeout 단서를 먼저 본다

## 한 줄 정리

DevTools 첫 pass에서는 `status`, `content-type`, body 첫 줄만 묶어 `403/429/503 + branded HTML`은 CDN/edge 후보로, `400/401/404/500 + 서비스 JSON`은 app-owned JSON 후보로 먼저 나누면 된다.
