# 첫 요청은 H2인데 다음은 H3 원인 라우터

## 한 줄 요약

> `first h2 -> next h3`는 대개 이상 징후보다 "이번 응답에서 H3 힌트를 배우고, 다음 새 connection에서만 그 힌트를 썼다"는 정상 흐름이다. 다만 같은 URL 두 줄이나 첫 줄 실패가 섞이면 recovery trace와 분리해서 읽어야 한다.

## 가능한 원인

1. 가장 흔한 경우는 이번 `h2` 응답에서 `Alt-Svc`나 HTTPS RR 단서를 배웠고, 그 힌트가 현재 row가 아니라 다음 새 connection에만 적용된 경우다. 즉 첫 줄 `h2`는 정상이고, 다음 줄 `h3`도 정상이다. 다음 문서: [브라우저의 HTTP 버전 선택: ALPN, Alt-Svc, Fallback 입문](./browser-http-version-selection-alpn-alt-svc-fallback.md), [H3 Discovery Observability Primer: Alt-Svc vs HTTPS RR 확인하기](./h3-discovery-observability-primer.md)
2. 브라우저가 이미 열어 둔 `h2` keep-alive connection을 재사용하다가, 그 연결이 닫히거나 새 connection이 필요해진 시점에만 `h3`를 선택한 경우다. 같은 origin이어도 요청마다 새로 협상하는 것이 아니라 connection 단위로 바뀐다는 점을 놓치면 "중간에 갑자기 protocol이 바뀌었다"처럼 보인다. 다음 문서: [Browser DevTools Protocol Column Labels Primer](./browser-devtools-protocol-column-labels-primer.md), [Request Timing Decomposition: DNS, Connect, TLS, TTFB, TTLB](./request-timing-decomposition-dns-connect-tls-ttfb-ttlb.md)
3. 같은 URL이 두 번 보이며 첫 줄이 `421`, `(failed)`, `canceled` 같은 종료로 끝난 뒤 둘째 줄이 `h3`로 다시 간 경우다. 이때는 정상 warm-up보다 wrong connection recovery, stale hint 교정, retry trace일 수 있다. 다음 문서: [같은 URL 요청이 두 번 보임 원인 라우터](./같은-url-요청이-두-번-보임-원인-라우터.md), [H3 Fallback Trace Bridge](./h3-fallback-trace-bridge.md)
4. 반대로 학습자가 기대한 건 "항상 H3여야 한다"인데 실제로는 네트워크, 브라우저, UDP 경로, 실험 설정 차이 때문에 어떤 새 connection만 `h3`가 되는 경우다. 이때 질문은 "왜 첫 줄이 h2였나"보다 "왜 어떤 새 connection은 여전히 h2로 남나"로 바뀐다. 다음 문서: [HTTP/2, HTTP/3 Downgrade Attribution, Alt-Svc, UDP Block](./http2-http3-downgrade-attribution-alt-svc-udp-block.md), [Alt-Svc Cache Lifecycle Basics](./alt-svc-cache-lifecycle-basics.md)

## 빠른 자기 진단

1. 첫 `h2` row의 response headers에 `Alt-Svc`가 있었는지 먼저 본다. 있었다면 그 row 자체가 H3여야 한다고 기대하지 말고, 다음 새 connection 관찰로 넘어간다.
2. 두 번째 `h3` row가 정말 새 connection인지 확인한다. `Connection ID`, `Remote Address`, waterfall의 `Connect` 흔적이 바뀌면 "같은 연결에서 말만 바뀜"이 아니라 새 연결 생성이다.
3. 두 줄 사이에 `421`, `(failed)`, `canceled`, `h3 -> h2` fallback 같은 실패 흔적이 있는지 본다. 있으면 정상 warm-up보다 recovery trace 문맥을 먼저 읽는다.
4. 같은 브라우저에서 새로고침, 새 탭, 잠시 idle 뒤 재요청처럼 "새 connection이 필요해지는 순간"에만 `h3`가 나타나는지 본다. 그렇다면 protocol 선택은 요청 단위보다 connection 단위라는 힌트다.
5. `Disable cache`나 body cache 여부와 `Protocol` 변화를 섞어 읽지 않는다. cache 실험은 응답 재사용 질문이고, `h2`/`h3`는 transport 선택 질문이다.
