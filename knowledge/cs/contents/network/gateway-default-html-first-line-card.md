# Gateway Default HTML First-Line Card

> 한 줄 요약: DevTools body 첫 줄이 `Bad Gateway`, `Gateway Timeout`, `503 Service Unavailable`처럼 짧고 브랜드가 거의 없으면 CDN branded page보다 gateway/proxy 기본 에러 페이지 후보를 먼저 열어 두는 편이 초급자 첫 분기를 더 안전하게 만든다.

## 핵심 개념

처음에는 body 전체를 읽지 말고 "`누가 말하는 문구인가`"만 가르면 된다.

- CDN branded page: `CloudFront`, `Cloudflare`처럼 vendor 이름이나 차단 문구가 전면에 나온다
- gateway 기본 페이지: `Bad Gateway`, `Gateway Timeout`, `Service Unavailable`처럼 매우 짧은 상태 문구가 앞에 나온다

이 문구는 판결문이 아니라 **표지판**이다. 표지판 비유는 "첫 갈림길"까지만 맞고, 실제 owner 확정은 `Status`, `Content-Type`, `Server`/`Via`까지 같이 봐야 한다.

## 한눈에 보기

| body 첫 줄/첫 인상 | 초급자 첫 해석 | 더 안전한 다음 확인 |
|---|---|---|
| `Generated by cloudfront`, `Attention Required! | Cloudflare` | CDN branded page 후보 | vendor header, `403/429/503` 조합인가 |
| `<title>502 Bad Gateway</title>`, `<h1>Bad Gateway</h1>` | generic proxy/gateway HTML 후보 | `Server: nginx` 같은 단서가 보이는가 |
| `<title>504 Gateway Time-out</title>` 또는 `Gateway Timeout` | timeout을 proxy가 대신 말한 HTML 후보 | waterfall `waiting`, `Via`, upstream timeout 문맥이 있는가 |
| `upstream connect error or disconnect/reset before headers` | Envoy/mesh local reply 후보 | HTML인지 plain text인지보다 `Server: envoy`, `502/503/504` 조합을 본다 |
| `503 Service Unavailable`만 덩그러니 보임 | generic front-door/proxy page 후보 | app 공통 JSON이 없는지, managed LB 문맥인지 본다 |

짧게 외우면 이렇다.

```text
브랜드 이름이 크다 -> CDN 후보
상태 문구만 짧다 -> generic proxy/gateway 후보
envoy 연결 문구가 바로 보인다 -> envoy local reply 후보
```

## vendor별 first-line clue

| vendor/문맥 | beginner가 자주 보는 첫 줄 | 첫 번역 | caveat |
|---|---|---|---|
| Nginx default page | `<title>502 Bad Gateway</title>`, `<title>504 Gateway Time-out</title>` | Nginx가 upstream 실패를 기본 HTML로 대신 보여 준 후보 | 커스텀 에러 페이지면 이 문구가 안 보일 수 있다 |
| Envoy local reply | `upstream connect error...`, `upstream request timeout`, `503 Service Unavailable` | app JSON보다 Envoy/mesh가 짧게 실패를 말한 후보 | Envoy는 plain text나 JSON으로도 surface할 수 있어서 "항상 HTML"은 아니다 |
| ALB 같은 managed LB front door | `502 Bad Gateway`, `503 Service Unavailable`, `504 Gateway Timeout`처럼 브랜드 없는 짧은 문구 | ALB나 앞단 LB가 generic status page를 보여 준 후보 | body만으로 ALB를 확정하기는 약하고, header/access log 문맥이 더 중요하다 |

핵심은 vendor 이름을 외우는 것보다 "`브랜드가 없는 짧은 상태 문구`는 app 커스텀 페이지보다 generic proxy page 쪽이 더 가깝다"를 먼저 잡는 것이다.

## CDN branded page와 어떻게 분리하나

| 질문 | CDN branded page 쪽 | generic proxy page 쪽 |
|---|---|---|
| vendor 이름이 body에 큰가 | `CloudFront`, `Cloudflare`가 눈에 띈다 | vendor 이름 없이 상태 문구만 짧다 |
| 문장이 설명형인가 | `The request could not be satisfied.`, `Sorry, you have been blocked`처럼 브랜드 문장이 붙는다 | `Bad Gateway`, `Gateway Timeout`처럼 상태 문구 위주다 |
| 첫 액션은 어디로 가나 | CDN/WAF/edge 정책 헤더부터 본다 | proxy timeout, upstream 연결, `Server`/`Via`부터 본다 |

즉 "`html이 왔다`"가 아니라 "`브랜드 HTML인가, generic proxy HTML인가`"로 한 번 더 쪼개야 한다.

## 흔한 오해와 함정

- `Bad Gateway`를 보면 app controller가 그 문구를 직접 냈다고 생각한다. 초급자 첫 분기에서는 proxy/gateway 기본 페이지 후보가 더 안전하다.
- `CloudFront`, `Cloudflare`가 안 보이니 app HTML이라고 단정한다. generic proxy page는 브랜드가 없는 경우가 흔하다.
- `Server: nginx`만 보이면 Nginx가 근본 원인이라고 확정한다. Nginx는 전달자일 수도 있다.
- Envoy body가 JSON이나 plain text로 오면 이 카드가 틀렸다고 느낀다. 이 카드는 "기본 HTML first-line clue" 입문용이고, Envoy는 기본 local reply 포맷이 더 다양하다는 caveat를 같이 기억해야 한다.
- ALB body만 보고 AWS ALB라고 확정한다. managed LB 계열은 짧은 generic page를 공유할 수 있어 header/log가 더 중요하다.

## 실무에서 쓰는 모습

| 장면 | DevTools에서 보이는 것 | 첫 메모 |
|---|---|---|
| A | `502`, `text/html`, `<title>502 Bad Gateway</title>`, `Server: nginx` | "CDN branded page보다 nginx generic gateway page 후보" |
| B | `504`, `text/html`, `Gateway Timeout`, `Via` 있음 | "app JSON보다 proxy timeout 기본 페이지 후보" |
| C | `503`, 첫 줄이 `upstream connect error or disconnect/reset before headers`, `Server: envoy` | "Envoy local reply 후보, app JSON 아님" |
| D | `503`, 짧은 HTML, 브랜드 없음, AWS front-door 문맥 | "managed LB generic page 후보, app HTML 확정 보류" |

초급자 다음 한 걸음은 한 줄이면 충분하다.

1. branded word가 있는지 본다
2. 없으면 `Server`/`Via`와 status를 같이 본다
3. `502/503/504`면 app 공통 `errorCode`를 찾기 전에 gateway/proxy owner를 먼저 적는다

## 한 줄 정리

body 첫 줄이 브랜드보다 `Bad Gateway`, `Gateway Timeout`, `503 Service Unavailable` 같은 짧은 상태 문구에 가깝다면 app HTML보다 generic gateway/proxy 기본 페이지 후보를 먼저 열고 `Server`/`Via`로 owner를 보강하면 된다.
