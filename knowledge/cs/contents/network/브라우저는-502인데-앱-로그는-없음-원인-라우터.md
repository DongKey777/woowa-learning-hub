# 브라우저는 502인데 앱 로그는 없음 원인 라우터

## 한 줄 요약

> `502`와 app 로그 부재가 같이 보이면 먼저 "애플리케이션이 실패했다"보다 "앞단 proxy가 upstream에 닿기 전에 혹은 닿고도 유효한 응답을 못 받아 대신 말했나?"를 의심하는 편이 맞다.

## 가능한 원인

1. 가장 흔한 갈래는 gateway나 reverse proxy가 upstream 연결 실패, invalid response, no healthy upstream을 자기 `502`로 표면화한 경우다. 이때 app은 요청을 아예 못 받았거나, 받아도 사용자에게 나간 응답 owner는 proxy다. 다음 문서: [Proxy Local Reply vs Upstream Error Attribution](./proxy-local-reply-vs-upstream-error-attribution.md), [Browser DevTools `502` vs `504` vs App `500` 분기 카드](./browser-devtools-502-504-app-500-decision-card.md)
2. 배포나 scale-in 직후면 connection draining이나 readiness 전환이 어긋나 기존 연결이 끊기면서 `502`가 튈 수 있다. app 예외가 없는 이유는 코드 실패가 아니라 upstream 교체 타이밍에서 응답 경로가 깨졌기 때문이다. 다음 문서: [LB Connection Draining, Deployment Safe Close](./lb-connection-draining-deployment-safe-close.md), [Load Balancer 헬스체크 실패 패턴](./load-balancer-healthcheck-failure-patterns.md)
3. TLS, ALPN, host routing이 기대와 달라 proxy가 upstream과 정상적으로 프로토콜을 맞추지 못한 경우다. 사용자는 단순 `502`만 보지만 실제 원인은 "앱 로직"보다 handshake나 route mismatch에 더 가깝다. 다음 문서: [ALPN Negotiation Failure, Routing Mismatch](./alpn-negotiation-failure-routing-mismatch.md), [Vendor-Specific Proxy Symptom Translation: Nginx, Envoy, ALB](./vendor-specific-proxy-symptom-translation-nginx-envoy-alb.md)
4. 사실 upstream은 뒤늦게 성공하거나 다른 홉에서 재시도 중인데, 앞단의 timeout과 retry budget이 먼저 깨져 `502`만 남은 경우다. 이때 app 로그가 없다는 사실만으로 "요청이 완전히 안 갔다"를 단정하면 hop-chain 원인을 놓친다. 다음 문서: [Timeout Budget Propagation Across Proxy, Gateway, Service Hops](./timeout-budget-propagation-proxy-gateway-service-hop-chain.md), [Proxy Retry Budget Discipline](./proxy-retry-budget-discipline.md)

## 빠른 자기 진단

1. DevTools에서 `Status`와 response body 첫 줄을 같이 본다. 짧은 `Bad Gateway` HTML이나 vendor 기본 문구가 강하면 app payload보다 proxy local reply 후보를 먼저 잡는다.
2. 응답 헤더의 `Server`, `Via`, `X-Request-Id`를 함께 본다. proxy 흔적은 강한데 app trace key가 안 이어지면 "app 로그 누락"보다 "proxy owner" 쪽으로 기울인다.
3. 같은 시각에 배포, scale-in, target deregistration, healthcheck flap이 있었는지 본다. `502`가 그 구간에만 몰리면 코드 예외보다 draining/readiness 분기가 더 유력하다.
4. `502` 직전의 upstream connect/reset, protocol mismatch, retry 흔적을 edge 로그에서 찾는다. app access log가 비어 있어도 proxy access log나 vendor 필드에는 원인이 남는 경우가 많다.
5. 사용자가 본 `502`와 app 팀이 본 성공/무응답이 같은 요청인지 request id로 대조한다. 서로 다른 홉의 다른 관찰을 같은 사건으로 뭉개면 원인 분기가 틀어진다.
