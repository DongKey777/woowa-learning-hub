# 다운로드는 시작됐는데 중간에 끊김 원인 라우터

## 한 줄 요약

> 다운로드가 이미 시작됐다면 질문은 "응답이 왜 안 나갔지?"보다 "어느 지점까지 나갔고 누가 중간 종료를 먼저 관찰했지?"로 바뀐다.

## 가능한 원인

1. 사용자가 페이지를 떠나거나 브라우저, 프록시, 모바일 네트워크가 먼저 연결을 닫은 경우다. edge에는 `499`, origin에는 `200`이나 일부 bytes sent만 남을 수 있어 "서버는 성공인데 사용자는 실패"처럼 보인다. 다음 문서: [Client Disconnect, 499, Broken Pipe, Cancellation in Proxy Chains](./client-disconnect-499-broken-pipe-cancellation-proxy-chain.md), [Access Log Correlation Recipes: Tomcat, Jetty, Undertow](./access-log-correlation-recipes-tomcat-jetty-undertow.md)
2. 응답은 커밋됐지만 그 뒤 write나 flush에서 늦게 실패한 경우다. `broken pipe`, `ClientAbortException`, `AsyncRequestNotUsableException`은 종종 "첫 실패 원인"이 아니라 이미 끊긴 연결에 대한 후속 표면이다. 다음 문서: [Spring `DisconnectedClientHelper` Breadcrumb Wiring: MVC Download, SSE, Async Late Write](./spring-disconnectedclienthelper-breadcrumb-wiring-mvc-download-sse-async-late-write.md), [Spring MVC Async `onError` -> `AsyncRequestNotUsableException` Crosswalk](./spring-mvc-async-onerror-asyncrequestnotusableexception-crosswalk.md)
3. proxy, CDN, LB의 idle timeout이나 buffering 정책이 body 전송 중간에 먼저 끝난 경우다. app은 계속 쓰고 있다고 생각하지만 앞단이 먼저 닫아 사용자는 다운로드 중단으로 본다. 다음 문서: [HTTP Response Compression, Buffering, Streaming Trade-offs](./http-response-compression-buffering-streaming-tradeoffs.md), [Client Disconnect, 499, Broken Pipe, Cancellation in Proxy Chains](./client-disconnect-499-broken-pipe-cancellation-proxy-chain.md)
4. TLS close 방식이나 transport 종료가 어긋나 truncation처럼 보이는 경우다. `close_notify` 없이 FIN/RST가 먼저 오면 파일이 일부만 온 장면과 인증서/암호화 문제를 섞어 읽기 쉽다. 다음 문서: [TLS `close_notify`, FIN/RST, Truncation](./tls-close-notify-fin-rst-truncation.md), [FIN, RST, Half-Close, EOF](./fin-rst-half-close-eof-semantics.md)

## 빠른 자기 진단

1. DevTools waterfall에서 `Waiting`이 길었는지, `Content Download`가 길다가 끊겼는지 먼저 나눈다. 다운로드 중단이면 대개 `body` 단계 해석이 먼저다.
2. edge와 origin에서 같은 요청의 `status`, `bytes_sent`, `duration`, disconnect 표면을 같이 본다. edge `499` + origin `200`은 partial delivery 후보고, 둘 다 짧게 끝나면 pre-commit 실패 후보가 더 크다.
3. app 로그의 예외 시각이 첫 write 직전인지, 이미 일부 바이트를 보낸 뒤인지 확인한다. late write면 business 예외보다 cleanup/중단 전파 해석이 더 맞다.
4. 문제 구간이 큰 파일, 느린 모바일 네트워크, 배포 직후, 특정 proxy hop에서만 반복되는지 적는다. 재현 조건이 좁혀지면 timeout, buffering, transport 종료 갈래가 빨리 갈린다.
5. 파일 크기나 checksum이 기대보다 짧다면 "응답을 누가 만들었나"보다 "마지막 정상 바이트가 어디까지 갔나"를 먼저 추적한다.
