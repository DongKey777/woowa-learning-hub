# 웹소켓은 연결됐는데 메시지가 늦음 원인 라우터

## 한 줄 요약

> ping/pong이 돈다고 사용자 데이터도 즉시 흐르는 것은 아니다. control frame은 살아 있어도 payload는 buffering, flush, backpressure 경로에서 따로 늦어질 수 있다.

## 가능한 원인

1. proxy buffering이나 중간 hop의 flush 정책 때문에 작은 이벤트가 모였다가 한꺼번에 나갈 수 있다. heartbeat는 통과하지만 사용자 메시지는 지연될 수 있다. 다음 문서: [WebSocket Proxy Buffering, Streaming Latency](./websocket-proxy-buffering-streaming-latency.md)
2. TLS record coalescing이나 애플리케이션 flush 타이밍 때문에 payload first byte와 chunk cadence가 밀릴 수 있다. 특히 작은 write를 자주 보내는 경로에서 체감이 크다. 다음 문서: [TLS Record Sizing, Flush, Streaming Latency](./tls-record-sizing-flush-streaming-latency.md)
3. 메시지 frame이 너무 크거나 앱이 한 번에 큰 payload를 만들어 보내면 control frame보다 data frame이 늦게 보일 수 있다. 이때는 "연결 유지"보다 frame sizing이 병목이다. 다음 문서: [WebSocket Fragmentation, Frame Sizing](./websocket-fragmentation-frame-sizing.md)
4. 수신 측이 데이터를 충분히 빨리 소비하지 못해 app queue나 TCP receiver window가 막힐 수 있다. ping/pong은 작은 control frame이라 지나가도 실제 데이터는 backpressure를 맞아 느려진다. 다음 문서: [TCP Zero Window, Persist Probe, Receiver Backpressure](./tcp-zero-window-persist-probe-receiver-backpressure.md)
5. heartbeat는 살아 있지만 reconnect, resend, slow consumer 정책이 약해 사용자 체감만 나빠질 수 있다. "연결이 안 끊긴다"와 "메시지가 제때 보인다"는 다른 운영 문제다. 다음 문서: [WebSocket Heartbeat, Backpressure, Reconnect](./websocket-heartbeat-backpressure-reconnect.md)

## 빠른 자기 진단

1. reconnect 흔적이 없는지 먼저 본다. 끊김이 없다면 "연결 실패"보다 payload 지연 경로를 보는 편이 빠르다.
2. 작은 heartbeat와 큰 사용자 메시지의 지연 차이를 비교한다. heartbeat만 빠르면 buffering, frame sizing, 수신 측 소비 속도 갈래가 강해진다.
3. 메시지가 한 건씩 늦는지, 몇 초 치가 묶여서 오는지 기록한다. 묶여서 오면 proxy buffering이나 flush batch 가능성이 높다.
4. 큰 payload, 이미지 메타데이터, 대량 이벤트처럼 유독 느린 메시지 종류가 있는지 본다. 특정 크기에서만 느리면 fragmentation이나 TLS record 경계를 의심할 만하다.
5. 송신 측 send queue, 수신 측 처리 지연, zero window 흔적을 같이 본다. 손실이 거의 없는데도 payload만 막히면 backpressure 쪽 단서가 더 직접적이다.
