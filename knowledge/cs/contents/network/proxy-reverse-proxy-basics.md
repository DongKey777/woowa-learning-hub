## 핵심 개념

**프록시(Proxy)** 는 중간에서 요청을 대신 처리해 주는 서버다. 어느 쪽 앞에 서느냐에 따라 이름이 달라진다.

- **포워드 프록시(Forward Proxy)**: 클라이언트 쪽에 놓인다. 클라이언트가 직접 목적지에 요청하지 않고 프록시를 통해 나간다.
- **리버스 프록시(Reverse Proxy)**: 서버 쪽에 놓인다. 외부 요청이 내부 서버에 직접 닿지 않고 프록시를 먼저 거친다.

입문자가 헷갈리는 부분은 "리버스"라는 단어다. 방향이 거꾸로인 게 아니라, 클라이언트 입장에서는 리버스 프록시가 실제 서버처럼 보인다는 의미다.

## 한눈에 보기

| 구분 | 위치 | 숨기는 대상 | 대표 사용 예 |
|---|---|---|---|
| 포워드 프록시 | 클라이언트 앞 | 클라이언트 IP | 사내 웹 필터, VPN 우회 |
| 리버스 프록시 | 서버 앞 | 서버 구조 | Nginx, API Gateway |

## 상세 분해

### 포워드 프록시가 하는 일

- 클라이언트의 요청을 목적지 서버에 대신 전달한다.
- 서버 입장에서는 실제 클라이언트가 누구인지 모른다(IP가 프록시로 보인다).
- 기업 내부망에서 특정 사이트 접근을 제한하거나, 국가별 콘텐츠 우회에 쓰인다.

### 리버스 프록시가 하는 일

- 외부 요청을 받아 내부 서버 중 하나에 전달한다.
- 클라이언트 입장에서는 내부 서버가 몇 대인지 모른다.
- 주요 역할: 로드 밸런싱, TLS 종료, 정적 파일 캐싱, 요청 압축.

### 실제 인프라에서의 흐름

```
클라이언트 → (인터넷) → 리버스 프록시(Nginx) → 애플리케이션 서버
```

리버스 프록시가 443 포트에서 TLS를 처리하고, 내부 애플리케이션 서버(8080)에는 HTTP로 전달한다. 이를 TLS 종료(termination)라고 한다.

## 흔한 오해와 함정

- "Nginx는 웹 서버다"라고만 알고 리버스 프록시 역할을 모르는 경우가 많다. Nginx는 정적 파일 제공(웹 서버)도 하지만 Spring Boot 앞에서 리버스 프록시로 쓰는 게 더 일반적이다.
- 포워드 프록시를 VPN과 동일하게 생각하기도 한다. VPN은 트래픽 전체를 암호화 터널로 보내지만, 포워드 프록시는 특정 프로토콜(주로 HTTP/HTTPS) 요청만 중계한다.
- 리버스 프록시가 있으면 클라이언트 실제 IP가 숨어버려 애플리케이션 로그에 프록시 IP만 남는다. `X-Forwarded-For` 헤더로 원래 IP를 전달받아야 한다.

## 실무에서 쓰는 모습

Spring Boot 서비스를 EC2에 배포할 때 Nginx를 앞에 두고 리버스 프록시로 설정한다. 이렇게 하면 SSL 인증서 관리는 Nginx가 담당하고, 애플리케이션 서버는 HTTP만 처리해도 된다. 여러 Spring Boot 인스턴스를 띄웠을 때는 Nginx가 라운드로빈으로 요청을 나눠준다.

## 면접/시니어 질문 미리보기

**Q. 포워드 프록시와 리버스 프록시의 차이를 설명해 주세요.**
포워드 프록시는 클라이언트 앞에서 서버에 대신 요청을 보내고 클라이언트를 숨긴다. 리버스 프록시는 서버 앞에서 외부 요청을 받아 내부 서버에 전달하고 서버 구조를 숨긴다.

**Q. 리버스 프록시를 쓰면 어떤 이점이 있나요?**
TLS 처리를 한 곳에서 관리할 수 있고, 여러 서버로 요청을 분산할 수 있다. 또한 정적 파일 캐싱, 속도 제한, 압축 같은 공통 기능을 애플리케이션 코드 없이 처리할 수 있다.

**Q. 리버스 프록시를 통과하면 클라이언트 IP가 사라지는데 어떻게 복원하나요?**
Nginx 같은 프록시가 `X-Forwarded-For` 헤더에 원래 클라이언트 IP를 담아 전달한다. Spring Boot에서는 이 헤더를 읽거나 `server.forward-headers-strategy=NATIVE` 설정으로 자동 복원할 수 있다.

## 한 줄 정리

포워드 프록시는 클라이언트를, 리버스 프록시는 서버를 숨기고 중개한다.
