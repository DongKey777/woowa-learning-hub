## 핵심 개념

TCP와 UDP는 둘 다 **전송 계층(Transport Layer)** 프로토콜이다. 같은 IP 네트워크 위에서 동작하지만, 데이터를 어떻게 전달하는지에서 크게 갈린다. 입문자가 가장 많이 헷갈리는 지점은 "TCP가 느리고 UDP가 빠르다"만 외우고 **왜** 그런지를 모르는 것이다. 차이는 연결과 신뢰성 보장 메커니즘에서 온다.

## 한눈에 보기

| 특성 | TCP | UDP |
|---|---|---|
| 연결 | 3-way handshake로 맺는다 | 연결 없이 바로 보낸다 |
| 순서 보장 | 그렇다 | 아니다 |
| 재전송 | 유실되면 자동 재전송 | 없다 |
| 흐름 제어 | 있다 | 없다 |
| 혼잡 제어 | 있다 | 없다 |
| 오버헤드 | 상대적으로 크다 | 상대적으로 작다 |
| 주요 사용처 | HTTP, HTTPS, DB 접속 | DNS, 스트리밍, 게임, QUIC |

## 상세 분해

### TCP의 신뢰성 보장

TCP는 아래 메커니즘으로 신뢰성을 지킨다.

- **3-way handshake**: 보내기 전에 연결을 먼저 맺는다. 양쪽이 준비됐음을 확인한다.
- **순서 번호(sequence number)**: 패킷마다 번호를 붙여 수신 측이 순서를 재조립한다.
- **ACK(확인 응답)**: 잘 받았다는 응답이 안 오면 재전송한다.
- **흐름 제어**: 수신 측 버퍼가 꽉 차지 않도록 속도를 맞춘다.

### UDP의 단순함

UDP는 위 메커니즘 없이 데이터그램을 그냥 던진다.

- 순서가 뒤바뀔 수 있다.
- 유실되어도 재전송 없다.
- 대신 연결 준비 없이 즉시 전송하므로 지연이 작다.
- 신뢰성이 필요하면 애플리케이션이 직접 구현한다.

### 어떤 상황에 어느 쪽을 쓰나

- **TCP**: 데이터가 정확히 도달해야 할 때. HTTP, 파일 전송, DB 통신.
- **UDP**: 약간의 유실보다 빠른 전달이 중요할 때. 실시간 음성/영상, DNS 조회, QUIC(HTTP/3 내부).

## 흔한 오해와 함정

- "UDP는 나쁜 프로토콜이다"가 아니라, 신뢰성 보장을 애플리케이션 층으로 올린 설계다. QUIC은 UDP 위에서 자체적으로 신뢰성을 구현한다.
- TCP를 쓴다고 데이터가 절대 안 깨지는 건 아니다. 전송 계층 신뢰성이지, 애플리케이션 로직 오류까지 막아주진 않는다.
- "TCP가 느리다"는 말은 연결·재전송 오버헤드 때문이지, 대역폭 자체가 작은 건 아니다.

## 실무에서 쓰는 모습

백엔드 개발자가 실제로 맞닥뜨리는 시나리오는 보통 이렇다.

- Spring Boot 서버가 DB에 접속할 때는 TCP를 쓴다. 쿼리 결과가 정확히 와야 하기 때문이다.
- DNS 조회는 UDP로 작은 패킷 하나를 보낸다. 빠른 응답이 우선이고, 실패 시 재시도는 resolver가 담당한다.
- 실시간 스트리밍 서비스는 UDP 기반으로 약간의 화질 저하를 감수하고 지연을 줄인다.

## 면접/시니어 질문 미리보기

**Q. TCP와 UDP의 차이를 설명해 주세요.**
핵심은 연결 지향 여부와 신뢰성 보장이다. TCP는 3-way handshake로 연결을 맺고 순서·재전송을 보장한다. UDP는 연결 없이 즉시 전송하며 신뢰성은 애플리케이션 몫이다.

**Q. HTTP/3는 왜 UDP를 쓰나요?**
QUIC 프로토콜이 UDP 위에서 자체적으로 신뢰성과 멀티플렉싱을 구현해 TCP의 HOL blocking 문제를 피하면서 속도를 높이기 위해서다.

**Q. TCP 연결이 느리다는 건 어떤 의미인가요?**
데이터 전송 전에 3-way handshake 왕복 지연이 추가되고, HTTPS라면 TLS handshake까지 더해진다는 의미다. 대역폭이 아닌 지연(latency) 관점의 비용이다.

## 한 줄 정리

TCP는 연결과 신뢰성을 프로토콜이 보장하고, UDP는 그 보장을 생략해 속도를 얻는다.
