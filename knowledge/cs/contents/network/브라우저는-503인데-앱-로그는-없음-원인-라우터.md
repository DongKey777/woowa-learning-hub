# 브라우저는 503인데 앱 로그는 없음 원인 라우터

## 한 줄 요약

> `503`와 app 로그 부재가 같이 보이면 먼저 "애플리케이션이 죽었다"보다 "앞단 gateway나 sidecar가 upstream을 고르지 못했거나, local reply로 먼저 실패를 선언했나?"를 의심하는 편이 맞다.

## 가능한 원인

1. gateway나 reverse proxy가 `no healthy upstream` 같은 상태를 자기 `503`으로 표면화했을 수 있다. 이때 app은 요청을 아예 못 받았거나, 받기 전에 upstream 선택이 끝나 버린다. 다음 문서: [Proxy Local Reply vs Upstream Error Attribution](./proxy-local-reply-vs-upstream-error-attribution.md), [Load Balancer 헬스체크 실패 패턴](./load-balancer-healthcheck-failure-patterns.md)
2. 배포나 scale-in 직후 readiness 전환과 draining이 어긋나 healthy target이 잠깐 비었을 수 있다. app 예외가 없는 이유는 코드 실패보다 target 교체 타이밍에서 proxy가 먼저 `503`을 만들었기 때문이다. 다음 문서: [Load Balancer 헬스체크 실패 패턴](./load-balancer-healthcheck-failure-patterns.md), [Vendor-Specific Proxy Symptom Translation: Nginx, Envoy, ALB](./vendor-specific-proxy-symptom-translation-nginx-envoy-alb.md)
3. mesh나 gateway의 local overload, adaptive concurrency, circuit breaking이 일부 route를 먼저 shedding했을 수 있다. 사용자는 단순 `503`만 보지만 app capacity 부족이 아니라 sidecar 정책 응답일 수도 있다. 다음 문서: [Service Mesh Local Reply, Timeout, Reset Attribution](./service-mesh-local-reply-timeout-reset-attribution.md), [Mesh Adaptive Concurrency, Local Reply, Metrics Tuning](./mesh-adaptive-concurrency-local-reply-metrics-tuning.md)
4. 실제 bottleneck은 upstream app보다 queue나 connection slot 부족인데, dispatch 전에 timeout이나 local reject가 나가 `503`처럼 보였을 수 있다. 이 경우 질문은 "왜 app이 느렸나"보다 "왜 upstream으로 보내기 전에 막혔나"로 바뀐다. 다음 문서: [Queue Saturation Attribution, Metrics, Runbook](./queue-saturation-attribution-metrics-runbook.md)
5. 사실 `503`은 맞지만 body owner를 app으로 오해했을 수 있다. 짧은 `Service Unavailable` HTML이나 vendor 기본 문구가 보이면 app 공통 JSON보다 proxy 기본 페이지 후보를 먼저 잡는 편이 안전하다. 다음 문서: [Browser DevTools `Server` / `Via` / `X-Request-Id` 1분 헤더 카드](./browser-devtools-gateway-error-header-clue-card.md), [Gateway Default HTML First-Line Card](./gateway-default-html-first-line-card.md)

## 빠른 자기 진단

1. DevTools에서 `Status`와 response body 첫 줄을 같이 본다. 짧은 `503 Service Unavailable` HTML이나 vendor 기본 문구가 강하면 app payload보다 proxy local reply 후보를 먼저 적는다.
2. 응답 헤더의 `Server`, `Via`, `X-Request-Id`를 함께 본다. proxy 흔적은 강한데 app trace key가 안 이어지면 "app 로그 누락"보다 "앞단 owner" 쪽으로 기운다.
3. 같은 시각에 배포, scale-in, target deregistration, readiness flap, healthcheck 실패가 있었는지 본다. `503`이 그 구간에만 몰리면 코드 예외보다 healthy upstream 공백 분기가 더 유력하다.
4. mesh나 gateway 메트릭에서 local reply reason, overload counter, queue wait, pending acquire가 튀는지 본다. 튄다면 app business 실패보다 보호 정책이나 dispatch 전 병목 쪽이 먼저다.
5. 사용자가 본 `503`과 app 팀이 본 무응답이 같은 요청인지 request id로 대조한다. 서로 다른 홉의 다른 관찰을 같은 사건으로 묶으면 원인 분기가 틀어진다.
