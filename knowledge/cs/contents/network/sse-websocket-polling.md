## 왜 이 비교가 중요한가

실시간 기능은 면접에서 자주 나온다.

- 알림
- 채팅
- 주식/게임 상태 갱신
- 작업 진행률 스트리밍

여기서 중요한 건 "최신 기술 이름"이 아니라,
**요구사항에 맞는 통신 방식을 고르는 기준**이다.

---

## 한 줄 정의

- `Polling`: 클라이언트가 주기적으로 서버에 "새 소식 있나요?"를 묻는다.
- `SSE`: 서버가 HTTP 연결 하나를 길게 유지하면서 클라이언트로 이벤트를 푸시한다.
- `WebSocket`: 한 번 연결을 맺으면 양방향으로 프레임을 주고받는다.

핵심 차이는 다음과 같다.

- 누가 먼저 말하나
- 연결을 얼마나 오래 유지하나
- 양방향이 필요한가
- 인프라에서 얼마나 다루기 쉬운가

---

## Polling

Polling은 가장 단순한 방식이다.

클라이언트가 일정 주기로 HTTP 요청을 보내고, 서버가 현재 상태를 응답한다.

### 장점

- 구현이 쉽다
- 기존 HTTP 인프라를 그대로 쓸 수 있다
- 프록시, 로드밸런서, 인증과 궁합이 좋다

### 단점

- 요청이 필요 없어도 계속 트래픽이 발생한다
- 응답이 없을 때도 네트워크와 서버 자원을 쓴다
- 주기를 짧게 하면 부하가 커지고, 길게 하면 지연이 커진다

### Long Polling

Polling을 조금 개선한 방식이다.

- 클라이언트가 요청을 보낸다
- 서버가 새 이벤트가 생길 때까지 응답을 늦춘다
- 응답 후 클라이언트가 즉시 다시 요청한다

짧은 주기 polling보다 낫지만,
여전히 HTTP 요청/응답의 반복이라는 구조는 그대로다.

---

## SSE

SSE(Server-Sent Events)는 **서버 -> 클라이언트 단방향 스트리밍**에 적합하다.

브라우저에서는 `EventSource`를 통해 주로 사용한다.

### 특징

- HTTP 연결을 유지한다
- 서버가 텍스트 이벤트를 계속 보낸다
- 브라우저가 자동 재연결을 지원한다
- 이벤트마다 `id`, `event`, `data`를 붙일 수 있다

### 장점

- HTTP 기반이라 도입이 비교적 쉽다
- 단방향 알림에는 WebSocket보다 단순하다
- 재연결과 last-event-id 처리로 유실 복구를 설계하기 좋다

### 단점

- 클라이언트 -> 서버로는 사실상 못 보낸다
- 텍스트 기반이라 바이너리 전송에는 맞지 않는다
- 브라우저와 프록시 설정에 따라 버퍼링 문제가 생길 수 있다

### 잘 맞는 사례

- 알림 피드
- 주문/결제 상태 스트리밍
- 로그 tailing
- 진행률 업데이트

---

## WebSocket

WebSocket은 HTTP로 시작해서 `Upgrade` 이후 **양방향 풀듀플렉스 연결**로 바뀐다.

### 특징

- 한 번 연결하면 클라이언트와 서버가 자유롭게 메시지를 주고받는다
- 텍스트와 바이너리를 모두 다루기 쉽다
- ping/pong으로 연결 생존 여부를 확인할 수 있다

### 장점

- 채팅, 협업 편집, 게임처럼 양방향성이 강한 기능에 적합하다
- 실시간 반응성이 좋다
- 불필요한 HTTP 헤더를 계속 주고받지 않아 효율적이다

### 단점

- 연결 상태 관리가 복잡하다
- 로드밸런서 뒤에서 세션 분배를 신경써야 할 수 있다
- 끊김, 재연결, 중복 메시지, 순서 보장을 직접 다뤄야 한다

### 실무에서 놓치기 쉬운 점

- 연결 수가 많아지면 파일 디스크립터와 메모리 사용량이 빠르게 늘어난다
- 서버가 상태를 들고 있으면 scale-out이 어려워진다
- 인증 만료, 네트워크 스위칭, 모바일 절전 모드에서 자주 끊긴다

## Reverse Proxy와 Nginx 실전 포인트

실서비스에서는 통신 방식만큼이나 **중간 프록시 설정**이 중요하다.

### SSE에서 자주 보는 설정 포인트

- 응답 버퍼링을 줄여야 이벤트가 늦게 도착하지 않는다
- 긴 연결이므로 타임아웃을 충분히 길게 잡아야 한다
- 압축이나 중간 캐시가 이벤트 전달을 방해할 수 있다

### WebSocket에서 자주 보는 설정 포인트

- `Connection: Upgrade`, `Upgrade: websocket` 헤더 전달이 필요하다
- 프록시 타임아웃과 idle timeout을 맞춰야 한다
- 여러 서버로 분산할 때는 sticky session이 필요할 수 있다

### 운영 관점 체크리스트

- 헬스 체크가 연결 유지 트래픽과 충돌하지 않는가
- 배포 시 기존 연결을 어떻게 정상 종료할 것인가
- 프록시 뒤에서 클라이언트 IP를 어떻게 추적할 것인가
- 메시지 폭주 시 backpressure를 어디서 처리할 것인가

## 어떤 상황에 무엇을 쓸까

### Polling을 고를 때

- 구현 복잡도를 가장 낮추고 싶다
- 실시간성이 아주 중요하지 않다
- 요청 빈도가 낮다

### SSE를 고를 때

- 서버가 클라이언트에게 일방적으로 알려주면 된다
- 브라우저 기반 서비스다
- 이벤트 스트림 형태가 자연스럽다

### WebSocket을 고를 때

- 서버와 클라이언트가 양방향으로 빠르게 상호작용해야 한다
- 채팅, 게임, 협업 편집처럼 메시지 왕복이 잦다
- 바이너리나 커스텀 프로토콜이 필요하다

### 실전 판단 기준

1. 양방향이 꼭 필요한가
2. 연결을 오래 유지해도 되는가
3. 프록시와 로드밸런서 제약을 감당할 수 있는가
4. 재연결과 메시지 중복 처리를 어디서 할 것인가
