# 아키텍처 예상 답변

## Q. 왜 legacy Node 백엔드를 archive로 옮겼나요?

### 30초 답변

“전환 과정에서 바로 삭제하면 비교, 롤백, 문서 호환성이 깨지기 때문입니다.”

### 2분 답변

“사주다방은 Java 백엔드로 기준선을 옮겼지만, 기존 Node 구현은 `archive/legacy-node-api`로 보존했습니다. 이렇게 해야 기능 비교, 회귀 참고, 문서 링크 호환, 롤백 판단이 쉬워집니다. 마이그레이션에서는 삭제보다 기준선 명확화가 더 중요했습니다.”

### 5분 답변

“마이그레이션에서 가장 위험한 건 새 코드를 쓰는 것보다 기존 기준선을 한 번에 파괴하는 것입니다. 사주다방도 Java 백엔드로 기준선을 옮기면서 기존 Node 구현을 바로 삭제하지 않았습니다. 대신 `archive/legacy-node-api`와 `archive/legacy-node-docs`로 옮겨서 reference로 남겼습니다.

이렇게 하면 기능 비교가 가능하고, 문제가 생겼을 때 롤백 판단 근거가 생깁니다. 또한 문서나 경로 참조를 한 번에 깨뜨리지 않고 기준선을 천천히 옮길 수 있습니다. 루트 `api/`를 심볼릭 링크로 둔 것도 이런 이유입니다.

즉 archive 전략은 게으른 보존이 아니라, 전환 리스크를 줄이기 위한 소프트웨어 공학적 선택입니다. 삭제보다 기준선 명확화와 비교 가능성이 더 중요하다는 걸 보여주는 사례입니다.”

### 꼬리질문

- 왜 big bang 교체를 안 했나?
- archive와 branch는 어떤 차이가 있나?
- 현재 기준선은 어떻게 명확히 했나?

## Q. port/adaptor를 왜 도입했나요?

### 30초 답변

“reading과 billing처럼 책임이 다른 영역의 결합도를 낮추기 위해서입니다.”

### 2분 답변

“사주다방은 코인 reserve/capture/refund가 reading 흐름과 연결되어 있습니다. 하지만 reading이 billing 구현에 직접 의존하면 경계가 흐려지므로 `BillingPort`를 두고 application service에서만 연결하게 했습니다. 이렇게 하면 교체 가능성과 테스트 용이성이 올라갑니다.”

### 5분 답변

“이 질문은 DIP와 bounded context를 같이 설명하면 좋습니다. 사주다방에서는 reading과 billing이 비즈니스적으로 연결되어 있습니다. 운세 생성 전에 코인을 reserve하고, 성공하면 capture, 실패하면 refund해야 하기 때문입니다. 그런데 reading 코드가 billing의 repository나 내부 구현을 직접 알면 결합도가 크게 올라갑니다.

그래서 `BillingPort`를 두고, reading 쪽 application service는 그 port만 의존하게 했습니다. 실제 구현은 billing 쪽이 담당하고, reading은 ‘무엇이 필요하다’는 계약만 압니다. 이 구조 덕분에 테스트가 쉬워지고, 구현을 바꾸거나 리팩터링할 때 영향 범위도 줄어듭니다.

즉 port/adaptor는 이론적인 패턴이 아니라, 사주다방처럼 서로 다른 도메인이 연결될 때 결합도를 제어하기 위한 실전 도구라고 설명할 수 있습니다.”
