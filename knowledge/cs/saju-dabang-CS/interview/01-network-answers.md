# 네트워크 예상 답변

## Q. SSE를 왜 사용했나요?

### 30초 답변

“운세 생성은 시간이 걸리고, 서버가 중간 결과를 푸시해야 해서 SSE를 사용했습니다. 서버→클라이언트 단방향이면 충분했고, WebSocket보다 단순했습니다.”

### 2분 답변

“사주다방은 `/api/readings/jobs/{jobId}/events`에서 progress, section, completed 이벤트를 보냅니다. 운세 생성 중간 결과를 사용자에게 보여줘야 해서 실시간 채널이 필요했는데, 양방향 상호작용은 필요하지 않았습니다. 그래서 WebSocket보다 SSE가 더 단순하고 운영하기도 쉬웠습니다. 추가로 Redis replay를 둬서, 연결이 늦게 붙은 경우에도 일부 이벤트를 복원할 수 있게 만들었습니다.”

### 5분 답변

“이 질문은 단순히 ‘실시간이라서 SSE’라고 답하면 부족합니다. 사주다방의 운세 생성은 몇 초 이상 걸릴 수 있고, 결과도 한 번에 완성되지 않습니다. 사용자 입장에서는 단순 로딩이 아니라, 진행 중인지, 부분 결과가 나왔는지, 완료됐는지를 알아야 합니다. 그래서 서버에서 클라이언트로 지속적으로 상태를 보내는 구조가 필요했습니다.

WebSocket도 실시간 통신 수단이지만, 사주다방은 클라이언트가 서버에 계속 메시지를 보내는 양방향 채널이 핵심은 아닙니다. 핵심은 서버가 progress, section, completed 이벤트를 순서대로 보내는 것입니다. 이런 경우에는 SSE가 더 단순하고 HTTP 기반이라 디버깅과 운영도 쉽습니다.

또 하나 중요한 건 replay입니다. 네트워크는 항상 안정적이지 않기 때문에, 사용자가 늦게 붙거나 연결이 끊겼다가 복구될 수 있습니다. 사주다방은 Redis에 이벤트를 잠시 저장해 두고, 연결이 붙으면 replay list를 먼저 읽어 이미 발생한 이벤트를 다시 전달합니다. 이 덕분에 “실시간”과 “복구 가능성”을 동시에 챙길 수 있었습니다.

즉 사주다방에서 SSE는 단순히 편한 기술 선택이 아니라, 사용자 경험과 네트워크 현실을 함께 고려한 설계라고 설명할 수 있습니다.”

### 꼬리질문

- 왜 WebSocket이 아니라 SSE인가?
- replay는 왜 필요한가?
- SSE 연결이 끊기면 어떻게 복구하나?

## Q. 도메인과 DNS는 어떻게 연결되나요?

### 30초 답변

“도메인은 사람이 읽는 이름이고, DNS가 그 이름을 실제 서버 IP로 연결합니다. 사주다방은 `api-prelaunch.saju-dabang.com`을 prelaunch Java 서버에 연결해 검증에 사용했습니다.”

### 2분 답변

“사주다방은 production과 prelaunch를 구분하기 위해 `api-prelaunch.saju-dabang.com` 같은 별도 도메인을 사용했습니다. 사용자는 도메인만 알면 되고, Route53 같은 DNS가 실제 EC2 IP로 연결해 줍니다. 서버를 교체해도 프론트는 도메인만 유지하면 되기 때문에 운영이 훨씬 유연해집니다.”

### 5분 답변

“도메인과 DNS를 설명할 때 단순히 ‘주소를 IP로 바꿔준다’고 끝내지 말고, 운영에서 왜 필요한지까지 말하면 좋습니다. 사주다방은 production과 prelaunch를 분리해서 운영합니다. 예를 들어 `api-prelaunch.saju-dabang.com`은 출시 전 Java 백엔드를 검증하기 위한 주소입니다.

사용자는 이 도메인만 알면 되고, 실제로 어느 EC2 서버가 뒤에 있는지는 DNS가 결정합니다. Route53에서 도메인을 특정 IP로 연결해 두고, 나중에 서버를 교체하면 DNS만 바꾸면 됩니다. 사주다방도 prelaunch 박스를 교체하면서 도메인은 유지하고, 새 인스턴스로만 넘기는 방식으로 운영했습니다.

그리고 HTTPS는 이 도메인 위에서 작동하는 보안 계층입니다. 로그인, 결제, 세션 정보가 오가기 때문에 평문 HTTP는 적절하지 않습니다. Caddy가 TLS를 받아 내부 컨테이너로 reverse proxy를 해줍니다. 더 나아가 토스 파트너 API 호출은 mTLS를 사용해서 상호 인증까지 수행합니다.

즉 이 질문은 단순 네트워크 개념이 아니라, 실제 서비스에서 도메인, DNS, HTTPS, reverse proxy가 어떻게 함께 돌아가는지를 설명하는 문제라고 볼 수 있습니다.”
