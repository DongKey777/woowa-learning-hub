# 데이터베이스 예상 답변

## Q. 멱등성을 어떻게 보장했나요?

### 30초 답변

“같은 orderId가 여러 번 들어와도 한 번만 적립되도록 DB 충돌 제약과 idempotency key를 사용했습니다.”

### 2분 답변

“사주다방은 결제 적립 시 `transactions.order_id`와 `coin_ledger.idempotency_key`를 함께 사용합니다. 네트워크 재시도나 앱 재진입 때문에 같은 적립 요청이 반복될 수 있는데, 이 경우에도 실제 잔액 반영은 한 번만 일어나게 설계했습니다. 단순 캐시가 아니라 DB 제약으로 보장한 점이 중요합니다.”

### 5분 답변

“멱등성은 특히 결제나 포인트 적립처럼 돈과 연결된 데이터에서 중요합니다. 사주다방도 IAP 적립에서 같은 `orderId`가 여러 번 들어올 수 있다는 전제를 두고 설계했습니다. 네트워크 재시도, 앱 재진입, 클라이언트 중복 호출은 실제로 흔한 상황입니다.

이걸 애플리케이션 메모리만으로 막으면 프로세스 재시작이나 멀티 인스턴스 환경에서 깨질 수 있습니다. 그래서 사주다방은 DB 수준에서 멱등성을 보장합니다. `transactions.order_id`와 `coin_ledger.idempotency_key`를 사용해 같은 주문이 여러 번 들어와도 실제 적립은 한 번만 일어나게 만들었습니다.

하지만 멱등성만으로는 부족합니다. 거래 기록, 잔액 변경, 주문 상태, ledger 기록이 함께 바뀌어야 하므로 트랜잭션도 필요합니다. 그래서 사주다방에선 멱등성과 트랜잭션을 같이 사용합니다. 중복 적립도 막고, 부분 성공으로 인한 데이터 모순도 막는 구조입니다.”

### 꼬리질문

- 멱등성과 트랜잭션 차이는?
- 왜 애플리케이션 캐시만으로 막지 않았나?
- duplicate 요청은 왜 에러가 아니라 정상 응답처럼 처리하나?

## Q. reserve/capture/refund 모델을 왜 썼나요?

### 30초 답변

“운세 생성은 비동기 작업이라 실패 가능성이 있어서, 코인을 즉시 최종 차감하지 않고 reserve 후 성공 시 capture, 실패 시 refund하는 모델을 사용했습니다.”

### 2분 답변

“사주다방은 운세 생성 전에 코인을 reserve하고, worker가 성공적으로 결과를 만들면 capture, 실패하면 refund합니다. 이렇게 해야 장시간 비동기 작업에서 결제 정합성을 유지할 수 있습니다. 이 구조는 상태 전이 모델이면서 보상 트랜잭션 아이디어와도 연결됩니다.”

### 5분 답변

“운세 생성은 시간이 오래 걸리고 실패 가능성도 있습니다. 만약 시작 순간에 코인을 최종 차감해버리면, 실패했을 때 되돌리는 로직이 훨씬 위험해집니다. 그래서 사주다방은 먼저 reserve 상태를 만들고, 성공 시 capture, 실패 시 refund하는 모델을 사용합니다.

이 구조는 데이터베이스 관점에서는 상태 전이 모델이고, 시스템 설계 관점에서는 보상 트랜잭션 아이디어와 연결됩니다. reserve는 ‘사용 예정’ 상태, capture는 ‘최종 확정’, refund는 ‘실패했으니 되돌림’입니다.

이 방식의 장점은 단순히 안전한 것뿐 아니라 설명 가능하다는 점입니다. 운영자가 봐도 지금 이 거래가 reserve인지 완료인지 환불인지 상태를 따라갈 수 있습니다. 즉 장시간 비동기 작업과 금전성 자원이 결합될 때 정합성을 지키기 위한 현실적인 방법이라고 설명할 수 있습니다.”
