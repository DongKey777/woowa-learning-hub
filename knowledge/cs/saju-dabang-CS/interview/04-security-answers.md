# 보안 예상 답변

## Q. 세션을 왜 Redis와 DB에 같이 저장했나요?

### 30초 답변

“Redis는 빠른 조회용이고, DB는 durable store입니다. Redis miss 시에도 세션을 복구해야 해서 둘을 같이 썼습니다.”

### 2분 답변

“사주다방은 로그인 세션을 Redis에 캐시하지만, Redis는 영속 저장소가 아니기 때문에 `auth_sessions` 테이블을 함께 둡니다. 캐시가 날아가도 DB에서 세션을 복구하고 다시 Redis를 채워서 사용자 경험을 유지할 수 있습니다.”

### 5분 답변

“세션을 Redis와 DB에 같이 두는 이유는 속도와 안정성을 동시에 확보하기 위해서입니다. Redis는 빠른 조회에 강하고, 세션처럼 자주 읽는 데이터에 적합합니다. 하지만 캐시 계층이라 flush, 만료, 장애가 발생할 수 있어서 그것만 믿기 어렵습니다.

사주다방은 그래서 `auth_sessions` 테이블을 durable store로 두고, Redis는 앞단 캐시로 사용합니다. 요청이 들어오면 먼저 Redis를 보고, miss가 나면 DB에서 세션을 복구합니다. 찾은 세션은 다시 Redis에 채워 넣습니다. 이 덕분에 캐시가 날아가도 로그인 경험이 바로 깨지지 않습니다.

운영 관점에서도 이게 중요합니다. 로그인 상태가 예기치 않게 풀리면 사용자 경험이 크게 나빠집니다. 따라서 사주다방은 단순 성능 최적화가 아니라, 캐시 miss를 전제로 한 복구 가능한 인증 구조를 택했습니다.”

## Q. mTLS는 왜 필요했나요?

### 30초 답변

“토스 같은 민감한 파트너 API는 서버 인증만으로 부족할 수 있어서, 클라이언트 인증서까지 포함한 mTLS가 필요했습니다.”

### 2분 답변

“사주다방은 토스 파트너 API 연동에서 client certificate와 private key를 사용합니다. 일반 HTTPS는 서버의 신원을 검증하지만, mTLS는 호출하는 클라이언트도 검증하므로 파트너 연동 보안 수준을 더 높일 수 있습니다.”

### 5분 답변

“HTTPS와 mTLS를 구분해서 설명하면 좋습니다. 일반 HTTPS는 서버의 신원을 검증하는 데 초점이 있습니다. 즉 사용자는 ‘내가 진짜 이 서버와 통신하는가’를 확인합니다. 하지만 파트너 API처럼 민감한 시스템 간 통신에서는 이것만으로 부족할 수 있습니다. 서버도 호출하는 클라이언트를 더 강하게 확인해야 하기 때문입니다.

사주다방은 토스 파트너 API 연동에서 client certificate와 private key를 사용해 `SSLContext`를 구성합니다. 이건 단순히 암호화된 채널을 만드는 게 아니라, 호출 주체까지 인증하는 구조입니다. 즉 HTTPS보다 더 강한 상호 인증 모델입니다.

이 질문에서는 보안을 막연하게 말하기보다, 사용자-facing HTTPS와 파트너 시스템 간 mTLS가 목적상 다르다는 점, 그리고 왜 사주다방 같은 서비스에서 토스 연동에 이런 수준의 보안이 필요한지까지 연결해서 설명하면 좋습니다.”
