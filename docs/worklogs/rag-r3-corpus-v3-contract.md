# R3 Corpus v3 Contract (2026-05-02)

> **Companion document to `rag-r3-system-spec-v1.md`**. The system spec defines the *ideal RAG architecture*; this document defines the *corpus shape that ideal architecture requires*. The current corpus is migrated to comply with this contract via Pilot (Phase 4) → Wave A/B/C/D (Phase 8).
>
> **Authoring principle (per user directive)**: design optimal system first → corpus follows. This contract is shaped by the system spec's stage-by-stage data needs. It is *not* shaped by the current corpus's accidental form (mostly frontmatter-less, anchor-polluted body, ad hoc retrieval-anchor-keywords inline metadata). The current corpus is brought into compliance by migration; the system spec does not regress to fit the corpus.
>
> **Verification discipline**: every field in this contract is justified by a specific stage in the system spec. Fields without a clear retrieval / ranking / safety purpose are *excluded*, not added "just in case". Each field's effect on retrieval quality is measured by the Real qrel suite (Phase 6), not by intuition.

---

## 0. Context

### 0.1 What v3 fixes vs v2

The v2 frontmatter contract (`PILOT_V2_FIELDS = (concept_id, doc_role, level, aliases, expected_queries)`, defined in `scripts/learning/rag/corpus_lint.py:57`) had **3 critical structural defects**:

1. **`expected_queries` was indexed as retrieval anchors**. The corpus_loader's `_emit_chunks` appended `frontmatter_expected_queries` into `chunk.body` (FTS / sidecar:body), `chunk.anchors` (sidecar:aliases), and the dense `chunk_body + anchor_suffix` path — so the same strings used as qrel seeds were *also* the indexed retrieval surface. Result: any qrel auto-generated from `expected_queries` matched its own doc by alias indexing alone, producing circular `recall=1.0`. Fixed in commit `054a1a3` (corpus_loader.py drops the parameter); the v3 contract enforces the separation structurally.

2. **No mission ↔ CS bridge metadata**. Woowa learners frequently ask about a mission concept ("roomescape DAO 패턴") and need a CS concept doc as the answer. v2 has no `mission_ids` field, so the system has no way to bridge.

3. **No contextual chunk prefix support**. Anthropic Contextual Retrieval (verified 50-100 token chunk-specific prefix, generated by Claude Haiku per chunk) reduces retrieval failure rate by 35-67% depending on stack (verified 2026-05-02 from https://www.anthropic.com/news/contextual-retrieval). v2 has no field for this.

v3 closes all three.

### 0.2 What v3 keeps from v2

- 5 pilot-required fields stay required (`concept_id`, `doc_role`, `level`, `aliases`, `expected_queries`). Their *meaning* is upgraded:
  - `expected_queries` becomes **qrel seed only** — never enters indexing channels.
  - `aliases` becomes **the** lexical hint channel — `expected_queries` and anchors no longer pollute it.
- Wave 2 fields from the prior plan (`canonical`, `language`, `source_priority`, `symptoms`, `intents`, `confusable_with`, `forbidden_neighbors`) are upgraded from optional to *measurement-gated required* (per-field lift verified by Phase 4 pilot).
- `prerequisites`, `next_docs`, `linked_paths` stay optional graph fields; promoted only after a graph retriever is operational (Phase 5 implementation).

### 0.3 Out of scope

- **Mass migration of all 2,274 docs at once**. Migration is wave-based; Wave A migrates failure-cohort docs first.
- **Auto-generation of frontmatter by an LLM without human review**. AI-assisted authoring is allowed for first-draft frontmatter, but every migrated doc passes a human review (or a separate AI session review) before counted as "v3 compliant".
- **Backwards compatibility with retrieval-anchor-keywords inline markdown metadata**. v3 docs put anchor phrases into the `aliases` field; the inline `retrieval-anchor-keywords:` line is migrated away. Lint marks remaining inline anchors as legacy until that doc's wave migration completes.

## 1. Design Principles (linked to system spec)

### 2.2 Per-field specification

#### 2.2.1 `schema_version` (int, required, == 3)

```yaml
schema_version: 3
```

Reserved values: `2` for legacy v2 (current minority of docs), `3` for this contract. Lint rejects any other value.

#### 2.2.2 `concept_id` (str, required, format `category/slug`)

```yaml
concept_id: spring/di
```

Stable identity. Used as:
- qrel `query_id` prefix (`spring/di:expected:1`)
- Concept catalog v3 primary key
- SignalRetriever rule key
- Trace artifact concept attribution

Format: `<category>/<slug>` where `category` matches the corpus folder structure (`spring`, `database`, etc.) and `slug` is kebab-case.

Uniqueness: every doc has a unique `concept_id`. `corpus_lint.py:check_concept_id_uniqueness` (already implemented) extends to v3.

Stability: `concept_id` is *immutable* once a doc is in the catalog. Renaming requires a migration entry that maps old→new, so qrels and references don't break.

#### 2.2.3 `canonical` (bool, Wave 2)

```yaml
canonical: true
```

If `true`, this doc is the *canonical* primer/reference for `concept_id`. SignalRetriever prefers canonical docs as injection candidates. If multiple docs share `concept_id` (via different roles like primer + deep_dive), only one is `canonical: true`.

Default: `false` (omit field). Lint enforces at most one `canonical: true` per `concept_id`.

#### 2.2.4 `category` (str, required, enum)

```yaml
category: spring
```

One of the 11 verified corpus categories: `algorithm`, `data-structure`, `database`, `design-pattern`, `language`, `network`, `operating-system`, `security`, `software-engineering`, `spring`, `system-design`. Must match the doc's folder placement under `knowledge/cs/contents/`.

#### 2.2.5 `doc_role` (str, required, enum, 8 values)

See §3 for the full taxonomy. Briefly:

```yaml
doc_role: primer
# enum: primer | bridge | deep_dive | playbook | chooser | symptom_router | drill | mission_bridge
```

#### 2.2.6 `level` (str, required, enum)

```yaml
level: beginner
# enum: beginner | intermediate | advanced
```

Used by:
- Stage 1 query plan to bias retrieval toward learner-appropriate level
- MetadataRanker filter: beginner query demotes `deep_dive` candidates
- Real qrel `level` cohort split

The legacy `**난이도: 🟢 Beginner**` markdown metadata is migrated to this field. Lint warns if both legacy markdown and frontmatter `level` exist with different values.

#### 2.2.7 `language` (str, Wave 2, enum)

```yaml
language: ko
# enum: ko | en | mixed
```

Heuristic: dominant character class.
- `ko`: > 30% CJK characters
- `en`: 0% CJK
- `mixed`: > 0% but ≤ 30% CJK

Used by:
- Stage 4 reranker language fallback (Korean/mixed → multilingual reranker; English-only path documented but never triggers Korean regression — see system spec §6.1)
- Per-language Real qrel cohort metric

#### 2.2.8 `source_priority` (int 0-100, Wave 2)

```yaml
source_priority: 90
```

Higher = more authoritative. Citation guard requires top-K to include at least one chunk with `source_priority >= 80`. Suggested values:
- 90-100: canonical primer / chooser docs
- 70-89: deep_dive / playbook / bridge
- 50-69: drill / supplementary
- < 50: legacy / deprecated docs (kept for history but rarely cited)

#### 2.2.9 `mission_ids` (list[str], Wave 2 / required for `mission_bridge` role)

```yaml
mission_ids:
  - missions/roomescape
  - missions/lotto
```

References Woowa Tech Course mission identifiers. Format: `missions/<mission-slug>`.

Used by:
- MissionBridgeRetriever: when QueryPlan binds a `mission_id`, candidates with matching `mission_ids` are boosted.
- Concept catalog v3 reverse index: `mission_id → list[concept_id]`.

#### 2.2.10 `review_feedback_tags` (list[str], Wave 2 / required for `mission_bridge` role)

```yaml
review_feedback_tags:
  - DI-vs-locator
  - transactional-boundary
```

Mentor PR comment patterns mined from `state/repos/<repo>/packets/topic-*.json`. Format: kebab-case slug.

Used by:
- Mission bridge query expansion: a learner asking "왜 멘토가 service locator 안 좋다고 했어?" gets routed to docs tagged `DI-vs-locator`.
- Future feedback-driven retrieval (Phase 9 multi-turn).

#### 2.2.11 `aliases` (list[str], required, lexical hint)

```yaml
aliases:
  - DI
  - dependency injection
  - 의존성 주입
  - new 대신 주입
```

The **only** lexical hint channel for retrieval. Maps to `lexical_sidecar:aliases` (boost 2.5x in BM25 — see system spec §4.1).

**Strict rule (C2)**: `aliases` does *not* repeat `expected_queries` strings. They are disjoint by contract. Lint enforces this in v3: `set(aliases) ∩ set(expected_queries) == ∅`.

Korean shortform note: include both colloquial Korean and English term variants. e.g., for spring/di: `["DI", "dependency injection", "의존성 주입", "주입", "new 대신 주입"]`.

#### 2.2.12 `symptoms` (list[str], Wave 2 for `symptom_router` / `playbook`)

```yaml
symptoms:
  - 저장은 됐는데 목록은 옛날 데이터
  - stale list after update
  - replica lag visible to user
```

Symptom phrases as the learner *would describe the bug*. Used by Stage 1 intent classifier (`symptom` intent triggers when a query contains a symptom phrase).

Required for `doc_role: symptom_router` and `doc_role: playbook`. Lint warns if these roles miss `symptoms`.

#### 2.2.13 `intents` (list[str], Wave 2, enum subset)

```yaml
intents:
  - definition
  - comparison
  - design
# enum: definition | comparison | symptom | mission_bridge | deep_dive | drill
```

Lists which intent labels this doc serves. Used by per-cohort fusion weight profile (system spec §5.2).

#### 2.2.14 `prerequisites` (list[str], optional, concept_id refs)

```yaml
prerequisites:
  - language/java-interface
  - software-engineering/coupling-cohesion-basics
```

Graph metadata. Used by the optional GraphExpander (system spec §4.7) for context companion expansion. Format: list of `concept_id`s.

#### 2.2.15 `next_docs` (list[str], optional, concept_id refs)

```yaml
next_docs:
  - spring/bean-container-lifecycle
  - spring/aop-proxy
```

"After learning this, what should the learner read next?" — used by the AI coach to suggest follow-up reading, *not* by retrieval directly.

#### 2.2.16 `linked_paths` (list[str], optional, path refs)

```yaml
linked_paths:
  - contents/spring/spring-bean-di-basics.md
```

Direct path references for the Stage 5 companion expansion. Format: corpus-relative path.

#### 2.2.17 `confusable_with` (list[str], Wave 2 for confusable cohort)

```yaml
confusable_with:
  - design-pattern/service-locator
  - design-pattern/factory-pattern-basics
```

Concept pairs where a confusable mistake is likely. The Real qrel `confusable_pairs` cohort uses these to seed test queries. Refs are `concept_id`.

#### 2.2.18 `forbidden_neighbors` (list[str], Wave 2 for qrel-risk cohort)

```yaml
forbidden_neighbors:
  - contents/design-pattern/object-oriented-design-pattern-basics.md
  - contents/spring/spring-transactional-basics.md
```

Docs that should *never* be retrieved for queries about this concept. Stage 3 fusion drops candidates matching `forbidden_neighbors`. The Real qrel `forbidden_neighbor` cohort enforces `forbidden_rate@5 == 0.0`.

#### 2.2.19 `expected_queries` (list[str], required, qrel seed only)

```yaml
expected_queries:
  - 처음 배우는데 DI가 뭐야?
  - new 대신 주입받는 이유?
  - 인터페이스로 주입하는 이유는?
```

**Strictly qrel seed**, never indexing surface. Used by `r3.eval.qrels.qrels_from_frontmatter_doc` to *initially populate* qrels (the human curator then revises / replaces with fully paraphrased queries for the `paraphrase_human` cohort).

**Structural enforcement**: corpus_loader.py removed `frontmatter_expected_queries` from `_emit_chunks` signature in commit `054a1a3`. The 4 indexing channels (chunk.body, chunk.anchors, sidecar:aliases, FTS) cannot accept these strings. Verified by `tests/unit/test_corpus_loader.py` and `tests/unit/test_lance_index_builder.py` (anchors expectation 6→3).

#### 2.2.20 `contextual_chunk_prefix` (str, optional, per-doc override)

```yaml
contextual_chunk_prefix: |
  이 문서는 Spring DI를 처음 배우는 학습자에게 의존성 주입의
  의미와 객체 조립 방식을 설명한다. 본 chunk는 ...
```

**Per-doc override**. By default, the indexer auto-generates a per-chunk contextual prefix using a Claude session at index build time (Anthropic Contextual Retrieval pattern, verified 50-100 tokens, Claude 3 Haiku in Anthropic's example).

If `contextual_chunk_prefix` is set in the frontmatter, that string is *prepended to every chunk* of this doc instead of auto-generation.

Rationale for both options:
- Auto-generation: zero authoring burden, but per-chunk Claude calls cost compute at index time.
- Per-doc override: deterministic, no compute cost, fits docs where chunks share a common context.

Per-chunk granular contextual prefixes (one prefix per chunk_id) are an *implementation* concern (Phase 5), not a frontmatter field — they're generated and stored alongside each chunk in the index, not in the markdown source.

## 3. Doc Role Taxonomy

### 3.1 The 8 roles

```yaml
doc_role:
  - primer            # 첫 도입; 큰 그림, 왜 필요한지
  - bridge            # 두 개념 연결; A vs B vs C 관계
  - deep_dive         # 내부 메커니즘; 어떻게 작동하는지
  - playbook          # 실전 시나리오; 어떻게 결정/실행하는지
  - chooser           # A vs B vs C 결정 가이드
  - symptom_router    # 증상 → 원인 매핑
  - drill             # 자가 진단 / 퀴즈 / 자기 검증
  - mission_bridge    # Woowa 미션 ↔ CS 개념 연결
```

### 3.2 Role detail

#### primer

**Purpose**: 학습자가 처음 만나는 문서. 큰 그림 + 왜 필요한지.

**Required fields**: schema_version, concept_id, canonical (보통 true), category, doc_role, level (보통 beginner), aliases, expected_queries, source_priority (≥ 85).

**Recommended length**: 800-2,000 단어.

**Example**:
> "Spring DI 처음 배우는데" → `contents/spring/spring-di-primer.md`
> "트랜잭션이 뭐야?" → `contents/database/transaction-primer.md`

**Linting**: primer는 `intents: [definition]` 최소 포함. `mission_bridge`와는 *상호 배타* (mission_bridge가 자체 primer 역할).

#### bridge

**Purpose**: 두 개의 가까운 개념을 연결. "A는 B랑 어떻게 다른가" 또는 "A 다음에 B로 어떻게 발전하는가".

**Required fields**: confusable_with (≥ 1), intents (보통 `[comparison]`).

**Example**: "DI ↔ IoC ↔ Service Locator 차이" → `contents/spring/di-vs-ioc-vs-service-locator-bridge.md`

#### deep_dive

**Purpose**: 내부 메커니즘. 입문자보다 *왜 그렇게 작동하는지*를 알고 싶은 학습자 대상.

**Required fields**: level (보통 `intermediate` 또는 `advanced`), prerequisites.

**Example**: "Spring Bean Container 라이프사이클 내부" → `contents/spring/bean-container-lifecycle-deep-dive.md`

#### playbook

**Purpose**: "이런 상황에서는 어떻게 결정/실행하는가". 실무 가이드.

**Required fields**: symptoms (실무 시나리오 주로 증상 기반).

**Example**: "트랜잭션 격리 수준 결정 가이드" → `contents/database/isolation-level-decision-playbook.md`

#### chooser

**Purpose**: A vs B vs C vs ... 중 어느 것을 쓸지 결정.

**Required fields**: confusable_with (≥ 2), forbidden_neighbors (각 후보의 잘못된 추천 케이스).

**Example**: "Strategy vs Template Method vs Factory" → `contents/design-pattern/strategy-template-factory-chooser.md`

**Lint**: chooser는 적어도 3개 이상의 후보 비교를 다뤄야 함 (chooser 단독 비교 = bridge 역할 차이).

#### symptom_router

**Purpose**: 학습자가 *증상*으로 표현한 문제를 원인 doc으로 라우팅.

**Required fields**: symptoms (다중, ≥ 3).

**Example**: "stale list after update — why?" → `contents/database/projection-freshness-symptom-router.md`

**Lint**: symptom_router는 자체로 정답이 아니라 *라우터*이므로, `linked_paths`로 cause docs를 가리켜야 함.

#### drill

**Purpose**: 자가 진단. 학습자가 답변하고 채점받는 형식.

**Required fields**: level (drill의 난이도), expected_queries (drill 질문 자체).

**Length**: 짧음 (200-500 단어). 답이 있는 quiz 형식.

#### mission_bridge

**Purpose**: Woowa 미션 컨텍스트 → CS 개념 연결.

**Required fields**: mission_ids (≥ 1), review_feedback_tags (가능 시), source_priority (≥ 75).

**Example**: "roomescape mission DAO 패턴 ↔ Repository 패턴" → `contents/software-engineering/roomescape-dao-vs-repository-bridge.md`

**Lint**: mission_bridge는 mission_ids가 *반드시 1개 이상* 있어야 함. 없으면 그냥 bridge.

### 3.3 Role exclusion rules

| Pair | Status |
|---|---|
| primer + canonical=true | OK; primer가 보통 canonical |
| primer + mission_bridge | ❌ exclusive (mission_bridge가 primer 역할 가짐) |
| chooser + bridge | ❌ exclusive (chooser는 ≥3 비교, bridge는 2개 연결) |
| symptom_router + linked_paths | symptom_router는 linked_paths 권장 (cause 가리킴) |
| drill + level | drill의 level은 *문제 난이도*이지 학습자 level이 아님 |

### 3.4 Doc role distribution target (full corpus)

After Wave A/B/C/D migration, 2,274 docs target distribution:

| Role | Approximate % | Approximate count |
|---|---|---|
| primer | 25% | ~570 |
| bridge | 15% | ~340 |
| deep_dive | 30% | ~680 |
| playbook | 15% | ~340 |
| chooser | 5% | ~115 (currently sparse — Wave C creates these) |
| symptom_router | 3% | ~70 (currently sparse) |
| drill | 2% | ~45 |
| mission_bridge | 5% | ~115 (currently 0 — Wave C creates these) |

Wave C is largely about creating *missing* chooser / symptom_router / mission_bridge docs.

## 4. Chunking Strategy

### 4.1 Chunk granularity

- **Target chunk size**: 200-400 tokens (current `MAX_CHARS_PER_CHUNK = 1600` chars ≈ 400 tokens — kept).
- **Minimum chunk size**: 60 chars (current `MIN_CHARS_PER_CHUNK`, kept; below this is nav fragment).
- **Section boundary**: H2 (`##`) is primary split; H3 (`###`) sub-split if H2 section too large; paragraph hard split last resort.
- **Anchor exclusion**: `[retrieval anchors]` suffix is *no longer appended to chunk.body* (commit `054a1a3` removed `frontmatter_expected_queries` source; `retrieval_anchors` from `_extract_retrieval_anchors` still appended for now, will be migrated to `aliases` field in Wave A).

### 4.2 Contextual chunk prefix (Anthropic Contextual Retrieval)

**Verified specification (https://www.anthropic.com/news/contextual-retrieval, fetched 2026-05-02)**:
- Chunk prefix size: 50-100 tokens
- Generated by Claude 3 Haiku per chunk in Anthropic's reference implementation
- Effect: 35% reduction in retrieval failure (embedding alone), 49% (with BM25), 67% (with rerank)

**Our implementation (Phase 5)**:
- Per-chunk auto-generation at index build time
- AI session prepends a doc-level + chunk-position context sentence:
  ```
  이 문서는 {doc_title}에 대해 다룬다. 본 chunk는 {section_path} 섹션에서
  {brief_chunk_role}을 설명한다.
  ```
- Output stored alongside chunk in the LanceDB row (new column `contextual_prefix`)
- Dense embedding is computed on `contextual_prefix + embedding_body` (clean section text), *not* on `body` (which still has retrieval anchors for FTS)
- Note: `embedding_body` is *separate* from `body` already (verified `corpus_loader.py:_emit_chunks`); v3 contract upgrades this with explicit contextual prefix prepending

**Override option**: doc author can set `contextual_chunk_prefix` in frontmatter to bypass auto-generation. Useful for docs where the entire body shares one obvious context (e.g., a glossary).

### 4.3 Anchor 분리 (separation contract)

The current `[retrieval anchors] phrase1 | phrase2` suffix in chunk.body is **legacy** (from before commit `054a1a3`). v3 migrates these into the `aliases` frontmatter field so they reach lexical_sidecar:aliases without polluting body / dense embedding / FTS body field.

Migration step (Wave A):
1. Extract phrases from current `_extract_retrieval_anchors(text)` for each doc
2. Merge into frontmatter `aliases` (deduped against existing aliases)
3. Remove inline `retrieval-anchor-keywords:` and `### Retrieval Anchors` sections
4. Re-index → chunk.body cleanly excludes the suffix

After Wave A, the `_emit_chunks` `anchor_suffix` logic can be retired (Phase 5 implementation flag).

### 4.4 Korean tokenization symmetry

Verified empirically on M4 (kiwipiepy 0.23.1, 2026-05-02): 0.036 ms/query warm.

**Index side** (`indexer.py:_lance_records`):
- `search_terms` field = kiwipiepy morphemes from `_embed_text(chunk)` (chunk_body without anchor pollution)
- Currently active.

**Query side** (`searcher.py:_korean_search_terms_query`):
- Same kiwipiepy.Kiwi instance, same morpheme rules
- Used in `_lance_candidate_pool` Korean FTS sub-query

def test_korean_tokenization_symmetry():

## 6. Linting Rules (corpus_lint.py v3 extension)

The current `scripts/learning/rag/corpus_lint.py:check_corpus_v2_pilot_frontmatter` (line 286) handles v2. v3 adds:

```python
PILOT_V3_FIELDS = (
    "concept_id", "doc_role", "level", "language", "aliases", "expected_queries"
)

WAVE2_V3_FIELDS = (
    "canonical", "category", "source_priority", "intents",
    # role-conditional:
    # symptoms: required for symptom_router | playbook
    # mission_ids: required for mission_bridge
    # confusable_with: required for chooser
)

VALID_DOC_ROLES = {
    "primer", "bridge", "deep_dive", "playbook",
    "chooser", "symptom_router", "drill", "mission_bridge",
}
VALID_LEVELS = {"beginner", "intermediate", "advanced"}
VALID_LANGUAGES = {"ko", "en", "mixed"}
VALID_INTENTS = {
    "definition", "comparison", "symptom",
    "mission_bridge", "deep_dive", "drill", "design", "troubleshooting",
}
VALID_CATEGORIES = {
    "algorithm", "data-structure", "database", "design-pattern",
    "language", "network", "operating-system", "security",
    "software-engineering", "spring", "system-design",
}

def check_corpus_v3_pilot_frontmatter(file_path, frontmatter):
    """v3 schema_version 3 pilot validation."""
    issues = []
    # 1. schema_version == 3
    if str(frontmatter.get("schema_version")) != "3":
        return issues  # not v3, skip
    # 2. all PILOT_V3_FIELDS present
    for field in PILOT_V3_FIELDS:
        if field not in frontmatter or frontmatter[field] in (None, "", []):
            issues.append(f"missing required v3 pilot field: {field}")
    # 3. doc_role enum
    if frontmatter.get("doc_role") not in VALID_DOC_ROLES:
        issues.append(f"invalid doc_role: {frontmatter.get('doc_role')}")
    # 4. level enum
    if frontmatter.get("level") not in VALID_LEVELS:
        issues.append(f"invalid level: {frontmatter.get('level')}")
    # 5. language enum
    if frontmatter.get("language") not in VALID_LANGUAGES:
        issues.append(f"invalid language: {frontmatter.get('language')}")
    # 6. aliases / expected_queries are lists of non-empty strings
    for f in ("aliases", "expected_queries"):
        v = frontmatter.get(f)
        if not isinstance(v, list) or not all(isinstance(s, str) and s.strip() for s in v):
            issues.append(f"{f} must be list[str] non-empty")
    # 7. CRITICAL: aliases ⊥ expected_queries (no overlap)
    overlap = set(frontmatter.get("aliases", [])) & set(frontmatter.get("expected_queries", []))
    if overlap:
        issues.append(
            f"aliases ⊥ expected_queries violation (overlap: {overlap}). "
            f"v3 contract requires structural separation."
        )
    # 8. category matches folder placement
    expected_category = derive_category_from_path(file_path)
    if frontmatter.get("category") != expected_category:
        issues.append(
            f"category mismatch: frontmatter says '{frontmatter.get('category')}', "
            f"folder is '{expected_category}'"
        )
    # 9. role-conditional requirements
    role = frontmatter.get("doc_role")
    if role in ("symptom_router", "playbook") and not frontmatter.get("symptoms"):
        issues.append(f"role={role} requires non-empty symptoms")
    if role == "mission_bridge" and not frontmatter.get("mission_ids"):
        issues.append(f"role=mission_bridge requires non-empty mission_ids")
    if role == "chooser" and len(frontmatter.get("confusable_with", [])) < 2:
        issues.append(f"role=chooser requires confusable_with with ≥ 2 candidates")
    return issues
```

This is the **strict mode**. Lint runs in two modes:
- `--strict-v3`: every v3 doc must pass (Pilot + Wave 2 fields). For new docs.
- `--lenient-v3`: only Pilot fields enforced; Wave 2 fields warned but not blocking. For migration.

## 13. Verdict

This contract defines the corpus shape that the R3 system requires. It is *driven* by the system spec — every field has a stage owner, every constraint has a verification path. The current corpus is migrated to comply via Pilot (50 docs) → Wave A (existing v2 → v3 auto) → Wave B (high-frequency primers) → Wave C (new chooser / symptom_router / mission_bridge docs) → Wave D (low-quality cleanup).

The contract is **structural defense**: aliases ⊥ expected_queries is enforced both in `corpus_loader.py` (cannot pollute) and in `corpus_lint.py:check_corpus_v3_pilot_frontmatter` (cannot author with overlap). Anchor pollution is migrated out via Wave A. Contextual chunk prefix is per-chunk auto-generated (Anthropic Contextual Retrieval pattern, verified 50-100 tokens, 35-67% failure rate reduction).

The Pilot 50 docs are the test bed; Real qrel suite (Phase 3) measures whether v3 corpus + R3 system meets calibrated cutover gates (Phase 6 → Phase 8 → Phase 10). Until Pilot baseline establishes the real numbers, this contract is *design only* — implementation lives in Phase 5 (corpus_lint v3 extension, migrate_frontmatter v3 tool, contextual_prefix indexer integration).
